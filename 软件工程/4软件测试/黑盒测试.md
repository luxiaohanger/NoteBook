## 黑盒测试

黑盒测试不关心程序内部实现，只根据**输入、输出、需求规格说明**来设计测试用例。

```java
public class LoginService {
    public String login(String username, String password) {
        if (username == null || username.length() < 3 || username.length() > 10) {
            return "用户名非法";
        }
        if (password == null || password.length() < 6 || password.length() > 12) {
            return "密码非法";
        }
        if (username.equals("admin") && password.equals("123456")) {
            return "登录成功";
        }
        return "用户名或密码错误";
    }
}
```

黑盒测试关注的是：

```text
输入 username、password
观察输出结果是否符合需求
```

而不是关心代码中 `if` 是怎么写的。

> 黑盒测试把被测程序看成一个“黑盒”，只检查输入和输出是否符合规格说明。

> 易错点：黑盒测试不是不需要理解需求，而是**不依赖代码内部结构**设计测试用例。

## 等价类划分

等价类划分是把输入数据划分成若干类，认为**同一类中的数据对程序的影响相同**，因此每类选一个代表值进行测试。

对于如下需求：

```text
用户名长度必须在 3 到 10 之间。
密码长度必须在 6 到 12 之间。
```

可以划分等价类：

| 输入项   | 有效等价类                | 无效等价类                       |
| ----- | -------------------- | --------------------------- |
| 用户名长度 | $3 \leq len \leq 10$ | $len < 3$，$len > 10$，`null` |
| 密码长度  | $6 \leq len \leq 12$ | $len < 6$，$len > 12$，`null` |

测试用例示例：

```java
import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.assertEquals;

public class LoginServiceTest {

    LoginService service = new LoginService();

    @Test
    public void testValidUsernameAndPassword() {
        assertEquals("登录成功", service.login("admin", "123456"));
    }

    @Test
    public void testUsernameTooShort() {
        assertEquals("用户名非法", service.login("ab", "123456"));
    }

    @Test
    public void testUsernameTooLong() {
        assertEquals("用户名非法", service.login("abcdefghijk", "123456"));
    }

    @Test
    public void testPasswordTooShort() {
        assertEquals("密码非法", service.login("admin", "12345"));
    }

    @Test
    public void testPasswordTooLong() {
        assertEquals("密码非法", service.login("admin", "1234567890123"));
    }
}
```

这里没有穷举所有用户名和密码，而是从每个等价类中选代表值。

> 等价类划分的核心是“同类取一个代表”，减少测试数量。

> 易错点：既要考虑**有效等价类**，也要考虑**无效等价类**。

## 边界值分析

边界值分析认为程序最容易在输入范围的**边界附近**出错，所以应重点测试边界值及其附近的值。

对于需求：

$$
3 \leq username.length \leq 10
$$

用户名长度的关键测试值是：

| 类型   | 长度   |
| ---- | ---- |
| 下边界外 | $2$  |
| 下边界  | $3$  |
| 下边界内 | $4$  |
| 上边界内 | $9$  |
| 上边界  | $10$ |
| 上边界外 | $11$ |

测试代码：

```java
@Test
public void testUsernameBoundaryValues() {
    LoginService service = new LoginService();

    assertEquals("用户名非法", service.login("ab", "123456"));          // 2
    assertEquals("用户名或密码错误", service.login("abc", "123456"));   // 3
    assertEquals("用户名或密码错误", service.login("abcd", "123456"));  // 4

    assertEquals("用户名或密码错误", service.login("abcdefghi", "123456"));   // 9
    assertEquals("用户名或密码错误", service.login("abcdefghij", "123456"));  // 10
    assertEquals("用户名非法", service.login("abcdefghijk", "123456"));      // 11
}
```

注意：这里 `"abc"` 不是合法账号，所以返回 `"用户名或密码错误"`，但它通过了用户名长度检查。

对于密码长度：

$$
6 \leq password.length \leq 12
$$

测试代码：

```java
@Test
public void testPasswordBoundaryValues() {
    LoginService service = new LoginService();

    assertEquals("密码非法", service.login("admin", "12345"));          // 5
    assertEquals("登录成功", service.login("admin", "123456"));         // 6
    assertEquals("用户名或密码错误", service.login("admin", "1234567")); // 7

    assertEquals("用户名或密码错误", service.login("admin", "12345678901"));   // 11
    assertEquals("用户名或密码错误", service.login("admin", "123456789012"));  // 12
    assertEquals("密码非法", service.login("admin", "1234567890123"));        // 13
}
```

> 边界值分析重点测最小值、最大值、刚小于边界、刚大于边界。

> 易错点：边界值分析通常是等价类划分的补充，不是替代。

## 决策表

决策表适合测试**多个条件组合决定不同动作**的业务规则。

例如登录需求如下：

```text
1. 用户名格式合法；
2. 密码格式合法；
3. 用户名和密码匹配；
4. 若用户名非法，返回“用户名非法”；
5. 若密码非法，返回“密码非法”；
6. 若账号密码匹配，返回“登录成功”；
7. 否则返回“用户名或密码错误”。
```

可以构造决策表：

| 条件 / 规则 | R1    | R2   | R3   | R4       |
| ------- | ----- | ---- | ---- | -------- |
| 用户名合法   | N     | Y    | Y    | Y        |
| 密码合法    | -     | N    | Y    | Y        |
| 账号密码匹配  | -     | -    | Y    | N        |
| **动作**  | 用户名非法 | 密码非法 | 登录成功 | 用户名或密码错误 |

其中 `-` 表示该条件无关。

对应测试代码：

```java
@Test
public void testDecisionTable() {
    LoginService service = new LoginService();

    // R1：用户名非法
    assertEquals("用户名非法", service.login("ab", "123456"));

    // R2：用户名合法，密码非法
    assertEquals("密码非法", service.login("admin", "123"));

    // R3：用户名合法，密码合法，账号密码匹配
    assertEquals("登录成功", service.login("admin", "123456"));

    // R4：用户名合法，密码合法，但账号密码不匹配
    assertEquals("用户名或密码错误", service.login("admin", "654321"));
}
```

决策表的优点是可以系统地覆盖条件组合，避免遗漏业务规则。

> 决策表适合“多个条件共同决定结果”的场景，例如权限判断、优惠规则、审批流程。

> 易错点：条件组合很多时，需要合并不影响结果的条件，否则测试用例数量会爆炸。

## 状态转换

状态转换测试适合测试对象具有**状态变化**的系统，例如订单、账号、门禁、自动售货机、TCP 连接等。

例如订单系统有如下状态：

```text
CREATED：已创建
PAID：已支付
SHIPPED：已发货
CANCELLED：已取消
```

状态转换规则：

| 当前状态      | 操作     | 下一个状态     |
| --------- | ------ | --------- |
| CREATED   | pay    | PAID      |
| CREATED   | cancel | CANCELLED |
| PAID      | ship   | SHIPPED   |
| PAID      | cancel | 不允许       |
| SHIPPED   | cancel | 不允许       |
| CANCELLED | pay    | 不允许       |

代码示例：

```java
public class Order {

    enum State {
        CREATED, PAID, SHIPPED, CANCELLED
    }

    private State state = State.CREATED;

    public State getState() {
        return state;
    }

    public void pay() {
        if (state != State.CREATED) {
            throw new IllegalStateException("当前状态不能支付");
        }
        state = State.PAID;
    }

    public void ship() {
        if (state != State.PAID) {
            throw new IllegalStateException("当前状态不能发货");
        }
        state = State.SHIPPED;
    }

    public void cancel() {
        if (state != State.CREATED) {
            throw new IllegalStateException("当前状态不能取消");
        }
        state = State.CANCELLED;
    }
}
```

测试合法状态转换：

```java
@Test
public void testValidStateTransition() {
    Order order = new Order();

    assertEquals(Order.State.CREATED, order.getState());

    order.pay();
    assertEquals(Order.State.PAID, order.getState());

    order.ship();
    assertEquals(Order.State.SHIPPED, order.getState());
}
```

测试非法状态转换：

```java
import static org.junit.jupiter.api.Assertions.assertThrows;

@Test
public void testInvalidStateTransition() {
    Order order = new Order();

    order.pay();

    assertThrows(IllegalStateException.class, () -> {
        order.cancel();
    });
}
```

还可以测试：

```java
@Test
public void testCancelledOrderCannotBePaid() {
    Order order = new Order();

    order.cancel();

    assertThrows(IllegalStateException.class, () -> {
        order.pay();
    });
}
```

> 状态转换测试关注“某状态下执行某操作后，状态是否正确变化”。

> 易错点：不仅要测试允许的转换，也要测试不允许的转换。

## 四种黑盒测试方法对比

| 方法    | 关注点    | 适用场景          | 示例                 |
| ----- | ------ | ------------- | ------------------ |
| 等价类划分 | 输入数据分类 | 输入范围较大，无法穷举   | 用户名长度合法 / 非法       |
| 边界值分析 | 边界附近数据 | 有最大值、最小值、范围限制 | 长度 $2,3,4,9,10,11$ |
| 决策表   | 条件组合   | 多个条件决定结果      | 登录规则、折扣规则          |
| 状态转换  | 状态变化   | 对象行为依赖当前状态    | 订单、账号、连接状态         |

> 记忆方式：输入范围用等价类，范围边缘用边界值，多条件组合用决策表，有状态变化用状态转换。

> 考试中常见题型是给出需求，让你选择合适方法并写出测试用例。
