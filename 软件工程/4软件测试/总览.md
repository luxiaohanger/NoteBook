## 软件测试的层次

软件测试的层次是指按照**测试对象的粒度和集成范围**，从小到大逐步验证软件质量。常见层次包括：**单元测试、集成测试、系统测试、验收测试**。

> 越靠前越关注“代码局部是否正确”，越靠后越关注“整个系统是否满足用户需求”。

> 考试中常考测试层次的顺序、测试对象、测试目的和责任主体。

---

## 单元测试

单元测试是对软件中最小可测试单元进行测试，例如函数、方法、类或模块。它主要验证局部逻辑是否正确。

常见测试对象：

| 对象 | 示例                    |
| -- | --------------------- |
| 函数 | `calculateTotal()`    |
| 方法 | `UserService.login()` |
| 类  | `Order` 类             |
| 模块 | 工具类、DAO、业务组件          |

示例：

```java
public int add(int a, int b) {
    return a + b;
}
```

对应单元测试：

```java
@Test
public void testAdd() {
    Calculator c = new Calculator();
    assertEquals(5, c.add(2, 3));
}
```

> 单元测试通常由开发人员编写，强调快速、自动化、可重复执行。

> 易错点：单元测试不是测试整个系统，而是测试很小的代码单元。

---

## 集成测试

集成测试是在多个单元已经分别通过测试之后，检查它们组合在一起时是否能正确协作。它主要关注模块之间的接口、数据传递和调用关系。

常见测试内容：

| 测试内容 | 示例                        |
| ---- | ------------------------- |
| 模块接口 | Controller 是否正确调用 Service |
| 数据传递 | DTO、VO、PO 转换是否正确          |
| 调用顺序 | 订单创建后是否调用库存扣减             |
| 异常传播 | DAO 抛异常后 Service 是否正确处理   |

示例：

```java
public class OrderService {
    private InventoryService inventoryService;

    public void createOrder(Order order) {
        inventoryService.reduceStock(order.getProductId(), order.getCount());
    }
}
```

集成测试关注的是：

```text
OrderService 是否正确调用 InventoryService
InventoryService 是否正确扣减库存
两个模块之间的数据是否传递正确
```

> 集成测试重点不是单个函数内部逻辑，而是模块之间“连起来是否正常”。

> 易错点：集成测试常发现接口不匹配、参数含义不一致、调用顺序错误等问题。

---

## 系统测试

系统测试是把已经集成完成的软件作为一个完整系统进行测试，检查系统是否满足需求规格说明书中的功能和非功能要求。

系统测试通常包括：

| 类型    | 关注点             |
| ----- | --------------- |
| 功能测试  | 系统功能是否符合需求      |
| 性能测试  | 响应时间、吞吐量、并发能力   |
| 安全测试  | 权限控制、数据泄露、注入攻击  |
| 兼容性测试 | 不同浏览器、系统、设备是否正常 |
| 易用性测试 | 用户操作是否方便        |
| 恢复测试  | 系统故障后能否恢复       |

示例：

对于“图书借阅系统”，系统测试会检查：

```text
用户能否登录
用户能否查询图书
用户能否借阅图书
库存是否正确减少
超期是否正确罚款
管理员能否维护图书信息
系统在高并发下是否稳定
```

> 系统测试把软件当成一个整体，验证功能、性能、安全、可靠性等方面。

> 易错点：系统测试不只测功能，也包括非功能需求。

---

## 验收测试

验收测试是从用户或客户角度出发，判断软件是否满足合同、需求规格说明书或业务目标。它通常是交付前的重要测试阶段。

常见形式：

| 类型         | 含义               |
| ---------- | ---------------- |
| Alpha 测试   | 用户在开发方环境下进行测试    |
| Beta 测试    | 用户在真实使用环境中进行测试   |
| 用户验收测试 UAT | 用户根据业务流程判断是否可以交付 |

示例：

对于“网上选课系统”，验收测试关注：

```text
学生是否能按规则选课
课程容量是否受限制
管理员是否能发布课程
退课是否符合学校规定
成绩录入是否符合教务流程
```

> 验收测试关心“用户能不能接受这个系统”，而不只是程序是否能运行。

> 易错点：验收测试通常依据需求规格说明、合同或用户业务场景进行。

---

## 测试层次对比

| 测试层次 | 测试对象      | 测试目的           | 常见执行者      |
| ---- | --------- | -------------- | ---------- |
| 单元测试 | 函数、类、模块   | 验证局部代码逻辑       | 开发人员       |
| 集成测试 | 模块之间的组合   | 验证接口和协作        | 开发人员、测试人员  |
| 系统测试 | 完整系统      | 验证系统整体功能和非功能需求 | 测试人员       |
| 验收测试 | 面向用户的完整系统 | 判断是否满足用户需求并可交付 | 用户、客户、测试人员 |

> 记忆顺序：单元测试 $\rightarrow$ 集成测试 $\rightarrow$ 系统测试 $\rightarrow$ 验收测试。

> 粒度越来越大，测试环境越来越接近真实使用环境，测试目标越来越偏向用户需求。
