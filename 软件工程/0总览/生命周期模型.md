## 软件生命周期模型

软件生命周期模型是把软件从**开始生产到最终报废**的全过程划分为若干阶段，每个阶段都有明确的典型输入、输出、主要活动和执行人。课件中给出的典型生命周期包括：需求工程、软件设计、软件构造、软件测试、软件交付、软件维护等阶段。

$$
\text{软件生命周期}=\text{需求工程}\rightarrow\text{软件设计}\rightarrow\text{软件构造}\rightarrow\text{软件测试}\rightarrow\text{软件交付}\rightarrow\text{软件维护}
$$

> 软件生命周期模型就是“软件从被提出、被开发、被使用，到最后退役的阶段划分”。

> 易错点：生命周期模型强调“有哪些阶段”；过程模型强调“这些阶段怎么组织、怎么执行、怎么控制”。课件指出，过程模型是在生命周期模型基础上进一步说明各阶段任务、活动、对象及其组织控制过程。

---

## 需求工程

需求工程阶段的目标是建立解决方案，核心是理解现实问题，并制定成本效益比有效的软件解决方案。其工作基础来自客户、用户和外部环境等，常用方法包括结构化分析和面向对象分析。

常见活动：

| 活动     | 说明                  |
| ------ | ------------------- |
| 系统需求开发 | 从业务和环境角度理解系统要解决的问题  |
| 软件需求开发 | 明确软件系统应该提供哪些功能和质量要求 |
| 需求管理   | 维护需求基线，处理需求变更       |

主要制品：

| 制品     | 说明                         |
| ------ | -------------------------- |
| 需求分析模型 | 如 DFD、ERD、用例图、概念类图、顺序图、状态图 |
| 需求文档   | 对系统需求和软件需求的正式描述            |
| 需求基线   | 被确认后的需求版本，后续开发以此为依据        |

> 需求工程回答“系统到底要解决什么问题”。

> 易错点：需求工程不只是“写需求文档”，还包括需求开发和需求管理；需求变更后要维护需求基线。

---

## 软件设计

软件设计阶段的目标是建立由抽象软件实体组成的软件结构。它把需求转化为系统结构、模块结构、接口和人机交互方案。课件中把软件设计进一步分为体系结构设计、详细设计和人机交互设计。

主要活动：

| 活动     | 关注点                   | 常见制品         |
| ------ | --------------------- | ------------ |
| 体系结构设计 | 整体功能组织与质量属性           | 体系结构模型、原型和文档 |
| 详细设计   | 模块内部结构、可开发性、可复用性、可维护性 | 详细设计模型和文档    |
| 人机交互设计 | 易用性和用户交互质量            | 界面原型         |

常用方法：

| 方法     | 说明            |
| ------ | ------------- |
| UML 包图 | 表示模块、包和层次结构   |
| 构件图    | 表示构件及其接口关系    |
| 部署图    | 表示软件如何部署到硬件节点 |
| 类图     | 表示类、属性、方法和关系  |
| 顺序图    | 表示对象之间的消息交互   |

> 软件设计回答“系统应该怎么组织、模块之间怎么协作”。

> 易错点：体系结构设计偏“整体结构和质量属性”，详细设计偏“模块内部结构和实现准备”，人机交互设计偏“界面和用户体验”。

---

## 软件构造

软件构造阶段的目标是把设计方案转化为可运行的软件。它关注程序质量，包括可读性、可靠性、性能和可维护性等。主要任务包括编程、集成、测试与调试、构建等。

主要活动：

| 活动   | 说明            |
| ---- | ------------- |
| 编程   | 根据设计方案编写源代码   |
| 集成   | 将多个模块或构件组合成系统 |
| 调试   | 定位和修复程序错误     |
| 构建   | 生成可执行程序或可部署产物 |
| 单元测试 | 验证局部代码单元是否正确  |

主要制品：

| 制品    | 说明           |
| ----- | ------------ |
| 源代码   | 程序员编写的代码     |
| 可执行程序 | 编译、构建后可运行的软件 |

> 软件构造就是“把设计图纸变成能运行的程序”。

> 易错点：软件构造不等于单纯编码，还包括集成、调试、构建和部分测试活动。

---

## 软件测试

软件测试阶段的目标是保障软件产品质量，重点验证程序正确性以及软件是否符合需求。测试工作以软件需求、软件设计和程序代码为基础，常用黑盒测试、白盒测试等方法。

测试层次：

| 测试层次 | 说明              | 常见执行者    |
| ---- | --------------- | -------- |
| 单元测试 | 测试单个函数、类或模块     | 程序员或测试人员 |
| 集成测试 | 测试多个模块组合后的接口和协作 | 测试人员     |
| 系统测试 | 测试整个系统是否满足需求    | 测试人员     |

主要制品：

| 制品        | 说明              |
| --------- | --------------- |
| 测试计划      | 说明测试范围、方法、资源和安排 |
| 测试报告      | 记录测试结果、缺陷和质量判断  |
| 通过测试的软件产品 | 满足交付条件的软件版本     |

> 软件测试就是“检查软件是不是做对了、是不是符合需求”。

> 易错点：测试不只在编码之后才出现，测试计划可能在设计和构造阶段就开始准备；单元测试通常和构造阶段联系很紧密。

---

## 软件交付

软件交付阶段的目标是把通过测试的软件产品交给用户，使用户能够实际使用。其关注点是交付的有效性，主要任务包括安装部署、用户培训和文档支持。

主要活动：

| 活动    | 说明                |
| ----- | ----------------- |
| 安装与部署 | 把软件部署到目标环境        |
| 用户培训  | 教用户如何使用系统         |
| 文档支持  | 提供用户手册、安装说明、运维文档等 |

主要制品：

| 制品      | 说明           |
| ------- | ------------ |
| 已交付软件产品 | 用户可以使用的软件    |
| 用户使用手册  | 帮助用户理解和操作系统  |
| 部署文档    | 说明安装、配置和运行环境 |

> 软件交付就是“把开发完成的软件真正送到用户手里并让它能用”。

> 易错点：交付不只是“发一个安装包”，还包括部署、培训和文档支持。

---

## 软件维护

软件维护阶段发生在软件交付之后，目标是在产品终结前保证软件正常使用。维护阶段需要在变更成本、质量和收益之间进行平衡，常用方法包括重构、逆向工程、再工程等。

维护类型可以这样理解：

| 类型    | 说明                    |
| ----- | --------------------- |
| 改正性维护 | 修复交付后发现的缺陷            |
| 适应性维护 | 适应操作系统、数据库、硬件、政策等环境变化 |
| 完善性维护 | 增加新功能或改进已有功能          |
| 预防性维护 | 改善结构，降低未来维护难度         |

常见维护方法：

| 方法   | 说明                 |
| ---- | ------------------ |
| 重构   | 不改变外部行为，改善内部结构     |
| 逆向工程 | 从已有系统中恢复设计、需求或结构信息 |
| 再工程  | 对旧系统进行分析、重构和重新实现   |

> 软件维护就是“软件交付后，继续修、改、扩展和优化”。

> 易错点：维护不是生命周期之外的事情，而是生命周期的重要阶段；在演化模型中，维护和新开发的边界会变得模糊。
