## 软件过程模型

软件过程模型是在软件生命周期模型基础上，进一步说明各阶段的任务、活动、对象，以及这些活动如何被组织和控制。生命周期模型更强调“有哪些阶段”，过程模型更强调“这些阶段怎么执行、怎么衔接、怎么管理”。

---
## 各过程模型对比

| 模型    | 驱动因素    | 核心特点        | 适用场景        | 主要风险           |
| ----- | ------- | ----------- | ----------- | -------------- |
| 构建-修复 | 个人编码    | 先写再修        | 小程序、低质量要求   | 结构恶化、难维护       |
| 瀑布模型  | 文档和阶段   | 顺序开发、阶段验证   | 需求稳定、技术成熟   | 需求变化难处理        |
| 增量迭代  | 增量需求    | 分批交付、逐步扩展   | 稳定领域的大型系统   | 需要开放架构和清晰范围    |
| 演化模型  | 用户反馈    | 持续演化、适应变化   | 不稳定领域的大型系统  | 容易退化为构建-修复     |
| 原型模型  | 原型验证    | 用原型澄清需求     | 新颖领域、需求不清   | 原型成本高、劣质代码进入产品 |
| 螺旋模型  | 风险      | 每轮先分析和解决风险  | 高风险大型项目     | 管理复杂、依赖风险识别    |
| RUP   | 最佳实践    | 可裁剪的重量级过程框架 | 大型团队、大型项目   | 裁剪困难、过程成本高     |
| 敏捷过程  | 用户价值和变化 | 快速交付、响应变化   | 需求变化快、时间压力大 | 依赖团队能力，文档可能不足  |

---

## 构建-修复模型

构建-修复模型是最早、最自然的软件开发方式：先构建第一个版本，再不断修复和维护。它严格来说不算完整的软件过程模型，因为它几乎没有对开发活动进行规划和组织，主要依赖开发人员个人能力。

基本流程：

1. **构建**：直接写出第一个可运行版本；
2. **修复**：发现问题后不断修改；
3. **维护**：继续在已有代码上打补丁。

优点：

| 方面 | 说明              |
| -- | --------------- |
| 简单 | 不需要复杂文档、流程和管理   |
| 快速 | 小程序可以直接编码，启动成本低 |
| 灵活 | 开发者可以随时改动       |

缺点：

| 缺点      | 说明                 |
| ------- | ------------------ |
| 缺少需求分析  | 没有验证需求真实性，风险很大     |
| 缺少设计    | 软件结构会在不断修改中恶化      |
| 缺少测试和文档 | 维护困难，质量难保证         |
| 不适合复杂系统 | 系统复杂度超过个人控制能力后容易失败 |

适用场景：

| 场景          | 是否适合 |
| ----------- | ---- |
| 几百行的小程序     | 适合   |
| 质量要求不高的临时工具 | 适合   |
| 大型系统        | 不适合  |
| 长期维护系统      | 不适合  |

> 构建-修复就是“先写出来，坏了再改”。

> 易错点：构建-修复模型不是规范化的软件过程模型，考试中常作为反面例子出现。

---

## 瀑布模型

瀑布模型把软件开发划分为线性顺序阶段，典型阶段包括需求开发、软件设计、软件实现、软件测试、软件交付和软件维护。它强调每个阶段的结果都要经过验证，且以文档为核心来组织和控制开发活动。

基本流程：

1. **需求开发**：明确系统要做什么；
2. **软件设计**：确定系统结构和模块设计；
3. **软件实现**：编码实现设计方案；
4. **软件测试**：验证软件是否正确、是否满足需求；
5. **软件交付**：安装部署并交给用户；
6. **软件维护**：交付后修改缺陷、适应变化。

优点：

| 优点    | 说明               |
| ----- | ---------------- |
| 阶段清晰  | 每个阶段有明确输入、输出和任务  |
| 便于管理  | 适合用里程碑、文档和评审进行控制 |
| 关注点分离 | 需求、设计、实现、测试分开处理  |
| 质量较可控 | 每阶段产物都需要验证       |

缺点：

| 缺点      | 说明                   |
| ------- | -------------------- |
| 过度依赖文档  | 文档成本高，且难以准确描述变化中的需求  |
| 线性假设不现实 | 实际开发常常需要后续工作才能验证前面阶段 |
| 用户参与不足  | 用户通常只在需求和验收阶段参与较多    |
| 里程碑粒度粗  | 缺陷发现晚，修复成本高          |

适用场景：

| 场景       | 是否适合 |
| -------- | ---- |
| 需求成熟稳定   | 适合   |
| 技术成熟可靠   | 适合   |
| 复杂度适中    | 适合   |
| 需求频繁变化   | 不适合  |
| 高风险探索型项目 | 不适合  |

> 瀑布模型就是“一个阶段做完并验证后，再进入下一个阶段”。

> 易错点：瀑布模型不是完全不允许返回修改，课件强调它允许活动反复和迭代，但核心仍是文档驱动、阶段顺序明显。

## 增量迭代模型

增量迭代模型把系统分成多个相对独立的增量，每次迭代完成一部分可运行、可交付的软件。它通常在项目早期确定整体目标和范围，然后通过多次迭代逐步交付功能。

基本流程：

1. **确定目标和范围**：项目早期明确总体方向；
2. **确定核心体系结构**：先设计能支撑后续增量的架构；
3. **开发增量 1**：完成核心功能或基本需求；
4. **开发增量 2 到 n**：不断加入新功能；
5. **渐进交付**：每个增量都可以交付、测试和反馈。

特点：

| 特点    | 说明              |
| ----- | --------------- |
| 迭代式开发 | 系统不是一次完成，而是多轮完成 |
| 渐进交付  | 每次交付一个可运行的软件增量  |
| 可并行开发 | 不同增量或构件可以并行开发   |
| 需求驱动  | 增量需求是迭代规划的主要依据  |

优点：

| 优点     | 说明           |
| ------ | ------------ |
| 符合实践   | 复杂系统通常难以一次完成 |
| 缩短交付时间 | 可以先交付核心功能    |
| 加强用户反馈 | 用户可以早期使用并反馈  |
| 降低风险   | 问题可以在较早增量中暴露 |

缺点：

| 缺点         | 说明             |
| ---------- | -------------- |
| 要求开放式体系结构  | 新构件加入不能破坏已有系统  |
| 需要清晰项目前景   | 早期要能确定范围和总体规划  |
| 不适合高度不稳定领域 | 需求变化太频繁时难以规划增量 |

适用场景：

| 场景       | 是否适合 |
| -------- | ---- |
| 大规模软件系统  | 适合   |
| 需求比较成熟稳定 | 适合   |
| 希望分阶段交付  | 适合   |
| 需求极不稳定   | 不太适合 |

> 增量迭代就是“先做核心，再一块一块加功能，每块都能交付”。

> 易错点：增量迭代模型不是完全没有前期规划，它要求早期明确目标、范围和核心体系结构。

## 演化模型

演化模型也是迭代、并行开发和渐进交付，但它比增量迭代模型更能应对频繁需求变更。它模糊了“新开发”和“维护”的界限，系统在持续反馈中不断演化。

基本流程：

1. **初始开发**：先开发一个初始版本；
2. **交付演化 1**：交付后获得用户反馈；
3. **维护和修改**：根据反馈调整需求、设计和实现；
4. **交付演化 2 到 n**：不断修改、扩展和再交付；
5. **持续演化**：系统在使用中持续成长。

与增量迭代模型的区别：

| 对比项   | 增量迭代模型  | 演化模型      |
| ----- | ------- | --------- |
| 需求稳定性 | 需求较成熟稳定 | 需求变化频繁    |
| 前期范围  | 需要较明确范围 | 早期范围可能不清晰 |
| 迭代目标  | 按计划增加功能 | 根据反馈不断演化  |
| 适用领域  | 成熟稳定领域  | 不稳定领域     |

优点：

| 优点      | 说明           |
| ------- | ------------ |
| 适应需求变化  | 适合不确定性较多的系统  |
| 用户反馈充分  | 每次演化都可根据反馈调整 |
| 降低风险    | 可以在不断演化中修正方向 |
| 支持大规模系统 | 适合复杂系统持续发展   |

缺点：

| 缺点         | 说明                |
| ---------- | ----------------- |
| 早期范围难确定    | 计划、进度和商务协商难准确     |
| 容易退化为构建-修复 | 后续迭代若忽略分析设计，质量会下降 |
| 管理难度较高     | 需求和计划持续变化         |

适用场景：

| 场景         | 是否适合 |
| ---------- | ---- |
| 不稳定领域的大型系统 | 适合   |
| 用户需求变化频繁   | 适合   |
| 需要持续反馈和调整  | 适合   |
| 合同范围必须固定   | 不太适合 |

> 演化模型就是“软件先活起来，再根据真实反馈不断长大”。

> 易错点：演化模型不是随便改代码，仍然需要每轮演化中的需求、设计、实现和测试，否则会退化成构建-修复。

## 原型模型

原型模型通过构造原型来解决不确定性。课件区分了抛弃式原型和演化式原型：抛弃式原型用于探索和验证需求，最后不进入产品；演化式原型则在迭代中不断扩充，最终成为真正的软件产品。

基本流程：

1. **原型需求**：找出不清楚、不确定的需求；
2. **设计原型**：快速设计一个模拟版本；
3. **构建原型**：实现可演示、可验证的原型；
4. **评估原型**：让客户和用户反馈；
5. **明确需求**：解决不确定性；
6. **正式开发**：需求清晰后可按瀑布等方式开发。

两类原型：

| 类型    | 含义            | 是否进入最终产品 |
| ----- | ------------- | -------- |
| 抛弃式原型 | 用来探索需求，验证完就丢弃 | 否        |
| 演化式原型 | 从核心原型不断扩充成产品  | 是        |

优点：

| 优点     | 说明              |
| ------ | --------------- |
| 加强用户交流 | 用户可以看到接近未来产品的东西 |
| 需求更清晰  | 能提前发现误解和遗漏      |
| 适合新颖领域 | 对未知问题探索能力强      |

缺点：

| 缺点         | 说明               |
| ---------- | ---------------- |
| 原型成本可能较高   | 可能消耗大量时间和费用      |
| 抛弃式原型容易被保留 | 低质量原型代码进入产品会降低质量 |
| 容易误导用户     | 用户可能把原型误认为最终系统   |

适用场景：

| 场景        | 是否适合    |
| --------- | ------- |
| 新颖领域      | 适合      |
| 需求不确定性大   | 适合      |
| 用户界面需求不清楚 | 适合      |
| 需求非常稳定    | 没必要重点使用 |

> 原型模型就是“先做一个样子给用户看，看清楚后再正式做”。

> 易错点：抛弃式原型的重点是“抛弃”，不能把临时代码直接当成正式产品代码。

## 螺旋模型

螺旋模型是风险驱动的过程模型。它的基本思想是尽早识别并解决高风险问题，如果风险无法解决，也应尽早发现，避免到项目后期才暴露造成更大损失。课件强调，螺旋模型充分利用原型方法解决风险，并且是迭代与瀑布的结合：开发阶段类似瀑布，风险分析是迭代的。

基本流程：

1. **确定目标、方案和约束**：明确本轮迭代要解决什么；
2. **评估方案并发现风险**：分析需求、技术、架构、成本等风险；
3. **寻找风险解决方法**：常通过原型、实验、评审解决不确定性；
4. **落实风险解决方案**：完成本轮开发和验证；
5. **计划下一轮迭代**：进入下一圈螺旋。

核心公式化理解：

$$
\text{螺旋模型}=\text{迭代开发}+\text{风险分析}+\text{原型验证}
$$

与原型模型的区别：

| 对比项  | 原型模型     | 螺旋模型           |
| ---- | -------- | -------------- |
| 主要目标 | 解决需求不确定性 | 解决各种项目风险       |
| 风险范围 | 主要是需求风险  | 需求、技术、架构、设计等风险 |
| 驱动力  | 原型驱动需求澄清 | 风险驱动过程组织       |

优点：

| 优点     | 说明        |
| ------ | --------- |
| 降低风险   | 高风险问题尽早处理 |
| 适合复杂项目 | 能处理多类不确定性 |
| 结合原型优势 | 用原型验证关键问题 |

缺点：

| 缺点        | 说明             |
| --------- | -------------- |
| 管理复杂      | 模型本身复杂，不易组织    |
| 依赖风险识别能力  | 风险识别不准会影响效果    |
| 原型也有成本和风险 | 与原型模型类似，可能消耗资源 |

适用场景：

| 场景        | 是否适合 |
| --------- | ---- |
| 高风险大型软件系统 | 适合   |
| 技术不确定性强   | 适合   |
| 架构风险较高    | 适合   |
| 小型简单项目    | 不适合  |

> 螺旋模型就是“每转一圈都先看风险，先解决最危险的问题”。

> 易错点：螺旋模型不是单纯的迭代模型，它的关键词是“风险驱动”。

## Rational 统一过程模型

Rational 统一过程，即 RUP，是一种总结传统最佳实践并支持过程裁剪的通用过程框架。它不是固定死的单一流程，而是允许开发者根据项目特点定制适合自己的过程模型。

RUP 的四个阶段：

| 阶段 | 主要目标             |
| -- | ---------------- |
| 初始 | 明确项目范围、商业目标和可行性  |
| 细化 | 稳定需求和体系结构，降低主要风险 |
| 构造 | 大规模实现、测试并形成产品    |
| 交付 | 部署、交付给用户并修正问题    |

RUP 的核心工作流：

| 工作流     | 说明           |
| ------- | ------------ |
| 商业建模    | 理解业务背景       |
| 需求      | 获取、分析和管理需求   |
| 分析和设计   | 建立软件结构和设计模型  |
| 实现      | 编码和集成        |
| 测试      | 验证软件质量       |
| 部署      | 发布和交付        |
| 配置和变更管理 | 管理版本与变更      |
| 项目管理    | 计划和控制项目      |
| 环境      | 工具、规范和开发环境支持 |

RUP 六个核心实践方法：

| 实践        | 说明                |
| --------- | ----------------- |
| 迭代式开发     | 分多轮开发，降低一次性失败风险   |
| 管理需求      | 不只开发需求，还要持续管理需求变化 |
| 基于组件的体系结构 | 提高可维护、可复用和可开发性    |
| 可视化建模     | 使用 UML 等模型表达系统    |
| 验证软件质量    | 尽早、持续地开展验证        |
| 控制软件变更    | 管理需求、代码和配置变化      |

优点：

| 优点     | 说明            |
| ------ | ------------- |
| 吸收最佳实践 | 过程组织较成熟       |
| 可裁剪    | 可根据项目规模和特点定制  |
| 适用面广   | 大项目、小项目都可调整使用 |
| 工具支持强  | 有配套工程工具支持     |

缺点：

| 缺点           | 说明               |
| ------------ | ---------------- |
| 没有充分考虑交付后的维护 | 维护问题不是核心重点       |
| 裁剪困难         | 不一定每个项目都能裁剪出有效过程 |
| 偏重量级         | 对小团队可能成本较高       |

适用场景：

| 场景         | 是否适合  |
| ---------- | ----- |
| 大型团队开发大型项目 | 适合    |
| 需要规范过程管理   | 适合    |
| 可以投入过程裁剪成本 | 适合    |
| 极小型快速项目    | 不一定适合 |

> RUP 就是“把成熟软件工程实践打包成一个可裁剪的过程框架”。

> 易错点：RUP 不是一个固定流程，而是一个需要裁剪和配置的过程模板。

---

## 敏捷过程

敏捷过程不是某一个固定模型，而是一组思想、原则和轻量级过程方法。它反对传统过程中过度强调计划、文档、工具和纪律而牺牲灵活性，强调快速交付、客户合作和响应变化。课件列出了敏捷宣言的四组价值观：个体和互动高于流程和工具，工作的软件高于详尽文档，客户合作高于合同谈判，响应变化高于遵循计划。

敏捷宣言核心价值观：

| 更重视   | 相比之下  |
| ----- | ----- |
| 个体和互动 | 流程和工具 |
| 工作的软件 | 详尽的文档 |
| 客户合作  | 合同谈判  |
| 响应变化  | 遵循计划  |

注意：右项也有价值，只是敏捷更重视左项。

敏捷原则关键词：

| 关键词   | 含义               |
| ----- | ---------------- |
| 持续交付  | 尽早、持续交付有价值的软件    |
| 欢迎变化  | 即使开发后期也可以接受需求变化  |
| 短周期   | 几周或一两个月交付一次可工作软件 |
| 客户协作  | 业务人员和开发人员持续合作    |
| 自组织团队 | 最好的架构、需求和设计来自团队  |
| 技术卓越  | 良好设计和持续改进增强敏捷能力  |
| 可持续开发 | 团队保持稳定节奏         |
| 定期反思  | 团队根据反馈调整做法       |

常见敏捷方法：

| 方法    | 说明                     |
| ----- | ---------------------- |
| XP    | 极限编程，强调测试驱动、结对编程、持续集成等 |
| Scrum | 以 Sprint 为核心的迭代管理方法    |
| FDD   | 特性驱动开发                 |
| ASD   | 自适应软件开发                |
| DSDM  | 动态系统开发方法               |

优点：

| 优点    | 说明           |
| ----- | ------------ |
| 适应变化  | 能快速响应需求变化    |
| 交付快   | 重视短周期交付可工作软件 |
| 用户参与强 | 客户持续参与反馈     |
| 团队灵活  | 强调个体、互动和自组织  |

缺点：

| 缺点           | 说明             |
| ------------ | -------------- |
| 对团队能力要求高     | 需要自律、沟通和技术能力   |
| 文档可能不足       | 后期维护和交接可能受影响   |
| 大型项目治理较难     | 多团队协作时需要额外管理机制 |
| 不适合强合同固定范围项目 | 变化频繁可能与合同冲突    |

适用场景：

| 场景           | 是否适合  |
| ------------ | ----- |
| 需求快速变化       | 适合    |
| 时间压力大        | 适合    |
| 用户能持续参与      | 适合    |
| 小到中型团队       | 适合    |
| 严格固定需求和流程的项目 | 不一定适合 |

> 敏捷就是“少做无效流程，快速交付可用软件，并根据反馈不断调整”。

> 易错点：敏捷不是不要文档、不要计划、不要设计，而是反对过度文档、僵化计划和脱离用户价值的设计。

## 极限编程 XP

极限编程是敏捷方法的一种。它的思想是把简单有效的方法用到“极限”：如果单元测试有用，就一直做测试；如果代码评审有用，就结对编程；如果集成测试有用，就持续集成；如果短迭代有用，就小版本发布。

XP 常见实践：

| 活动   | 实践          | 含义                  |
| ---- | ----------- | ------------------- |
| 迭代规划 | 规划游戏        | 开发者估算成本，客户按价值选择特性   |
| 需求开发 | 现场客户        | 用户代表始终参与开发          |
| 软件设计 | 简单设计        | 只做满足当前需求的设计         |
| 软件设计 | 重构          | 不改变外部行为，改善内部结构      |
| 软件实现 | 结对编程        | 两人共用一台电脑，一个写代码，一个评审 |
| 软件实现 | 编码规范        | 所有人遵循统一编码标准         |
| 软件实现 | 代码集体所有      | 每个人都可以修改任意代码        |
| 软件测试 | 测试驱动        | 先写测试，再写实现           |
| 软件测试 | 持续集成        | 频繁集成，每次集成都跑测试       |
| 软件交付 | 小版本发布       | 频繁发布可工作软件           |
| 其他   | 每周 40 小时工作制 | 保持可持续开发能力           |

TDD 三步：

1. **Red**：先写失败的测试；
2. **Green**：写最简单代码让测试通过；
3. **Refactor**：重构代码，保持测试通过。

$$
\text{TDD}=\text{Red}\rightarrow\text{Green}\rightarrow\text{Refactor}
$$

> XP 就是“把测试、重构、集成、用户参与这些好实践做到极致”。

> 易错点：结对编程不是一个人写、另一个人旁观，而是一个人驾驶编码，另一个人导航评审，角色可以互换。
