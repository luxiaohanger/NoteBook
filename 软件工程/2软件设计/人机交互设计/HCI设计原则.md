## 人机交互设计原则

人机交互设计，即 HCI Design，目标是让人与计算机界面之间的交互更高效、自然、可用。好的界面不只是“功能完整”，而是让用户能够顺利完成任务，并且尽量感觉不到系统本身的复杂性。课件中强调：好的界面应符合用户预期，允许用户完成所需任务，并且最好是“透明的”，让用户忘记自己正在使用计算机。

> 人机交互设计的核心不是让用户适应系统，而是让系统适应用户。

> 考试中要把“界面美观”和“交互可用”区分开：好看不等于好用。

---

## 可用性原则

可用性是人机交互设计的核心目标，不是单一属性，而是多个指标的综合。课件将可用性分为五个方面：易学性、效率、易记性、容错性和满意度。

| 属性  | 含义           | 设计关注点          |
| --- | ------------ | -------------- |
| 易学性 | 用户能快速学会操作    | 新手引导、清晰导航、直观按钮 |
| 效率  | 熟练用户能高效完成任务  | 快捷键、批量操作、默认值   |
| 易记性 | 用户隔一段时间后仍能使用 | 操作路径稳定、图标语义清楚  |
| 容错性 | 错误少，且能恢复     | 撤销、确认、错误提示     |
| 满意度 | 用户使用过程愉悦     | 响应快、界面清爽、反馈友好  |

> 可用性就是用户能不能“学得会、用得快、记得住、错得少、愿意用”。

> 易学性和效率可能冲突：新手喜欢简单直观，专家喜欢快捷高效，设计时要平衡。

---

## 让用户掌控系统

让用户掌控系统是经典的黄金原则之一。用户应该能够控制交互节奏，知道当前发生了什么，并且能够撤销、重做或退出当前操作。课件将其概括为 “Place the user in control”。

常见设计方式包括：

| 设计方式      | 作用             |
| --------- | -------------- |
| 撤销 Undo   | 用户误操作后可以恢复     |
| 重做 Redo   | 用户可以重新执行被撤销的操作 |
| 取消 Cancel | 长任务或危险任务可中止    |
| 返回 Back   | 用户可以回到上一状态     |
| 明确出口      | 用户知道如何退出页面或流程  |

> 用户掌控系统，就是不要让用户觉得自己被系统“困住”。

> 危险操作必须尤其注意，例如删除、支付、提交等操作，通常需要确认或提供撤销机制。

---

## 减少用户记忆负担

人的短时记忆有限，课件中提到 $7 \pm 2$ 原则，并强调“识别优于回忆”：人更容易从可见选项中识别目标，而不是凭记忆输入命令。

设计方法包括：

| 方法      | 说明              |
| ------- | --------------- |
| 让选项可见   | 用菜单、按钮、图标代替记忆命令 |
| 提供默认值   | 系统预填常见选项        |
| 分层展示    | 复杂信息逐步展开        |
| 自动保存历史  | 系统替用户记住最近使用内容   |
| 使用图形和图标 | 图像记忆通常优于文字记忆    |

> 好界面不是考用户记忆力，而是把用户需要的信息放在合适位置。

> 命令行效率高，但记忆负担大；图形界面易学，因为用户更多是在“识别”而不是“回忆”。

---

## 保持一致性

一致性是指相似的操作应该有相似的表现和结果。课件中的黄金原则将其概括为 “Be consistent”。Nielsen 启发式原则中也强调“一致性与标准”，即界面应遵循平台规范。

一致性包括：

| 类型   | 示例                         |
| ---- | -------------------------- |
| 视觉一致 | 相同功能使用相同颜色、图标、按钮样式         |
| 操作一致 | 同类页面的提交、返回、取消位置一致          |
| 语义一致 | “删除”始终表示删除，不在某处表示隐藏        |
| 平台一致 | 移动端遵守 iOS / Android 常见交互习惯 |
| 反馈一致 | 成功、失败、警告提示风格统一             |

> 一致性可以降低学习成本，让用户把一个地方学会的经验迁移到另一个地方。

> 不一致会让用户怀疑自己操作错了，考试中常作为界面缺陷分析点。

---

## 系统状态可见

系统应该持续告诉用户当前状态，例如是否正在加载、是否保存成功、操作是否失败。Nielsen 第一条启发式原则就是“系统状态可见”。

常见方式包括：

| 场景    | 反馈方式              |
| ----- | ----------------- |
| 页面加载  | 进度条、骨架屏、加载动画      |
| 表单提交  | “提交中”“提交成功”“提交失败” |
| 文件上传  | 上传进度百分比           |
| 后台任务  | 状态栏、通知、日志         |
| 长时间计算 | 剩余时间估计、中断按钮       |

> 用户最怕的不是等待，而是不知道系统有没有在工作。

> 响应时间超过用户预期时，必须给出反馈；否则用户可能重复点击或直接放弃。

---

## 匹配用户的现实世界

界面应使用用户熟悉的词语、概念和任务模型，而不是暴露系统内部结构。课件在可视化设计部分强调：应按照任务模型设计界面隐喻，不要把软件系统的内部构造机制暴露给用户。

例如：

| 不好的设计          | 更好的设计                |
| -------------- | -------------------- |
| “执行数据库事务”      | “保存订单”               |
| “写入对象状态”       | “保存修改”               |
| “HTTP 500”     | “服务器暂时无法处理，请稍后重试”    |
| “选择操作类型 1/2/3” | “压缩文件 / 解压文件 / 查看文件” |

> 用户关心的是完成任务，不关心系统内部怎么实现。

> 考试中遇到“暴露内部机制”的界面，一般应改成面向用户任务的表达。

## 防止错误

好的交互设计应优先防止错误，而不是等用户犯错后再提示。课件中协作式设计强调低出错率设计，包括屏蔽不适当选项、禁止非法输入、出错时提供建设性指导。

常见方法包括：

| 方法     | 示例              |
| ------ | --------------- |
| 禁用非法按钮 | 未选择文件时，“删除”按钮置灰 |
| 限制输入格式 | 手机号输入框只允许数字     |
| 二次确认   | 删除账号前要求确认       |
| 实时校验   | 密码长度不足时即时提示     |
| 合理默认值  | 避免用户误选危险配置      |

> 防错比报错更重要。

> 错误提示不能只说“错误”，要说明哪里错、为什么错、怎么改。

---

## 良好的错误提示

课件总结了出错信息四原则：清晰语言、精炼准确、建设性帮助、友好态度。

| 原则    | 含义       | 示例                                                   |
| ----- | -------- | ---------------------------------------------------- |
| 清晰语言  | 不使用晦涩错误码 | “密码至少需要 8 位”                                         |
| 精炼准确  | 不模糊、不空泛  | “邮箱格式不正确”                                            |
| 建设性帮助 | 给出解决方案   | “请使用 [name@example.com](mailto:name@example.com) 格式” |
| 友好态度  | 不责备用户    | “请检查输入内容”                                            |

错误提示反例：

```text
Error 0x00043
```

改进后：

```text
上传失败：文件大小超过 20MB，请压缩后重新上传。
```

> 好的错误提示应该像助手，而不是像警告牌。

> 错误信息不要威胁用户，也不要把技术细节直接丢给用户。

## 识别优于回忆

识别优于回忆是 Nielsen 启发式原则之一，也与人的记忆特点直接相关。课件指出，人识别事物的能力强于回忆事物的能力，这也是 GUI 通常比命令行更适合普通用户的重要原因。

设计体现包括：

| 设计    | 作用         |
| ----- | ---------- |
| 菜单    | 用户不必记命令    |
| 图标    | 用户通过视觉识别功能 |
| 自动补全  | 用户不必完整输入   |
| 最近使用  | 用户不必重新查找   |
| 面包屑导航 | 用户知道当前位置   |

> 让用户“看见可选项”，比要求用户“记住命令”更友好。

> 但专家用户可能更偏好快捷键和命令，因此识别和效率之间也要平衡。

## 灵活性与效率

界面应同时照顾新手、熟练用户和专家用户。课件将用户分为三类：新手用户、熟练用户和专家用户，并指出不同用户需要不同交互机制。

| 用户类型 | 设计侧重            |
| ---- | --------------- |
| 新手用户 | 易学性、引导、图形界面     |
| 熟练用户 | 易学性和效率折中        |
| 专家用户 | 快捷键、命令行、批量操作、热键 |

例如，一个编辑器可以同时提供：

```text
菜单：文件 -> 保存
按钮：保存图标
快捷键：Ctrl + S
命令：save
```

> 同一个功能可以有多种入口，新手靠菜单，专家靠快捷键。

> 只为新手设计会降低效率，只为专家设计会提高学习门槛。

## 简约设计

简约设计不是功能少，而是界面中不放无关信息。Nielsen 原则中强调“简约设计”，课件 GUI 原则也提到 “Few is much”，即少即是多。

设计要求包括：

| 要求     | 说明            |
| ------ | ------------- |
| 去除无关信息 | 不让次要内容干扰主要任务  |
| 突出主操作  | 主按钮视觉权重更高     |
| 控制层级   | 信息分组清晰        |
| 减少弹窗   | 避免频繁打断用户      |
| 避免文字墙  | 用表格、列表、卡片组织信息 |

> 每多一个无关元素，用户注意力就会被稀释一点。

> 简约不是空白，而是让重要内容更容易被发现。

## 及时反馈

反馈是交互设计的重要原则。课件给出了 Shneiderman 的响应时间经验准则：打字、光标移动、鼠标定位应在 $50 \sim 150$ 毫秒内响应；简单频繁任务约 $1$ 秒；普通任务约 $2 \sim 4$ 秒；复杂任务约 $8 \sim 12$ 秒；超过 $15$ 秒通常具有破坏性。

| 响应时间            | 用户感受       |
| --------------- | ---------- |
| $50 \sim 150ms$ | 近似即时       |
| $1s$            | 基本流畅       |
| $2 \sim 4s$     | 可以接受，但需要提示 |
| $8 \sim 12s$    | 必须显示进度     |
| $>15s$          | 用户可能焦虑或放弃  |

> 反馈让用户知道操作被系统接收了。

> 长任务没有反馈时，用户常会重复点击，导致重复提交或系统异常。

## 合理导航

导航帮助用户快速找到任务入口，并知道自己当前所在位置。课件将导航分为全局结构和局部结构：全局结构按任务模型组织功能，局部结构通过布局、颜色、字体等制造视觉线索。

| 导航层次 | 作用       | 示例               |
| ---- | -------- | ---------------- |
| 全局导航 | 组织主要功能入口 | 顶部导航栏、侧边栏、底部 Tab |
| 局部导航 | 指引当前页面操作 | 按钮位置、颜色、字号、面包屑   |

常见导航控件：

| 控件     | 适合场景    |
| ------ | ------- |
| 顶部导航栏  | 全局入口较少  |
| 侧边栏    | 多层级系统   |
| 底部 Tab | 移动端核心功能 |
| 面包屑    | 深层页面    |
| 搜索框    | 信息密集系统  |

> 好导航就像餐厅菜单，让用户快速找到自己想要的东西。

> 导航结构应符合用户任务模型，而不是开发者的模块划分。

## 符合人的物理和认知特性

人机交互设计必须考虑人的记忆、感知、注意力、动作能力等特点。课件提到 Fitts 定律：目标越大、越近，点击越快。

$$
T \text{ 与 } D \text{ 正相关，与 } W \text{ 负相关}
$$

其中：

| 参数  | 含义           |
| --- | ------------ |
| $T$ | 完成指向动作所需时间   |
| $D$ | 光标到目标的距离     |
| $W$ | 目标宽度，即点击容忍范围 |

设计启示：

| 原则        | 示例             |
| --------- | -------------- |
| 重要按钮要大    | “提交”“保存”按钮不宜过小 |
| 高频操作要近    | 移动端主操作放在拇指舒适区  |
| 危险操作要远    | “删除”远离“保存”     |
| 屏幕边缘适合放菜单 | 边缘等效点击范围更大     |

> 越重要、越常用的操作，越应该容易点到。

> 危险按钮不能和常用按钮挨得太近，否则容易误触。

## 可访问性设计

可访问性设计，也叫 Accessibility 或 A11y，要求系统能被更多类型的用户使用。课件总结了 WCAG 的 POUR 四原则：可感知、可操作、可理解、健壮性。

| 原则  | 含义        | 示例                 |
| --- | --------- | ------------------ |
| 可感知 | 用户能感知信息   | 图片提供 alt 文本，颜色对比足够 |
| 可操作 | 用户能完成操作   | 所有功能可用键盘访问         |
| 可理解 | 信息和操作容易理解 | 表单错误可识别并可修复        |
| 健壮性 | 能兼容辅助技术   | 支持屏幕阅读器            |

常见要求包括：

| 要求          | 说明         |
| ----------- | ---------- |
| 不只靠颜色传达信息   | 色盲用户也能理解   |
| 表单控件有 label | 屏幕阅读器可读    |
| 正文字号足够大     | 通常不小于 16px |
| 支持键盘 Tab 遍历 | 不依赖鼠标      |
| 动画可关闭       | 照顾前庭障碍用户   |

> 可访问性不是额外功能，而是好设计的一部分。

> “红色表示错误，绿色表示正确”如果没有文字或图标辅助，对色盲用户不友好。

## AI 时代的人机交互原则

AI / LLM 时代的人机交互从传统 GUI 转向对话式、多模态、Agent 式交互。课件指出，传统 GUI 中用户学习软件词汇，而 AI 时代软件要理解用户的自然语言意图。

AI 界面需要额外关注：

| 原则    | 含义                |
| ----- | ----------------- |
| 透明性   | 告诉用户 AI 的局限和不确定性  |
| 可解释性  | 提供依据、来源或推理过程摘要    |
| 渐进式信任 | 从建议到辅助，再逐步扩大权限    |
| 人工覆盖权 | 用户能拒绝、修改、撤销 AI 决策 |
| 优雅降级  | AI 失败时提供替代路径      |

> AI 时代的好设计，是让计算机理解人，而不是让人迁就计算机。

> AI 系统尤其要避免“看起来很确定但实际错误”的幻觉问题，因此引用、置信度和人工确认很重要。
