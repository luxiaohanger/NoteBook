## 人际交互设计原则

| 原则体系          | 经典叫法                                                        | 中文叫法           | 识别关键词                 |
| ------------- | ----------------------------------------------------------- | -------------- | --------------------- |
| Pressman 黄金原则 | **Place the user in control**                               | 让用户掌控系统        | 撤销、退出、取消、返回、用户可控      |
| Pressman 黄金原则 | **Reduce the user's memory load**                           | 减少用户记忆负担       | 默认值、菜单、自动补全、识别优于回忆    |
| Pressman 黄金原则 | **Be consistent**                                           | 保持一致性          | 相同操作相同结果、风格统一、平台规范    |
| Nielsen 启发式原则 | **Visibility of system status**                             | 系统状态可见         | 加载提示、进度条、保存成功、当前状态    |
| Nielsen 启发式原则 | **Match between system and the real world**                 | 系统与现实世界匹配      | 用户语言、现实概念、任务模型、少用技术术语 |
| Nielsen 启发式原则 | **User control and freedom**                                | 用户控制与自由        | 撤销、重做、取消、紧急出口         |
| Nielsen 启发式原则 | **Consistency and standards**                               | 一致性与标准         | 统一图标、统一术语、遵循平台习惯      |
| Nielsen 启发式原则 | **Error prevention**                                        | 防止错误           | 置灰、输入限制、二次确认、危险操作隔离   |
| Nielsen 启发式原则 | **Recognition rather than recall**                          | 识别优于回忆         | 选项可见、菜单、图标、历史记录       |
| Nielsen 启发式原则 | **Flexibility and efficiency of use**                       | 使用灵活性与效率       | 快捷键、热键、批量操作、专家模式      |
| Nielsen 启发式原则 | **Aesthetic and minimalist design**                         | 美观且简约设计        | 少即是多、去除无关信息、突出重点      |
| Nielsen 启发式原则 | **Help users recognize, diagnose, and recover from errors** | 帮助用户识别、诊断并恢复错误 | 错误原因、解决建议、友好错误提示      |
| Nielsen 启发式原则 | **Help and documentation**                                  | 帮助与文档          | 帮助中心、FAQ、上下文帮助、任务导向文档 |
| GUI 常用原则      | **Feedback**                                                | 及时反馈           | 操作后有响应、进度显示、结果提示      |
| GUI 常用原则      | **Shortcut**                                                | 提供快捷方式         | Ctrl+S、热键、宏命令、快速入口    |
| GUI 常用原则      | **Good error message**                                      | 良好的错误提示        | 清晰、准确、建设性、友好          |
| GUI 常用原则      | **Avoid mistake**                                           | 避免错误           | 防错、灰色屏蔽、合法性检查         |
| GUI 常用原则      | **Using user language**                                     | 使用用户语言         | 面向业务表达，而不是系统内部术语      |
| GUI 常用原则      | **Identify exit clear**                                     | 清晰标识出口         | 退出、返回、关闭、取消按钮明显       |

课件主要涉及三组 HCI 原则：**Pressman 三条黄金原则**、**Nielsen 十条可用性启发式原则**、以及 **GUI 设计常用原则**。其中 Pressman 三条是总纲，Nielsen 十条是更细的检查清单，GUI 原则是图形界面设计中的具体化表达。

> 做题时先判断题目描述的是“控制、记忆、一致性、反馈、防错、错误恢复、简约、效率、帮助”中的哪一类，再写出对应经典叫法。

## Place the User in Control

让用户掌控系统。用户应能控制交互过程，而不是被系统强迫沿着唯一流程走。

常见识别方式：

| 题目现象     | 对应判断                      |
| -------- | ------------------------- |
| 可以撤销误操作  | Place the user in control |
| 可以取消长任务  | Place the user in control |
| 可以返回上一步  | Place the user in control |
| 可以退出当前流程 | Place the user in control |
| 删除后可恢复   | Place the user in control |

反例识别：

| 问题描述      | 违反原则   |
| --------- | ------ |
| 用户点错后无法撤销 | 违反用户掌控 |
| 进入流程后不能退出 | 违反用户掌控 |
| 长时间任务不能取消 | 违反用户掌控 |

> 看到“撤销、返回、取消、退出、中断”，优先想到 **Place the user in control**。

> 它和 Nielsen 的 **User control and freedom** 基本对应。

---

## Reduce the User's Memory Load

减少用户记忆负担。系统应替用户记住信息，尽量让用户通过识别完成操作，而不是强迫用户回忆命令、路径或格式。课件中也提到人的短时记忆有限，以及 GUI 优于命令行的重要原因是“识别强于回忆”。

常见识别方式：

| 题目现象          | 对应判断   |
| ------------- | ------ |
| 提供菜单而不是要求输入命令 | 减少记忆负担 |
| 提供默认值         | 减少记忆负担 |
| 自动补全输入        | 减少记忆负担 |
| 记录最近使用项目      | 减少记忆负担 |
| 用图标、列表展示可选项   | 减少记忆负担 |
| 分步骤展示复杂任务     | 减少记忆负担 |

反例识别：

| 问题描述           | 违反原则     |
| -------------- | -------- |
| 用户必须记住复杂命令     | 违反减少记忆负担 |
| 表单要求用户重复填写已知信息 | 违反减少记忆负担 |
| 操作路径隐藏很深且无提示   | 违反减少记忆负担 |

> 看到“默认值、自动补全、菜单、历史记录、选项可见”，优先想到 **Reduce the user's memory load**。

> 它和 Nielsen 的 **Recognition rather than recall** 关系最密切。

---

## Be Consistent

保持一致性。相似操作应有相似结果，相似界面元素应有相似含义。课件将 **Be consistent** 作为黄金原则之一，并在 Nielsen 原则中进一步扩展为 **Consistency and standards**。

常见识别方式：

| 题目现象                          | 对应判断   |
| ----------------------------- | ------ |
| 所有页面按钮位置一致                    | 一致性    |
| 同一图标始终表示同一功能                  | 一致性    |
| 相同颜色表达相同状态                    | 一致性    |
| 操作流程在不同模块中相似                  | 一致性    |
| 遵循 Windows / iOS / Android 习惯 | 一致性与标准 |

反例识别：

| 问题描述              | 违反原则  |
| ----------------- | ----- |
| 有的页面“确定”在左，有的页面在右 | 违反一致性 |
| 同一个图标在不同页面含义不同    | 违反一致性 |
| 同一功能有多个不同名称       | 违反一致性 |
| 不遵守平台常见快捷键        | 违反标准  |

> 看到“统一、相同、标准、平台规范、相似操作相似结果”，优先想到 **Be consistent** 或 **Consistency and standards**。

---

## Visibility of System Status

系统状态可见。系统应及时告诉用户当前发生了什么，尤其是在等待、加载、提交、保存、上传等场景中。课件在反馈部分给出了响应时间经验准则，强调用户需要及时反馈。

常见识别方式：

| 题目现象      | 对应判断   |
| --------- | ------ |
| 显示加载动画    | 系统状态可见 |
| 显示上传进度    | 系统状态可见 |
| 提示“保存成功”  | 系统状态可见 |
| 显示当前步骤    | 系统状态可见 |
| 长任务显示剩余时间 | 系统状态可见 |

反例识别：

| 问题描述       | 违反原则     |
| ---------- | -------- |
| 点击提交后界面无反应 | 违反系统状态可见 |
| 文件上传没有进度提示 | 违反系统状态可见 |
| 保存后不知道是否成功 | 违反系统状态可见 |

> 看到“加载、进度、状态、提示、等待”，优先想到 **Visibility of system status**。

---

## Match Between System and the Real World

系统与现实世界匹配。界面应使用用户熟悉的业务语言、现实概念和任务流程，而不是暴露系统内部结构。课件在精神模型和可视化设计部分强调，应按照用户任务模型设计界面隐喻，不要把软件内部构造机制暴露给用户。

常见识别方式：

| 题目现象                | 对应判断      |
| ------------------- | --------- |
| 使用用户熟悉的业务词汇         | 系统与现实世界匹配 |
| 界面流程符合真实业务流程        | 系统与现实世界匹配 |
| 图标隐喻符合现实经验          | 系统与现实世界匹配 |
| 用“购物车”“订单”“付款”等用户概念 | 系统与现实世界匹配 |

反例识别：

| 问题描述            | 违反原则     |
| --------------- | -------- |
| 页面显示数据库字段名      | 违反现实世界匹配 |
| 把内部模块名暴露给用户     | 违反现实世界匹配 |
| 错误提示全是系统错误码     | 违反现实世界匹配 |
| 用户要理解系统内部机制才能操作 | 违反现实世界匹配 |

例如：

```text
错误：Transaction Commit Failed
```

更符合 HCI 的表达是：

```text
订单保存失败，请稍后重试。
```

> 看到“用户语言、现实隐喻、业务流程、不要暴露内部实现”，优先想到 **Match between system and the real world**。

---

## User Control and Freedom

用户控制与自由。用户可能误入某个页面或执行错误操作，因此系统应提供明显的“紧急出口”。这条原则与 Pressman 的 **Place the user in control** 高度重合。

常见识别方式：

| 题目现象      | 对应判断    |
| --------- | ------- |
| 支持撤销 Undo | 用户控制与自由 |
| 支持重做 Redo | 用户控制与自由 |
| 弹窗可以关闭    | 用户控制与自由 |
| 流程可以取消    | 用户控制与自由 |
| 操作可以返回    | 用户控制与自由 |

反例识别：

| 问题描述     | 违反原则      |
| -------- | --------- |
| 弹窗没有关闭按钮 | 违反用户控制与自由 |
| 误删无法恢复   | 违反用户控制与自由 |
| 只能继续不能退出 | 违反用户控制与自由 |

> 看到“紧急出口、撤销、重做、取消、返回”，优先想到 **User control and freedom**。

---

## Consistency and Standards

一致性与标准。系统内部要一致，同时也要符合用户从其他系统中获得的习惯。

常见识别方式：

| 题目现象              | 对应判断   |
| ----------------- | ------ |
| 放大镜图标表示搜索         | 一致性与标准 |
| 齿轮图标表示设置          | 一致性与标准 |
| Ctrl + S 表示保存     | 一致性与标准 |
| 红色按钮表示危险操作        | 一致性与标准 |
| 移动端底部 Tab 不超过常见数量 | 一致性与标准 |

反例识别：

| 问题描述         | 违反原则     |
| ------------ | -------- |
| 使用反常图标表达常见功能 | 违反一致性与标准 |
| 系统内部术语前后不一致  | 违反一致性与标准 |
| 快捷键不符合平台习惯   | 违反一致性与标准 |

> 看到“平台规范、通用习惯、标准图标、统一术语”，优先想到 **Consistency and standards**。

---

## Error Prevention

防止错误。优秀的设计应优先避免用户犯错，而不是只在错误发生后提示。课件在低出错率设计中提到：不适当选项应灰色屏蔽，数值输入域不应允许字母字符。

常见识别方式：

| 题目现象       | 对应判断 |
| ---------- | ---- |
| 禁用当前不可用按钮  | 防止错误 |
| 输入框限制非法字符  | 防止错误 |
| 删除前二次确认    | 防止错误 |
| 危险操作远离常用操作 | 防止错误 |
| 自动检查输入格式   | 防止错误 |

反例识别：

| 问题描述         | 违反原则   |
| ------------ | ------ |
| 金额输入框允许输入字母  | 违反防止错误 |
| 删除按钮紧挨保存按钮   | 违反防止错误 |
| 危险操作没有确认     | 违反防止错误 |
| 明知条件不满足仍允许提交 | 违反防止错误 |

> 看到“置灰、禁止非法输入、确认、约束、预防”，优先想到 **Error prevention**。

---

## Recognition Rather Than Recall

识别优于回忆。界面应把可用选项、操作对象和必要信息显示出来，减少用户记忆负担。

常见识别方式：

| 题目现象      | 对应判断   |
| --------- | ------ |
| 提供菜单列表    | 识别优于回忆 |
| 提供候选项     | 识别优于回忆 |
| 提供最近使用记录  | 识别优于回忆 |
| 面包屑显示当前位置 | 识别优于回忆 |
| 自动补全搜索词   | 识别优于回忆 |

反例识别：

| 问题描述        | 违反原则     |
| ----------- | -------- |
| 用户必须记住完整命令  | 违反识别优于回忆 |
| 必须记住上一步填写内容 | 违反识别优于回忆 |
| 页面跳转后没有位置提示 | 违反识别优于回忆 |

> 看到“菜单、图标、候选项、历史、面包屑”，优先想到 **Recognition rather than recall**。

---

## Flexibility and Efficiency of Use

使用灵活性与效率。系统应同时照顾新手和专家用户。课件指出，新手用户重视易学性，专家用户重视效率，因此界面应为不同用户提供差异化机制。

常见识别方式：

| 题目现象       | 对应判断   |
| ---------- | ------ |
| 菜单和快捷键同时存在 | 灵活性与效率 |
| 提供批量操作     | 灵活性与效率 |
| 提供专家模式     | 灵活性与效率 |
| 提供命令行或热键   | 灵活性与效率 |
| 提供个性化配置    | 灵活性与效率 |

反例识别：

| 问题描述           | 违反原则     |
| -------------- | -------- |
| 熟练用户只能重复点击多层菜单 | 违反灵活性与效率 |
| 高频操作没有快捷入口     | 违反灵活性与效率 |
| 批量任务只能逐个处理     | 违反灵活性与效率 |

> 看到“快捷键、热键、专家用户、批量操作、高频操作”，优先想到 **Flexibility and efficiency of use**。

---

## Aesthetic and Minimalist Design

美观且简约的设计。界面不应包含无关信息，视觉层级应清晰，主要任务应突出。课件中 GUI 原则也写到 **Few is much**，即“少即是多”。

常见识别方式：

| 题目现象         | 对应判断 |
| ------------ | ---- |
| 页面只保留与任务相关内容 | 简约设计 |
| 主按钮突出        | 简约设计 |
| 信息分组清楚       | 简约设计 |
| 去掉冗余文字       | 简约设计 |
| 视觉层级清晰       | 简约设计 |

反例识别：

| 问题描述       | 违反原则   |
| ---------- | ------ |
| 页面堆满无关信息   | 违反简约设计 |
| 广告和提示干扰主任务 | 违反简约设计 |
| 所有按钮视觉权重一样 | 违反简约设计 |
| 内容没有层次     | 违反简约设计 |

> 看到“少即是多、无关信息、视觉层级、突出重点”，优先想到 **Aesthetic and minimalist design**。

---

## Help Users Recognize, Diagnose, and Recover from Errors

帮助用户识别、诊断并恢复错误。错误信息不仅要告诉用户出错，还要告诉用户哪里错、为什么错、怎么改。课件总结了错误信息四原则：清晰语言、精炼准确、建设性帮助、友好态度。

常见识别方式：

| 题目现象       | 对应判断   |
| ---------- | ------ |
| 错误提示说明具体原因 | 帮助识别错误 |
| 错误提示给出修改建议 | 帮助恢复错误 |
| 提示语不用技术错误码 | 良好错误提示 |
| 错误提示语气友好   | 良好错误提示 |

反例识别：

| 问题描述                | 违反原则     |
| ------------------- | -------- |
| 只显示 Error 404 / 500 | 违反错误恢复原则 |
| 只说“操作失败”            | 违反错误恢复原则 |
| 错误提示责备用户            | 违反错误恢复原则 |
| 没有告诉用户怎么修复          | 违反错误恢复原则 |

错误提示反例：

```text
Error Code: 0x00031
```

改进：

```text
上传失败：文件超过 20MB，请压缩后重新上传。
```

> 看到“错误原因、解决建议、友好提示、恢复”，优先想到 **Help users recognize, diagnose, and recover from errors**。

---

## Help and Documentation

帮助与文档。即使系统易用，也应在用户需要时提供帮助。文档应易搜索、面向任务、简明准确。

常见识别方式：

| 题目现象     | 对应判断  |
| -------- | ----- |
| 提供帮助中心   | 帮助与文档 |
| 提供 FAQ   | 帮助与文档 |
| 鼠标悬停显示说明 | 帮助与文档 |
| 表单旁有填写示例 | 帮助与文档 |
| 文档按任务组织  | 帮助与文档 |

反例识别：

| 问题描述            | 违反原则    |
| --------------- | ------- |
| 复杂功能没有说明        | 违反帮助与文档 |
| 文档无法搜索          | 违反帮助与文档 |
| 文档只讲系统模块，不讲用户任务 | 违反帮助与文档 |

> 看到“帮助、文档、FAQ、说明、示例、教程”，优先想到 **Help and documentation**。

---

## Feedback

及时反馈。用户操作后，系统应及时给出响应。课件中 GUI 原则明确列出 **Feedback**，并在反馈部分给出不同任务的响应时间要求。

常见识别方式：

| 题目现象           | 对应判断 |
| -------------- | ---- |
| 点击按钮后按钮变为“处理中” | 及时反馈 |
| 保存后显示“保存成功”    | 及时反馈 |
| 加载时显示进度条       | 及时反馈 |
| 输入错误即时提示       | 及时反馈 |
| AI 生成时流式输出     | 及时反馈 |

反例识别：

| 问题描述          | 违反原则   |
| ------------- | ------ |
| 用户点击后没有任何反应   | 违反及时反馈 |
| 长时间等待无提示      | 违反及时反馈 |
| 表单提交成功与失败没有区别 | 违反及时反馈 |

> 看到“操作后有没有响应”，优先判断 **Feedback** 或 **Visibility of system status**。

---

## Shortcut

提供快捷方式。系统应为熟练用户和专家用户提供更高效的操作入口。

常见识别方式：

| 题目现象         | 对应判断     |
| ------------ | -------- |
| Ctrl + S 保存  | Shortcut |
| Alt + Tab 切换 | Shortcut |
| 命令面板         | Shortcut |
| 批量处理入口       | Shortcut |
| 自定义快捷键       | Shortcut |

反例识别：

| 问题描述           | 违反原则   |
| -------------- | ------ |
| 高频操作只能通过多级菜单完成 | 缺少快捷方式 |
| 专家用户没有快速入口     | 缺少快捷方式 |
| 重复任务不能批量执行     | 效率不足   |

> 看到“快捷键、热键、快速入口、专家用户”，优先想到 **Shortcut** 或 **Flexibility and efficiency of use**。

---

## Good Error Message

良好的错误提示。它是 Nielsen 错误恢复原则在 GUI 设计中的具体表达。

识别标准：

| 标准  | 含义       |
| --- | -------- |
| 清晰  | 用户知道哪里错  |
| 准确  | 不空泛      |
| 建设性 | 告诉用户如何解决 |
| 友好  | 不责备用户    |

反例：

```text
输入错误。
```

更好：

```text
邮箱格式不正确，请输入类似 name@example.com 的地址。
```

> 看到“错误提示质量”，优先想到 **Good error message**。

---

## Avoid Mistake

避免错误。它和 Nielsen 的 **Error prevention** 对应，强调通过设计降低用户犯错概率。

常见识别方式：

| 题目现象       | 对应判断          |
| ---------- | ------------- |
| 输入框限制格式    | Avoid mistake |
| 不可用按钮置灰    | Avoid mistake |
| 危险操作确认     | Avoid mistake |
| 默认选择安全项    | Avoid mistake |
| 删除按钮远离保存按钮 | Avoid mistake |

> 看到“防止误操作、降低出错率”，优先想到 **Avoid mistake**。

---

## Using User Language

使用用户语言。界面表达应面向用户，而不是面向程序员或系统实现。

常见识别方式：

| 题目现象               | 对应判断   |
| ------------------ | ------ |
| 用“保存订单”而不是“提交事务”   | 使用用户语言 |
| 用“付款失败”而不是“支付接口异常” | 使用用户语言 |
| 用业务流程命名菜单          | 使用用户语言 |
| 错误提示避免技术码          | 使用用户语言 |

> 看到“业务语言、用户能懂、非技术术语”，优先想到 **Using user language** 或 **Match between system and the real world**。

---

## Identify Exit Clear

清晰标识出口。用户应明确知道如何退出当前页面、弹窗或流程。

常见识别方式：

| 题目现象       | 对应判断   |
| ---------- | ------ |
| 弹窗有关闭按钮    | 清晰标识出口 |
| 表单有取消按钮    | 清晰标识出口 |
| 多步骤流程有返回按钮 | 清晰标识出口 |
| 全屏页面有退出入口  | 清晰标识出口 |

反例识别：

| 问题描述          | 违反原则  |
| ------------- | ----- |
| 弹窗无法关闭        | 出口不清晰 |
| 页面只能继续不能返回    | 出口不清晰 |
| 用户不知道如何退出当前模式 | 出口不清晰 |

> 看到“退出、关闭、取消、返回入口是否明显”，优先想到 **Identify exit clear**。

---

## HCI 原则识别题答题模板

遇到界面案例分析题，可以按下面步骤判断：

1. **看用户是否知道系统状态**：如果没有加载、进度、成功失败提示，答 **Visibility of system status / Feedback**；
2. **看用户是否能撤销退出**：如果不能取消、返回、撤销，答 **User control and freedom / Place the user in control**；
3. **看是否要求用户记忆**：如果必须记命令、路径、格式，答 **Reduce memory load / Recognition rather than recall**；
4. **看界面是否统一**：如果图标、术语、按钮位置不统一，答 **Consistency and standards**；
5. **看是否提前防错**：如果非法输入可提交、危险操作无确认，答 **Error prevention / Avoid mistake**；
6. **看错误提示是否有用**：如果只有错误码或“操作失败”，答 **Good error message / Error recovery**；
7. **看是否符合用户语言**：如果暴露数据库、事务、接口、错误码，答 **Match between system and real world / Using user language**；
8. **看是否简洁**：如果页面堆满无关信息，答 **Aesthetic and minimalist design / Few is much**；
9. **看是否照顾专家用户**：如果高频操作无快捷方式，答 **Flexibility and efficiency / Shortcut**；
10. **看是否有帮助材料**：如果复杂功能无说明，答 **Help and documentation**。

> 识别题不要只写“界面不好”，要写出违反了哪条经典原则，再说明如何修改。

> 标准格式：**现象 → 违反原则 → 修改建议**。
