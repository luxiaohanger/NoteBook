## 设计原则总览

> 软件设计原则的目的，是让代码更容易维护、扩展、测试和复用。核心目标可以概括为：**高内聚、低耦合、依赖清晰、变化隔离**。
### SOLID 原则

| 字母    | 原则      | 中文名    | 核心含义                      |
| ----- | ------- | ------ | ------------------------- |
| **S** | **SRP** | 单一职责原则 | 一个类最好只有一个引起它变化的原因         |
| **O** | **OCP** | 开闭原则   | 对扩展开放，对修改关闭               |
| **L** | **LSP** | 里氏替换原则 | 子类对象应能替换父类对象且不破坏程序正确性     |
| **I** | **ISP** | 接口隔离原则 | 接口应小而专一，客户端不应依赖不需要的方法     |
| **D** | **DIP** | 依赖倒置原则 | 高层模块和低层模块都应依赖抽象，而不是依赖具体实现 |

### 其他原则
| 原则                                            | 含义             | 常见违反                      | 修改方向               |
| --------------------------------------------- | -------------- | ------------------------- | ------------------ |
| Global Variables Consider Harmful             | 避免全局可变状态       | 多模块共享全局变量，修改来源不清          | 封装状态，通过参数或对象传递     |
| To be Explicit                                | 依赖、输入、输出和错误要明确 | 隐式依赖环境变量、全局状态、魔法约定        | 显式参数、显式接口、清晰命名     |
| Do not Repeat                                 | 不重复表达同一知识      | 多处复制相同判断和计算               | 提取方法、规则表、策略        |
| Programming to Interface / Design by Contract | 面向接口和契约编程      | 上层依赖具体类，前置/后置条件不清         | 定义接口、前置条件、后置条件和不变式 |
| LoD                                           | 只与直接朋友通信       | `a.getB().getC()` 消息链     | 增加委托方法，隐藏内部结构      |
| 组合优于继承                                        | 用组合复用行为        | 为复用少量方法滥用继承               | 对象内部持有可复用组件        |


> 通俗理解：这些原则本质上都在解决一个问题：**当需求变化时，代码改动范围能不能尽量小**。

## ISP

> **ISP**，即 Interface Segregation Principle，接口隔离原则。它要求接口应小而专一，客户端不应该被迫依赖自己不需要的方法。

### 修改前：胖接口

```java
interface Worker {
    void code();
    void test();
    void manage();
}

class Programmer implements Worker {
    public void code() {
        System.out.println("写代码");
    }

    public void test() {
        System.out.println("测试代码");
    }

    public void manage() {
        throw new UnsupportedOperationException("程序员不负责管理");
    }
}
```

问题：`Programmer` 被迫实现 `manage` 方法。

### 修改后：拆分接口

```java
interface Coder {
    void code();
}

interface Tester {
    void test();
}

interface Manager {
    void manage();
}

class Programmer implements Coder, Tester {
    public void code() {
        System.out.println("写代码");
    }

    public void test() {
        System.out.println("测试代码");
    }
}

class ProjectManager implements Manager {
    public void manage() {
        System.out.println("管理项目");
    }
}
```

> 通俗理解：**接口不要太胖，不要让实现类被迫做自己不该做的事**。

> 考试注意：ISP 强调的是**接口粒度**，SRP 强调的是**类的职责**。

## LSP

> **LSP**，即 Liskov Substitution Principle，里氏替换原则。它要求子类对象必须能够替换父类对象，并且不破坏程序原有正确性。

### 修改前：正方形继承矩形

```java
class Rectangle {
    protected int width;
    protected int height;

    public void setWidth(int width) {
        this.width = width;
    }

    public void setHeight(int height) {
        this.height = height;
    }

    public int area() {
        return width * height;
    }
}

class Square extends Rectangle {
    public void setWidth(int width) {
        this.width = width;
        this.height = width;
    }

    public void setHeight(int height) {
        this.width = height;
        this.height = height;
    }
}
```

测试代码：

```java
class AreaTest {
    public void resize(Rectangle rectangle) {
        rectangle.setWidth(5);
        rectangle.setHeight(4);

        // 对矩形期望面积是 20
        System.out.println(rectangle.area());
    }
}
```

如果传入 `Square`，面积变成 `16`，破坏了父类语义。

### 修改后：重新抽象

```java
interface Shape {
    int area();
}

class Rectangle implements Shape {
    private final int width;
    private final int height;

    public Rectangle(int width, int height) {
        this.width = width;
        this.height = height;
    }

    public int area() {
        return width * height;
    }
}

class Square implements Shape {
    private final int side;

    public Square(int side) {
        this.side = side;
    }

    public int area() {
        return side * side;
    }
}
```

> 通俗理解：**凡是能用父类的地方，换成子类也不能出问题**。

> 考试注意：子类不能随意加强前置条件、削弱后置条件、破坏父类不变式。

## SRP

> **SRP**，即 Single Responsibility Principle，单一职责原则。一个类应该只有一个引起它变化的原因。

### 修改前：一个类负责太多事情

```java
class OrderService {
    public void validateOrder() {
        System.out.println("校验订单");
    }

    public double calculateTotal() {
        System.out.println("计算订单总价");
        return 100.0;
    }

    public void saveOrder() {
        System.out.println("保存订单到数据库");
    }

    public void sendEmail() {
        System.out.println("发送订单邮件");
    }
}
```

问题：校验规则、价格规则、数据库保存、邮件发送任意一个变化，都可能导致 `OrderService` 修改。

### 修改后：拆分职责

```java
class OrderValidator {
    public void validate() {
        System.out.println("校验订单");
    }
}

class OrderPriceCalculator {
    public double calculateTotal() {
        System.out.println("计算订单总价");
        return 100.0;
    }
}

class OrderRepository {
    public void save() {
        System.out.println("保存订单到数据库");
    }
}

class EmailService {
    public void send() {
        System.out.println("发送订单邮件");
    }
}

class OrderService {
    private final OrderValidator validator = new OrderValidator();
    private final OrderPriceCalculator calculator = new OrderPriceCalculator();
    private final OrderRepository repository = new OrderRepository();
    private final EmailService emailService = new EmailService();

    public void createOrder() {
        validator.validate();
        calculator.calculateTotal();
        repository.save();
        emailService.send();
    }
}
```

> 通俗理解：**一个类不要又当裁判、又当会计、又当仓库管理员、又当邮递员**。

> 考试注意：SRP 不是要求每个类只有一个方法，而是要求只有一个主要变化原因。

## OCP

> **OCP**，即 Open-Closed Principle，开闭原则。软件实体应当对扩展开放，对修改关闭。也就是说，新增功能时尽量通过增加新代码完成，而不是频繁修改旧代码。

### 修改前：新增类型必须修改旧 `switch`

```java
class DiscountService {
    public double discount(String userType, double price) {
        switch (userType) {
            case "VIP":
                return price * 0.8;
            case "SVIP":
                return price * 0.7;
            default:
                return price;
        }
    }
}
```

如果新增学生折扣，就必须修改 `DiscountService`。

### 修改后：使用策略模式

```java
interface DiscountStrategy {
    double discount(double price);
}

class VipDiscount implements DiscountStrategy {
    public double discount(double price) {
        return price * 0.8;
    }
}

class SuperVipDiscount implements DiscountStrategy {
    public double discount(double price) {
        return price * 0.7;
    }
}

class StudentDiscount implements DiscountStrategy {
    public double discount(double price) {
        return price * 0.9;
    }
}

class DiscountService {
    public double calculate(double price, DiscountStrategy strategy) {
        return strategy.discount(price);
    }
}
```

调用方式：

```java
DiscountService service = new DiscountService();

double price = service.calculate(100.0, new StudentDiscount());
System.out.println(price);
```

> 通俗理解：**新增功能尽量加新类，不要总是改旧类**。

> 考试注意：OCP 常通过接口、多态、策略模式、工厂模式实现。

## DIP

> **DIP**，即 Dependency Inversion Principle，依赖倒置原则。高层模块不应该依赖低层模块，二者都应该依赖抽象；抽象不应该依赖细节，细节应该依赖抽象。

### 修改前：高层直接依赖低层实现

```java
class MySqlUserDao {
    public void save(String username) {
        System.out.println("保存到 MySQL：" + username);
    }
}

class UserService {
    private MySqlUserDao userDao = new MySqlUserDao();

    public void register(String username) {
        userDao.save(username);
    }
}
```

问题：`UserService` 是高层业务模块，却直接依赖 `MySqlUserDao` 这个低层细节。

### 修改后：依赖抽象并注入实现

```java
interface UserDao {
    void save(String username);
}

class MySqlUserDao implements UserDao {
    public void save(String username) {
        System.out.println("保存到 MySQL：" + username);
    }
}

class UserService {
    private final UserDao userDao;

    public UserService(UserDao userDao) {
        this.userDao = userDao;
    }

    public void register(String username) {
        userDao.save(username);
    }
}
```

调用方式：

```java
UserDao userDao = new MySqlUserDao();
UserService userService = new UserService(userDao);

userService.register("Alice");
```

> 通俗理解：**业务层不要直接依赖数据库实现，而是依赖一个抽象接口**。

> 考试注意：DIP 关注依赖方向，通常配合依赖注入使用。

## Global Variables Consider Harmful

> **Global Variables Consider Harmful** 指应尽量避免使用全局可变变量。全局变量会导致多个模块共享同一份状态，任何地方都可能修改它，导致错误难定位、测试困难、模块耦合增强。

### 修改前：使用全局变量

```java
class GlobalConfig {
    public static String databaseUrl = "localhost:3306";
}

class UserDao {
    public void connect() {
        System.out.println("连接数据库：" + GlobalConfig.databaseUrl);
    }
}

class OrderDao {
    public void connect() {
        System.out.println("连接数据库：" + GlobalConfig.databaseUrl);
    }
}
```

问题：

```java
GlobalConfig.databaseUrl = "wrong-url";
```

任何模块都可以修改 `databaseUrl`，一旦出错，很难判断是谁改的。

### 修改后：封装状态，通过对象传递

```java
class DatabaseConfig {
    private final String databaseUrl;

    public DatabaseConfig(String databaseUrl) {
        this.databaseUrl = databaseUrl;
    }

    public String getDatabaseUrl() {
        return databaseUrl;
    }
}

class UserDao {
    private final DatabaseConfig config;

    public UserDao(DatabaseConfig config) {
        this.config = config;
    }

    public void connect() {
        System.out.println("连接数据库：" + config.getDatabaseUrl());
    }
}
```

> 通俗理解：**不要让大家都直接改同一个公共变量，而是把状态封装起来，通过明确入口访问**。

> 考试注意：全局常量问题较小，真正危险的是**全局可变状态**。

## To be Explicit

> **To be Explicit** 指依赖、输入、输出、异常和约定都应该明确表达，避免让代码依赖隐含环境或魔法规则。

### 修改前：隐式依赖环境变量

```java
class FileService {
    public void save(String content) {
        String path = System.getenv("SAVE_PATH");
        System.out.println("保存到：" + path + "/" + content);
    }
}
```

问题：调用者不知道 `save` 方法依赖环境变量 `SAVE_PATH`。如果环境变量不存在，程序可能出错。

### 修改后：显式传入依赖

```java
class FileService {
    private final String savePath;

    public FileService(String savePath) {
        this.savePath = savePath;
    }

    public void save(String content) {
        System.out.println("保存到：" + savePath + "/" + content);
    }
}
```

调用方式：

```java
FileService service = new FileService("/data/files");
service.save("report.txt");
```

> 通俗理解：**代码需要什么，就明明白白写出来，不要藏在环境、全局变量或默认约定里**。

> 考试注意：显式化可以降低理解成本，也便于测试，因为测试时可以直接传入依赖。

## Do not Repeat

> **Do not Repeat**，也称 DRY 原则，指不要在多个地方重复表达同一份知识、规则或逻辑。重复会导致维护困难，一处修改时容易漏改其他地方。

### 修改前：重复判断会员折扣

```java
class OrderService {
    public double calculateBookPrice(double price, boolean vip) {
        if (vip) {
            return price * 0.8;
        }
        return price;
    }

    public double calculatePhonePrice(double price, boolean vip) {
        if (vip) {
            return price * 0.8;
        }
        return price;
    }
}
```

如果会员折扣从 `8` 折改成 `7` 折，需要改多处。

### 修改后：提取公共方法

```java
class DiscountService {
    public double applyVipDiscount(double price, boolean vip) {
        if (vip) {
            return price * 0.8;
        }
        return price;
    }
}

class OrderService {
    private final DiscountService discountService = new DiscountService();

    public double calculateBookPrice(double price, boolean vip) {
        return discountService.applyVipDiscount(price, vip);
    }

    public double calculatePhonePrice(double price, boolean vip) {
        return discountService.applyVipDiscount(price, vip);
    }
}
```

> 通俗理解：**同一条规则只写一次，改规则时也只改一个地方**。

> 考试注意：DRY 不是禁止所有相似代码，而是禁止**重复表达同一业务知识**。

## Programming to Interface

> **Programming to Interface** 指面向接口编程。高层模块不直接依赖具体实现类，而是依赖抽象接口，从而便于替换实现、扩展功能和单元测试。

### 修改前：直接依赖具体类

```java
class MySqlUserDao {
    public void save(String username) {
        System.out.println("保存用户到 MySQL：" + username);
    }
}

class UserService {
    private MySqlUserDao userDao = new MySqlUserDao();

    public void register(String username) {
        userDao.save(username);
    }
}
```

问题：`UserService` 和 `MySqlUserDao` 绑定太紧。如果以后改成 `PostgreSQL` 或 `Redis`，需要修改 `UserService`。

### 修改后：依赖接口

```java
interface UserDao {
    void save(String username);
}

class MySqlUserDao implements UserDao {
    public void save(String username) {
        System.out.println("保存用户到 MySQL：" + username);
    }
}

class RedisUserDao implements UserDao {
    public void save(String username) {
        System.out.println("保存用户到 Redis：" + username);
    }
}

class UserService {
    private final UserDao userDao;

    public UserService(UserDao userDao) {
        this.userDao = userDao;
    }

    public void register(String username) {
        userDao.save(username);
    }
}
```

调用方式：

```java
UserDao userDao = new MySqlUserDao();
UserService service = new UserService(userDao);
service.register("Alice");
```

> 通俗理解：**上层只关心“你能做什么”，不关心“你具体怎么做”**。

> 考试注意：面向接口编程可以降低耦合，是 DIP、OCP 的基础。

## Design by Contract

> **Design by Contract** 指契约式设计。模块之间通过明确的契约协作，契约包括前置条件、后置条件和不变式。

常见契约：

| 类型   | 含义               |
| ---- | ---------------- |
| 前置条件 | 调用方法前必须满足的条件     |
| 后置条件 | 方法执行后必须保证的结果     |
| 不变式  | 对象生命周期中始终应该成立的条件 |

### 示例：账户取款契约

```java
class Account {
    private double balance;

    public Account(double balance) {
        if (balance < 0) {
            throw new IllegalArgumentException("初始余额不能小于 0");
        }
        this.balance = balance;
    }

    public void withdraw(double amount) {
        // 前置条件
        if (amount <= 0) {
            throw new IllegalArgumentException("取款金额必须大于 0");
        }

        if (amount > balance) {
            throw new IllegalArgumentException("余额不足");
        }

        balance -= amount;

        // 后置条件 / 不变式
        if (balance < 0) {
            throw new IllegalStateException("余额不能为负数");
        }
    }

    public double getBalance() {
        return balance;
    }
}
```

> 通俗理解：**调用方法之前说清楚你要满足什么，调用之后我保证给你什么结果**。

> 考试注意：契约可以减少模块之间的猜测，避免调用者和被调用者责任不清。

## LoD

> **LoD**，即 Law of Demeter，迪米特法则，指一个对象应尽量只与自己的直接朋友通信，不要了解太深层的对象结构。

常见违反形式：

```java
a.getB().getC().doSomething();
```

这种写法叫**消息链**，说明当前对象知道了太多内部结构。

### 修改前：暴露内部结构

```java
class City {
    public String getName() {
        return "上海";
    }
}

class Address {
    private City city = new City();

    public City getCity() {
        return city;
    }
}

class User {
    private Address address = new Address();

    public Address getAddress() {
        return address;
    }
}

class OrderService {
    public void printUserCity(User user) {
        System.out.println(user.getAddress().getCity().getName());
    }
}
```

问题：`OrderService` 知道了 `User -> Address -> City` 的内部结构。

### 修改后：增加委托方法

```java
class City {
    public String getName() {
        return "上海";
    }
}

class Address {
    private City city = new City();

    public String getCityName() {
        return city.getName();
    }
}

class User {
    private Address address = new Address();

    public String getCityName() {
        return address.getCityName();
    }
}

class OrderService {
    public void printUserCity(User user) {
        System.out.println(user.getCityName());
    }
}
```

> 通俗理解：**不要跨好几层去拿别人的内部零件，应该让直接对象帮你完成请求**。

> 考试注意：LoD 主要用于降低对象之间的结构耦合。

## 组合优于继承

> **组合优于继承**指在复用代码时，优先让一个类持有另一个对象，而不是通过继承获得行为。继承会让子类强依赖父类实现，组合更灵活。

### 修改前：为了复用方法而继承

```java
class EmailSender {
    public void sendEmail(String message) {
        System.out.println("发送邮件：" + message);
    }
}

class OrderService extends EmailSender {
    public void createOrder() {
        System.out.println("创建订单");
        sendEmail("订单创建成功");
    }
}
```

问题：`OrderService` 本质上不是一种 `EmailSender`，只是需要使用发邮件功能。

### 修改后：使用组合

```java
class EmailSender {
    public void sendEmail(String message) {
        System.out.println("发送邮件：" + message);
    }
}

class OrderService {
    private final EmailSender emailSender;

    public OrderService(EmailSender emailSender) {
        this.emailSender = emailSender;
    }

    public void createOrder() {
        System.out.println("创建订单");
        emailSender.sendEmail("订单创建成功");
    }
}
```

> 通俗理解：**不是“我是它”，而是“我有它”**。

> 考试注意：继承适合稳定的 `is-a` 关系；组合适合 `has-a` 或 `use-a` 关系。
