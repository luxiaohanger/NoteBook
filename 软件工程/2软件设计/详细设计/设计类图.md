## 设计类图

> **设计类图**是在软件设计阶段使用的 UML 类图，用来描述系统中类的结构、属性、方法以及类之间的关系。

#### 设计类图和概念类图的区别

 **概念类图回答“业务中有什么对象”，设计类图回答“程序中要写哪些类”。**

| 对比项 | 概念类图          | 设计类图                                                        |
| --- | ------------- | ----------------------------------------------------------- |
| 阶段  | 需求分析 / 面向对象分析 | 软件设计                                                        |
| 关注点 | 业务概念          | 程序结构                                                        |
| 类   | 学生、课程、订单      | `StudentController`、`StudentService`、`StudentDao`、`Student` |
| 属性  | 重要业务属性        | 属性名、类型、可见性                                                  |
| 方法  | 通常省略          | 需要写主要方法                                                     |
| 关系  | 业务关系          | 调用、依赖、关联、继承、实现                                              |

#### 类的三栏结构

> 设计类图中，一个类通常分为三栏：**类名、属性、操作**。

```text
┌──────────────────────┐
│ 类名                  │
├──────────────────────┤
│ 属性                  │
├──────────────────────┤
│ 操作 / 方法            │
└──────────────────────┘
```

例如：

```text
┌────────────────────────────┐
│ Student                    │
├────────────────────────────┤
│ - id: Integer              │
│ - name: String             │
│ - major: String            │
├────────────────────────────┤
│ + getId(): Integer         │
│ + setName(name: String): void │
└────────────────────────────┘
```

符号含义：

```text
+ public，公有
- private，私有
# protected，受保护
~ package，包内可见
```

> 概念类图可以只写类名和属性；设计类图通常要写**属性类型、方法参数、返回值、可见性**。



#### 设计类图中类的来源

> 设计类图中的类一般来自概念类、用例、分层架构和设计模式。

常见类包括：

```text
实体类 Entity：Student、Course、Order
控制类 Controller：StudentController、OrderController
业务类 Service：StudentService、OrderService
数据访问类 DAO：StudentDao、OrderDao
工具类 Util：DateUtil、StringUtil
接口 Interface：UserRepository、PaymentService
```

例如学生管理系统：

```text
StudentController
StudentService
StudentDao
Student
```

> **概念类图中的“学生”到设计阶段可能变成 `Student` 实体类，并额外增加 Controller、Service、DAO 等实现类。**

#### 设计类图中的常见关系


常见关系：

```text
泛化：子类 ─────▷ 父类
实现：实现类 - - -▷ 接口
关联：类 ───── 类
聚合：整体 ◇──── 部分
组合：整体 ◆──── 部分
依赖：使用者 - - -> 被使用者
```


> 实现表示类实现接口。

```text
StudentDaoImpl - - -▷ StudentDao
```


> 依赖表示一个类临时使用另一个类，常见于方法调用、参数、局部变量。

```text
StudentController - - -> StudentService
StudentService - - -> StudentDao
```

> 关联表示一个类长期持有另一个类的引用，常见于成员变量。

```text
StudentService ─── StudentDao
```

如果 `StudentService` 中有：

```java
private StudentDao studentDao;
```

就可以画关联。

> 易错点：考试中如果不强调成员变量，可以把上层调用下层画成依赖；如果强调“持有对象引用”，可以画关联。


#### 示例：学生查询功能

> 题干：系统支持根据学号查询学生信息，前端通过 REST 接口访问后端，后端业务层调用 DAO 查询数据库。

可以抽取设计类：

```text
StudentController
StudentService
StudentDao
Student
```

类图可简化为：

```text
┌────────────────────────------------┐
│ StudentController                  │
├────────────────────────------------┤
│ - studentService: StudentService   │
├────────────────────────------------┤
│ + getStudent(id: Integer): Student │
└────────────────────────------------┘
            |
            | 依赖 / 关联
            v
┌────────────────────────---------------┐
│ StudentService                        │
├────────────────────────---------------┤
│ - studentDao: StudentDao              │
├────────────────────────---------------┤
│ + findStudent(id: Integer): Student   │
└────────────────────────---------------┘
            |
            | 依赖 / 关联
            v
┌────────────────────────----------┐
│ StudentDao                       │
├────────────────────────----------┤
│                                  │
├────────────────────────----------┤
│ + findById(id: Integer): Student │
└────────────────────────----------┘
            |
            | 返回 / 使用
            v
┌────────────────────────-------┐
│ Student                       │
├────────────────────────-------┤
│ - id: Integer                 │
│ - name: String                │
│ - major: String               │
├────────────────────────-------┤
│ + getId(): Integer            │
│ + setName(name: String): void │
└────────────────────────-------┘
```

> 通俗地说：
> **请求先进 Controller，再到 Service，再到 DAO，最后用 Entity 封装数据。**
