## 设计模式

**设计模式**是在软件设计中反复出现的问题的通用解决方案。它不是现成代码，而是一种可复用的设计经验。

---

## 设计模式分类

> 常见设计模式通常分为三大类：**创建型模式、结构型模式、行为型模式**。

| 类型    | 关注点         | 常见模式                                      |
| ----- | ----------- | ----------------------------------------- |
| 创建型模式 | 对象如何创建      | 单例、工厂方法、抽象工厂、建造者、原型                       |
| 结构型模式 | 类和对象如何组合    | 适配器、装饰器、代理、外观、桥接、组合、享元                    |
| 行为型模式 | 对象如何交互和分配职责 | 策略、观察者、模板方法、命令、责任链、状态、迭代器、中介者、备忘录、访问者、解释器 |

> 通俗理解：
> **创建型管“怎么造对象”，结构型管“怎么组对象”，行为型管“对象之间怎么协作”。**

---

## 单例模式

> **单例模式**保证一个类在系统中只有一个实例，并提供一个全局访问点。

### 代码示例

```java
class ConfigManager {
    private static final ConfigManager INSTANCE = new ConfigManager();

    private ConfigManager() {
    }

    public static ConfigManager getInstance() {
        return INSTANCE;
    }

    public String getConfig() {
        return "系统配置";
    }
}
```

使用：

```java
ConfigManager config = ConfigManager.getInstance();
System.out.println(config.getConfig());
```

> 通俗理解：**整个系统只需要一个对象，例如配置管理器、日志管理器。**

> 优点：节省资源，保证实例唯一。
> 缺点：容易形成全局状态，测试困难，可能增加耦合。

> 考试注意：单例模式常被问到，但也容易违反低耦合和可测试性原则。

---

## 工厂方法模式

> **工厂方法模式**把对象创建过程交给工厂方法，由子类或具体工厂决定创建哪种对象。

### 修改前：直接创建具体对象

```java
class UserService {
    public void save(String type) {
        if ("mysql".equals(type)) {
            MySqlDao dao = new MySqlDao();
            dao.save();
        } else if ("redis".equals(type)) {
            RedisDao dao = new RedisDao();
            dao.save();
        }
    }
}

class MySqlDao {
    public void save() {
        System.out.println("保存到 MySQL");
    }
}

class RedisDao {
    public void save() {
        System.out.println("保存到 Redis");
    }
}
```

问题：`UserService` 依赖具体类，还负责创建对象。

### 修改后：使用工厂方法

```java
interface UserDao {
    void save();
}

class MySqlDao implements UserDao {
    public void save() {
        System.out.println("保存到 MySQL");
    }
}

class RedisDao implements UserDao {
    public void save() {
        System.out.println("保存到 Redis");
    }
}

interface UserDaoFactory {
    UserDao createDao();
}

class MySqlDaoFactory implements UserDaoFactory {
    public UserDao createDao() {
        return new MySqlDao();
    }
}

class RedisDaoFactory implements UserDaoFactory {
    public UserDao createDao() {
        return new RedisDao();
    }
}

class UserService {
    private final UserDaoFactory factory;

    public UserService(UserDaoFactory factory) {
        this.factory = factory;
    }

    public void save() {
        UserDao dao = factory.createDao();
        dao.save();
    }
}
```

> 通俗理解：**对象不要自己到处 `new`，交给专门的工厂创建。**

> 优点：隐藏创建细节，符合开闭原则。
> 缺点：类数量增加，结构变复杂。

---

## 抽象工厂模式

> **抽象工厂模式**用于创建一组相关或相互依赖的对象，而不需要指定它们的具体类。

### 代码示例

```java
interface Button {
    void render();
}

interface TextBox {
    void render();
}

class WindowsButton implements Button {
    public void render() {
        System.out.println("渲染 Windows 按钮");
    }
}

class WindowsTextBox implements TextBox {
    public void render() {
        System.out.println("渲染 Windows 文本框");
    }
}

class MacButton implements Button {
    public void render() {
        System.out.println("渲染 Mac 按钮");
    }
}

class MacTextBox implements TextBox {
    public void render() {
        System.out.println("渲染 Mac 文本框");
    }
}

interface UIFactory {
    Button createButton();
    TextBox createTextBox();
}

class WindowsFactory implements UIFactory {
    public Button createButton() {
        return new WindowsButton();
    }

    public TextBox createTextBox() {
        return new WindowsTextBox();
    }
}

class MacFactory implements UIFactory {
    public Button createButton() {
        return new MacButton();
    }

    public TextBox createTextBox() {
        return new MacTextBox();
    }
}
```

> 通俗理解：**一个工厂负责生产一整套同风格产品，例如 Windows 风格按钮和文本框。**

> 优点：保证产品族一致性，便于切换一整套实现。
> 缺点：新增产品种类困难，例如新增 `Menu` 需要修改所有工厂。

## 建造者模式

> **建造者模式**将复杂对象的构建过程与表示分离，适合创建参数很多、构建步骤复杂的对象。

### 修改前：构造方法参数过多

```java
class Computer {
    private String cpu;
    private String memory;
    private String disk;
    private String gpu;

    public Computer(String cpu, String memory, String disk, String gpu) {
        this.cpu = cpu;
        this.memory = memory;
        this.disk = disk;
        this.gpu = gpu;
    }
}
```

问题：参数多时可读性差，容易传错顺序。

### 修改后：使用建造者

```java
class Computer {
    private String cpu;
    private String memory;
    private String disk;
    private String gpu;

    private Computer() {
    }

    public static class Builder {
        private Computer computer = new Computer();

        public Builder cpu(String cpu) {
            computer.cpu = cpu;
            return this;
        }

        public Builder memory(String memory) {
            computer.memory = memory;
            return this;
        }

        public Builder disk(String disk) {
            computer.disk = disk;
            return this;
        }

        public Builder gpu(String gpu) {
            computer.gpu = gpu;
            return this;
        }

        public Computer build() {
            return computer;
        }
    }
}
```

使用：

```java
Computer computer = new Computer.Builder()
        .cpu("i7")
        .memory("16GB")
        .disk("1TB")
        .gpu("RTX")
        .build();
```

> 通俗理解：**复杂对象一步步组装，而不是一次性塞一堆参数。**

> 优点：创建过程清晰，适合复杂对象。
> 缺点：需要额外的 Builder 类。

## 原型模式

> **原型模式**通过复制已有对象来创建新对象，而不是通过 `new` 从头创建。

### 代码示例

```java
class Document implements Cloneable {
    private String title;

    public Document(String title) {
        this.title = title;
    }

    public Document clone() {
        return new Document(this.title);
    }

    public void show() {
        System.out.println("文档标题：" + title);
    }
}
```

使用：

```java
Document doc1 = new Document("模板文档");
Document doc2 = doc1.clone();

doc2.show();
```

> 通俗理解：**有一个模板对象，需要新对象时复制一份。**

> 优点：创建复杂对象成本较低。
> 缺点：深拷贝和浅拷贝容易出错。

---

## 适配器模式

> **适配器模式**把一个类的接口转换成客户端期望的另一个接口，使原本接口不兼容的类能够协作。

### 修改前：接口不兼容

```java
class OldPrinter {
    public void oldPrint() {
        System.out.println("旧打印机打印");
    }
}

interface Printer {
    void print();
}
```

问题：客户端需要 `Printer`，但旧类只有 `oldPrint()`。

### 修改后：增加适配器

```java
class PrinterAdapter implements Printer {
    private final OldPrinter oldPrinter;

    public PrinterAdapter(OldPrinter oldPrinter) {
        this.oldPrinter = oldPrinter;
    }

    public void print() {
        oldPrinter.oldPrint();
    }
}
```

使用：

```java
Printer printer = new PrinterAdapter(new OldPrinter());
printer.print();
```

> 通俗理解：**插头不匹配时，加一个转换头。**

> 优点：复用旧类，不修改原有代码。
> 缺点：增加一层间接调用。

---

## 装饰器模式

> **装饰器模式**在不改变原对象结构的情况下，动态地给对象增加额外功能。

### 代码示例

```java
interface Coffee {
    double cost();
}

class SimpleCoffee implements Coffee {
    public double cost() {
        return 10.0;
    }
}

class MilkCoffee implements Coffee {
    private final Coffee coffee;

    public MilkCoffee(Coffee coffee) {
        this.coffee = coffee;
    }

    public double cost() {
        return coffee.cost() + 2.0;
    }
}

class SugarCoffee implements Coffee {
    private final Coffee coffee;

    public SugarCoffee(Coffee coffee) {
        this.coffee = coffee;
    }

    public double cost() {
        return coffee.cost() + 1.0;
    }
}
```

使用：

```java
Coffee coffee = new SugarCoffee(new MilkCoffee(new SimpleCoffee()));
System.out.println(coffee.cost());
```

> 通俗理解：**给基础对象一层一层加功能，比如咖啡加奶、加糖。**

> 优点：比继承更灵活，运行时可动态组合功能。
> 缺点：装饰层过多时对象结构复杂。

---

## 代理模式

> **代理模式**为某个对象提供一个代理对象，由代理控制对原对象的访问。

### 代码示例

```java
interface Image {
    void display();
}

class RealImage implements Image {
    private String filename;

    public RealImage(String filename) {
        this.filename = filename;
        load();
    }

    private void load() {
        System.out.println("加载图片：" + filename);
    }

    public void display() {
        System.out.println("显示图片：" + filename);
    }
}

class ImageProxy implements Image {
    private RealImage realImage;
    private String filename;

    public ImageProxy(String filename) {
        this.filename = filename;
    }

    public void display() {
        if (realImage == null) {
            realImage = new RealImage(filename);
        }
        realImage.display();
    }
}
```

> 通俗理解：**不直接找本人，而是通过代理人控制访问。**

> 优点：可以增加权限控制、延迟加载、日志记录等。
> 缺点：增加代理类，调用链变长。

---

## 外观模式

> **外观模式**为复杂子系统提供一个统一的简单接口，使客户端更容易使用系统。

### 修改前：客户端直接调用多个子系统

```java
class CPU {
    public void start() {
        System.out.println("CPU 启动");
    }
}

class Memory {
    public void load() {
        System.out.println("内存加载");
    }
}

class Disk {
    public void read() {
        System.out.println("磁盘读取");
    }
}
```

客户端需要知道很多细节：

```java
CPU cpu = new CPU();
Memory memory = new Memory();
Disk disk = new Disk();

cpu.start();
memory.load();
disk.read();
```

### 修改后：使用外观类

```java
class ComputerFacade {
    private final CPU cpu = new CPU();
    private final Memory memory = new Memory();
    private final Disk disk = new Disk();

    public void startComputer() {
        cpu.start();
        memory.load();
        disk.read();
    }
}
```

使用：

```java
ComputerFacade computer = new ComputerFacade();
computer.startComputer();
```

> 通俗理解：**把复杂操作包装成一个简单按钮。**

> 优点：简化客户端调用，降低客户端与子系统耦合。
> 缺点：外观类可能变成“大管家”，职责过重。

---

## 桥接模式

> **桥接模式**将抽象部分与实现部分分离，使二者可以独立变化。

### 代码示例

```java
interface Color {
    void applyColor();
}

class Red implements Color {
    public void applyColor() {
        System.out.println("红色");
    }
}

class Blue implements Color {
    public void applyColor() {
        System.out.println("蓝色");
    }
}

abstract class Shape {
    protected Color color;

    public Shape(Color color) {
        this.color = color;
    }

    abstract void draw();
}

class Circle extends Shape {
    public Circle(Color color) {
        super(color);
    }

    public void draw() {
        System.out.print("圆形使用");
        color.applyColor();
    }
}

class Rectangle extends Shape {
    public Rectangle(Color color) {
        super(color);
    }

    public void draw() {
        System.out.print("矩形使用");
        color.applyColor();
    }
}
```

> 通俗理解：**形状和颜色分开变化，不要为每种组合都建一个类。**

> 优点：避免类爆炸，抽象和实现可以独立扩展。
> 缺点：结构比直接继承复杂。

---

## 组合模式

> **组合模式**把对象组织成树形结构，使客户端可以用一致方式处理单个对象和组合对象。

### 代码示例

```java
interface FileComponent {
    void show();
}

class FileLeaf implements FileComponent {
    private String name;

    public FileLeaf(String name) {
        this.name = name;
    }

    public void show() {
        System.out.println("文件：" + name);
    }
}

class Folder implements FileComponent {
    private String name;
    private List<FileComponent> children = new ArrayList<>();

    public Folder(String name) {
        this.name = name;
    }

    public void add(FileComponent component) {
        children.add(component);
    }

    public void show() {
        System.out.println("文件夹：" + name);
        for (FileComponent child : children) {
            child.show();
        }
    }
}
```

> 通俗理解：**文件和文件夹都当成同一种“文件组件”处理。**

> 优点：适合树形结构，客户端处理简单。
> 缺点：可能弱化叶子节点和容器节点的差异。

---

## 享元模式

> **享元模式**通过共享对象来减少大量相似对象的内存开销。

### 代码示例

```java
class ChessPiece {
    private final String color;

    public ChessPiece(String color) {
        this.color = color;
    }

    public void display(int x, int y) {
        System.out.println(color + "棋子位置：" + x + "," + y);
    }
}

class ChessPieceFactory {
    private static final Map<String, ChessPiece> pieces = new HashMap<>();

    public static ChessPiece getPiece(String color) {
        if (!pieces.containsKey(color)) {
            pieces.put(color, new ChessPiece(color));
        }
        return pieces.get(color);
    }
}
```

使用：

```java
ChessPiece black1 = ChessPieceFactory.getPiece("黑色");
ChessPiece black2 = ChessPieceFactory.getPiece("黑色");

black1.display(1, 2);
black2.display(3, 4);
```

> 通俗理解：**相同的部分共享，不同的位置等外部状态单独传入。**

> 优点：节省内存。
> 缺点：需要区分内部状态和外部状态，设计复杂。

## 策略模式

> **策略模式**定义一系列算法，把每个算法封装起来，并使它们可以互相替换。

### 修改前：大量 `if-else`

```java
class DiscountService {
    public double calculate(String type, double price) {
        if ("vip".equals(type)) {
            return price * 0.8;
        } else if ("student".equals(type)) {
            return price * 0.9;
        }
        return price;
    }
}
```

问题：新增折扣类型需要修改原方法，违反 OCP。

### 修改后：使用策略模式

```java
interface DiscountStrategy {
    double discount(double price);
}

class VipDiscount implements DiscountStrategy {
    public double discount(double price) {
        return price * 0.8;
    }
}

class StudentDiscount implements DiscountStrategy {
    public double discount(double price) {
        return price * 0.9;
    }
}

class DiscountService {
    public double calculate(double price, DiscountStrategy strategy) {
        return strategy.discount(price);
    }
}
```

使用：

```java
DiscountService service = new DiscountService();
double result = service.calculate(100.0, new VipDiscount());
System.out.println(result);
```

> 通俗理解：**把不同算法做成不同对象，需要哪个就换哪个。**

> 优点：消除复杂条件判断，符合开闭原则。
> 缺点：策略类数量可能增加。

---

## 观察者模式

> **观察者模式**定义对象之间的一对多依赖关系。当一个对象状态发生变化时，所有依赖它的对象都会收到通知。

### 代码示例

```java
interface Observer {
    void update(String message);
}

class EmailObserver implements Observer {
    public void update(String message) {
        System.out.println("邮件通知：" + message);
    }
}

class SmsObserver implements Observer {
    public void update(String message) {
        System.out.println("短信通知：" + message);
    }
}

class Subject {
    private List<Observer> observers = new ArrayList<>();

    public void addObserver(Observer observer) {
        observers.add(observer);
    }

    public void notifyObservers(String message) {
        for (Observer observer : observers) {
            observer.update(message);
        }
    }
}
```

使用：

```java
Subject subject = new Subject();
subject.addObserver(new EmailObserver());
subject.addObserver(new SmsObserver());

subject.notifyObservers("订单已支付");
```

> 通俗理解：**一个对象发生变化，订阅它的人都会收到通知。**

> 优点：发布者和订阅者解耦，便于扩展。
> 缺点：通知链复杂时调试困难，可能出现循环通知。

---

## 模板方法模式

> **模板方法模式**在父类中定义算法骨架，把某些具体步骤延迟到子类实现。

### 代码示例

```java
abstract class DataImporter {
    public final void importData() {
        readData();
        parseData();
        saveData();
    }

    protected abstract void readData();

    protected void parseData() {
        System.out.println("解析数据");
    }

    protected void saveData() {
        System.out.println("保存数据");
    }
}

class CsvImporter extends DataImporter {
    protected void readData() {
        System.out.println("读取 CSV 文件");
    }
}

class ExcelImporter extends DataImporter {
    protected void readData() {
        System.out.println("读取 Excel 文件");
    }
}
```

> 通俗理解：**流程固定，某些步骤由子类自己实现。**

> 优点：复用流程骨架，减少重复代码。
> 缺点：依赖继承，父类变化可能影响子类。

---

## 命令模式

> **命令模式**把请求封装成对象，从而可以参数化请求、记录日志、撤销操作或延迟执行。

### 代码示例

```java
interface Command {
    void execute();
}

class Light {
    public void on() {
        System.out.println("开灯");
    }

    public void off() {
        System.out.println("关灯");
    }
}

class LightOnCommand implements Command {
    private final Light light;

    public LightOnCommand(Light light) {
        this.light = light;
    }

    public void execute() {
        light.on();
    }
}

class RemoteControl {
    private Command command;

    public void setCommand(Command command) {
        this.command = command;
    }

    public void pressButton() {
        command.execute();
    }
}
```

> 通俗理解：**把“做一件事”包装成一个对象。**

> 优点：请求发送者和执行者解耦，方便撤销、队列、日志。
> 缺点：命令类数量可能很多。

---

## 责任链模式

> **责任链模式**让多个对象都有机会处理请求，请求沿着链传递，直到某个对象处理为止。

### 代码示例

```java
abstract class Handler {
    protected Handler next;

    public void setNext(Handler next) {
        this.next = next;
    }

    public abstract void handle(int amount);
}

class Manager extends Handler {
    public void handle(int amount) {
        if (amount <= 1000) {
            System.out.println("经理审批");
        } else if (next != null) {
            next.handle(amount);
        }
    }
}

class Director extends Handler {
    public void handle(int amount) {
        if (amount <= 5000) {
            System.out.println("总监审批");
        } else if (next != null) {
            next.handle(amount);
        }
    }
}
```

使用：

```java
Handler manager = new Manager();
Handler director = new Director();

manager.setNext(director);
manager.handle(3000);
```

> 通俗理解：**请求一级一级往上交，谁能处理谁处理。**

> 优点：发送者不需要知道具体处理者，链可动态调整。
> 缺点：请求可能无人处理，链过长影响性能和调试。

---

## 状态模式

> **状态模式**允许对象在内部状态改变时改变其行为，看起来像是修改了对象的类。

### 修改前：大量状态判断

```java
class Order {
    private String state = "new";

    public void pay() {
        if ("new".equals(state)) {
            System.out.println("支付订单");
            state = "paid";
        } else {
            System.out.println("当前状态不能支付");
        }
    }

    public void ship() {
        if ("paid".equals(state)) {
            System.out.println("发货");
            state = "shipped";
        } else {
            System.out.println("当前状态不能发货");
        }
    }
}
```

### 修改后：使用状态模式

```java
interface OrderState {
    void pay(OrderContext context);
    void ship(OrderContext context);
}

class NewState implements OrderState {
    public void pay(OrderContext context) {
        System.out.println("支付订单");
        context.setState(new PaidState());
    }

    public void ship(OrderContext context) {
        System.out.println("未支付，不能发货");
    }
}

class PaidState implements OrderState {
    public void pay(OrderContext context) {
        System.out.println("已经支付，不能重复支付");
    }

    public void ship(OrderContext context) {
        System.out.println("发货");
    }
}

class OrderContext {
    private OrderState state = new NewState();

    public void setState(OrderState state) {
        this.state = state;
    }

    public void pay() {
        state.pay(this);
    }

    public void ship() {
        state.ship(this);
    }
}
```

> 通俗理解：**不同状态下，同一个操作有不同表现。**

> 优点：消除复杂状态判断，状态行为清晰。
> 缺点：状态类数量增加。

## 迭代器模式

> **迭代器模式**提供一种顺序访问集合元素的方法，而不暴露集合内部结构。

### 代码示例

```java
interface Iterator {
    boolean hasNext();
    String next();
}

class NameCollection {
    private String[] names = {"Alice", "Bob", "Cindy"};

    public Iterator iterator() {
        return new NameIterator();
    }

    private class NameIterator implements Iterator {
        private int index = 0;

        public boolean hasNext() {
            return index < names.length;
        }

        public String next() {
            return names[index++];
        }
    }
}
```

使用：

```java
NameCollection collection = new NameCollection();
Iterator iterator = collection.iterator();

while (iterator.hasNext()) {
    System.out.println(iterator.next());
}
```

> 通俗理解：**不管集合内部怎么存，都用统一方式一个个取出来。**

> 优点：隐藏集合内部结构，统一遍历方式。
> 缺点：简单集合中可能显得多余。

## 中介者模式

> **中介者模式**用一个中介对象封装多个对象之间的交互，使对象之间不直接相互引用。

### 代码示例

```java
class ChatRoom {
    public void sendMessage(String from, String message) {
        System.out.println(from + " 发送消息：" + message);
    }
}

class User {
    private String name;
    private ChatRoom chatRoom;

    public User(String name, ChatRoom chatRoom) {
        this.name = name;
        this.chatRoom = chatRoom;
    }

    public void send(String message) {
        chatRoom.sendMessage(name, message);
    }
}
```

> 通俗理解：**大家不互相直接联系，而是通过群聊或中介沟通。**

> 优点：减少对象之间的网状依赖。
> 缺点：中介者可能变得过于复杂。

## 备忘录模式

> **备忘录模式**在不破坏封装的前提下保存对象状态，以便以后恢复。

### 代码示例

```java
class EditorMemento {
    private final String content;

    public EditorMemento(String content) {
        this.content = content;
    }

    public String getContent() {
        return content;
    }
}

class Editor {
    private String content;

    public void setContent(String content) {
        this.content = content;
    }

    public EditorMemento save() {
        return new EditorMemento(content);
    }

    public void restore(EditorMemento memento) {
        this.content = memento.getContent();
    }

    public void show() {
        System.out.println(content);
    }
}
```

> 通俗理解：**给对象当前状态拍一张快照，以后可以回滚。**

> 优点：支持撤销、恢复。
> 缺点：保存状态可能占用较多内存。

## 访问者模式

> **访问者模式**把作用于某些对象结构上的操作封装到访问者中，使得可以在不改变对象结构的情况下增加新操作。

### 代码示例

```java
interface Element {
    void accept(Visitor visitor);
}

class Book implements Element {
    public void accept(Visitor visitor) {
        visitor.visitBook(this);
    }
}

class Computer implements Element {
    public void accept(Visitor visitor) {
        visitor.visitComputer(this);
    }
}

interface Visitor {
    void visitBook(Book book);
    void visitComputer(Computer computer);
}

class PriceVisitor implements Visitor {
    public void visitBook(Book book) {
        System.out.println("计算书的价格");
    }

    public void visitComputer(Computer computer) {
        System.out.println("计算电脑的价格");
    }
}
```

> 通俗理解：**对象结构不变，把新操作交给访问者去做。**

> 优点：便于增加新操作。
> 缺点：新增元素类型困难，因为所有访问者接口都要改。

## 解释器模式

> **解释器模式**为某种语言或表达式定义语法表示，并提供解释执行的方法。

### 代码示例

```java
interface Expression {
    boolean interpret(String context);
}

class TerminalExpression implements Expression {
    private String data;

    public TerminalExpression(String data) {
        this.data = data;
    }

    public boolean interpret(String context) {
        return context.contains(data);
    }
}

class OrExpression implements Expression {
    private Expression left;
    private Expression right;

    public OrExpression(Expression left, Expression right) {
        this.left = left;
        this.right = right;
    }

    public boolean interpret(String context) {
        return left.interpret(context) || right.interpret(context);
    }
}
```

使用：

```java
Expression java = new TerminalExpression("Java");
Expression python = new TerminalExpression("Python");
Expression language = new OrExpression(java, python);

System.out.println(language.interpret("I like Java"));
```

> 通俗理解：**给一种简单语言定义规则，然后按规则解释它。**

> 优点：适合简单语法规则。
> 缺点：复杂语法会导致类数量膨胀，性能较差。

| ---- | --- | ----------- | ---------- |
| 单例   | 创建型 | 保证唯一实例      | 配置、日志      |

| 抽象工厂 | 创建型 | 创建一组相关对象    | 跨平台 UI     |
| 建造者  | 创建型 | 分步骤构建复杂对象   | 参数多、构造复杂   |
| 原型   | 创建型 | 复制已有对象      | 创建成本高      |
| 适配器  | 结构型 | 转换接口        | 旧接口兼容      |
| 装饰器  | 结构型 | 动态增加功能      | 加功能组合      |
| 代理   | 结构型 | 控制对象访问      | 权限、缓存、延迟加载 |
| 外观   | 结构型 | 简化子系统入口     | 封装复杂调用     |
| 桥接   | 结构型 | 抽象与实现分离     | 多维变化       |
| 组合   | 结构型 | 树形结构统一处理    | 文件目录、菜单    |
| 享元   | 结构型 | 共享相似对象      | 大量细粒度对象    |
| 策略   | 行为型 | 替换算法        | 折扣、排序、支付   |
| 观察者  | 行为型 | 状态变化通知订阅者   | 事件通知       |
| 模板方法 | 行为型 | 固定流程，子类实现步骤 | 导入流程、处理流程  |
| 命令   | 行为型 | 请求封装为对象     | 撤销、队列、遥控器  |
| 责任链  | 行为型 | 请求沿链传递      | 审批、过滤器     |
| 状态   | 行为型 | 状态决定行为      | 订单状态、工作流   |
| 迭代器  | 行为型 | 统一遍历集合      | 集合遍历       |
| 中介者  | 行为型 | 用中介协调对象     | 聊天室、组件通信   |
| 备忘录  | 行为型 | 保存和恢复状态     | 撤销、快照      |
| 访问者  | 行为型 | 操作与对象结构分离   | 对稳定结构增加操作  |
| 解释器  | 行为型 | 解释语法表达式     | 规则引擎、简单表达式 |
