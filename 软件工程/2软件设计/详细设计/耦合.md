## 耦合

**耦合**指模块之间相互依赖的程度。耦合越强，一个模块修改时越容易影响其他模块；耦合越弱，系统越容易维护、测试和扩展。

耦合程度从低到高大致是：**数据耦合 < 印记耦合 < 控制耦合 < 公共耦合 < 内容耦合**。

---

### 耦合类型对比

| 类型   | 典型表现                        | 问题            | 修改方向          |
| ---- | --------------------------- | ------------- | ------------- |
| 数据耦合 | 只传必要参数                      | 问题较小          | 保持接口简单        |
| 印记耦合 | 传整个对象但只用部分字段                | 依赖了不必要的数据结构   | 只传所需字段或 DTO   |
| 控制耦合 | 传 `flag`、`type`、`mode` 控制逻辑 | 调用者依赖被调用者内部流程 | 拆方法、多态、策略模式   |
| 公共耦合 | 使用全局变量、共享数据区                | 修改影响范围大，难追踪   | 封装共享状态，使用依赖注入 |
| 内容耦合 | 直接访问或修改内部字段、内部逻辑            | 破坏封装，维护困难     | 通过接口、方法、封装访问  |

----

### 数据耦合

**数据耦合**指模块之间只通过必要的简单数据进行交互，例如传递用户名、金额、编号等。

```java
class UserService {
    public void updateUsername(int userId, String newName) {
        // 只传递必要数据
        System.out.println("修改用户 " + userId + " 的名字为 " + newName);
    }
}
```

---

### 印记耦合

**印记耦合**指一个模块把整个数据结构传给另一个模块，但对方实际上只使用其中一部分字段。

#### 修改前：传整个对象，但只用一个字段

```java
class User {
    int id;
    String name;
    String password;
    String address;
}

class EmailService {
    public void sendWelcomeEmail(User user) {
        // 实际只用了 name
        System.out.println("欢迎你，" + user.name);
    }
}
```

这里 `EmailService` 依赖了整个 `User` 结构。以后 `User` 类变化，可能会影响 `EmailService`。

#### 修改后：只传需要的字段

```java
class EmailService {
    public void sendWelcomeEmail(String username) {
        System.out.println("欢迎你，" + username);
    }
}
```

#### 修改后：使用 DTO

```java
class UserWelcomeDTO {
    String name;

    public UserWelcomeDTO(String name) {
        this.name = name;
    }
}

class EmailService {
    public void sendWelcomeEmail(UserWelcomeDTO dto) {
        System.out.println("欢迎你，" + dto.name);
    }
}
```

> **只传所需字段，或者定义专门的 DTO**。

### 控制耦合

**控制耦合**指一个模块通过传递控制参数，如 `flag`、`type`、`mode`，来控制另一个模块内部执行哪一段逻辑。

#### 修改前：用 flag 控制内部逻辑

```java
class ReportService {
    public void exportReport(String data, String type) {
        if ("PDF".equals(type)) {
            System.out.println("导出 PDF 报表：" + data);
        } else if ("EXCEL".equals(type)) {
            System.out.println("导出 Excel 报表：" + data);
        }
    }
}
```

这里调用者必须知道 `ReportService` 内部支持哪些 `type`，因此调用者和被调用者逻辑绑定较紧。

#### 修改方法一：拆分方法

```java
class ReportService {
    public void exportPdf(String data) {
        System.out.println("导出 PDF 报表：" + data);
    }

    public void exportExcel(String data) {
        System.out.println("导出 Excel 报表：" + data);
    }
}
```

这样调用者不再传 `type` 控制内部流程，而是直接调用明确的方法。

#### 修改方法二：使用策略模式

```java
interface ReportExporter {
    void export(String data);
}

class PdfExporter implements ReportExporter {
    public void export(String data) {
        System.out.println("导出 PDF 报表：" + data);
    }
}

class ExcelExporter implements ReportExporter {
    public void export(String data) {
        System.out.println("导出 Excel 报表：" + data);
    }
}

class ReportService {
    public void exportReport(String data, ReportExporter exporter) {
        exporter.export(data);
    }
}
```

使用方式：

```java
ReportService service = new ReportService();

service.exportReport("销售数据", new PdfExporter());
service.exportReport("销售数据", new ExcelExporter());
```

### 公共耦合

**公共耦合**指多个模块共同依赖同一个全局变量、全局对象或共享数据区。

#### 修改前：多个模块共享全局变量

```java
class GlobalConfig {
    public static String databaseUrl = "localhost:3306";
}

class UserDao {
    public void connect() {
        System.out.println("连接数据库：" + GlobalConfig.databaseUrl);
    }
}

class OrderDao {
    public void connect() {
        System.out.println("连接数据库：" + GlobalConfig.databaseUrl);
    }
}
```

问题是：任何地方都可以修改 `GlobalConfig.databaseUrl`，导致系统行为不可控。

#### 修改后：封装共享状态

```java
class DatabaseConfig {
    private final String databaseUrl;

    public DatabaseConfig(String databaseUrl) {
        this.databaseUrl = databaseUrl;
    }

    public String getDatabaseUrl() {
        return databaseUrl;
    }
}

class UserDao {
    private final DatabaseConfig config;

    public UserDao(DatabaseConfig config) {
        this.config = config;
    }

    public void connect() {
        System.out.println("连接数据库：" + config.getDatabaseUrl());
    }
}
```

> **大家都直接改同一个公共黑板，谁改了什么很难追踪**。

> 修改方向：**减少全局变量，封装共享状态，通过构造方法、配置对象或依赖注入传递依赖**。

### 内容耦合

**内容耦合**指一个模块直接访问、修改或依赖另一个模块的内部实现细节。

#### 修改前：直接修改对象内部字段

```java
class Account {
    public double balance;
}

class PaymentService {
    public void pay(Account account, double amount) {
        // 直接修改 Account 内部数据
        account.balance -= amount;
    }
}
```

问题是：`PaymentService` 直接操作了 `Account` 的内部状态。如果以后余额计算规则改变，例如增加冻结金额、手续费、余额校验，就会很难维护。

#### 修改后：通过接口或方法访问

```java
class Account {
    private double balance;

    public Account(double balance) {
        this.balance = balance;
    }

    public void withdraw(double amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("金额必须大于 0");
        }

        if (balance < amount) {
            throw new IllegalArgumentException("余额不足");
        }

        balance -= amount;
    }

    public double getBalance() {
        return balance;
    }
}

class PaymentService {
    public void pay(Account account, double amount) {
        account.withdraw(amount);
    }
}
```

> 修改方向：**隐藏内部实现，通过 public 方法、接口或服务层进行访问**。
