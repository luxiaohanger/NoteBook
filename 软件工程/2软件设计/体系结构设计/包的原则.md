## 包设计原则

包设计原则用于指导**如何划分包、如何控制包之间的依赖关系**。核心目标是：让系统结构清晰、依赖方向稳定、修改影响范围小。

| 原则           | 核心含义             | 解决的问题           |
| ------------ | ---------------- | --------------- |
| 重用发布等价原则 REP | 可重用的类应一起发布       | 包的发布边界不清        |
| 共同重用原则 CRP   | 经常一起被重用的类放在同一个包  | 使用一个类却被迫依赖无关类   |
| 共同封闭原则 CCP   | 会因同一原因变化的类放在同一个包 | 一次修改影响多个包       |
| 无环依赖原则 ADP   | 包依赖图不能出现环        | 包之间互相依赖，难以编译和维护 |
| 稳定依赖原则 SDP   | 不稳定包依赖稳定包        | 依赖方向混乱          |
| 稳定抽象原则 SAP   | 越稳定的包越应该抽象       | 稳定包难以扩展         |

---

## 重用发布等价原则 REP

重用发布等价原则要求：**包是发布和重用的基本单位**。如果一组类经常作为一个整体被复用，那么它们应该放在同一个包中，并以同一个版本发布。

> 能一起被别人拿去用的类，就应该一起打包、一起发布。

> 易错点：不要把完全无关的类仅仅因为“都可能被复用”就放进同一个包，重用边界必须清晰。

---

## 共同重用原则 CRP

共同重用原则要求：**一个包中的类应该被共同重用**。如果使用者只需要包中的一个类，却不得不依赖很多无关类，说明包划分过粗。

例如：

```text
util
├── StringUtil
├── DateUtil
├── FileUtil
├── NetworkUtil
```

如果 `StringUtil` 和 `NetworkUtil` 没有共同使用场景，把它们放在一个大 `util` 包中可能违反 CRP。

更合理的划分：

```text
text
└── StringUtil

time
└── DateUtil

io
└── FileUtil

network
└── NetworkUtil
```

> 使用一个包时，不应该被迫依赖一堆自己不用的类。

> 易错点：`util`、`common`、`misc` 这类大杂烩包在考试中通常是坏味道。

---

## 共同封闭原则 CCP

共同封闭原则要求：**会因为同一个原因发生变化的类，应该放在同一个包中**。这样一次需求变化通常只影响一个包，而不会扩散到多个包。

例如校园卡系统中：

```text
card
├── CampusCard
├── CardService
├── CardRepository
```

如果“校园卡规则”变化，这些类可能一起修改，因此放在同一个包比较合理。

> 会一起变的类，就应该放在一起。

> 易错点：CRP 关注“重用时是否一起用”，CCP 关注“修改时是否一起变”，二者角度不同。

---

## 无环依赖原则 ADP

无环依赖原则要求：**包之间的依赖关系不能形成环**。

错误示例：

```text
A 包依赖 B 包
B 包依赖 C 包
C 包依赖 A 包
```

形成环：

$$
A \rightarrow B \rightarrow C \rightarrow A
$$

这种结构会导致编译、测试、发布都变得困难。修改一个包可能牵动整个环。

解决方法通常有：

| 方法    | 含义               |
| ----- | ---------------- |
| 抽取公共包 | 把共同依赖的内容抽到独立包    |
| 依赖倒置  | 让高层包定义接口，低层包实现接口 |
| 拆分包   | 把造成循环依赖的类拆出去     |

> 包依赖图应该是有向无环图。

> 易错点：类之间可以有调用关系，但包之间不能形成循环依赖。

---

## 稳定依赖原则 SDP

稳定依赖原则要求：**包应该依赖比自己更稳定的包**。

稳定包通常具有这些特点：

| 特征     | 说明             |
| ------ | -------------- |
| 被很多包依赖 | 改动影响范围大        |
| 自己依赖较少 | 不容易受别人变化影响     |
| 职责核心   | 通常包含接口、抽象、基础模型 |

不稳定包通常具有这些特点：

| 特征        | 说明        |
| --------- | --------- |
| 依赖很多包     | 容易受别人变化影响 |
| 被依赖较少     | 修改影响范围小   |
| 靠近界面或外部系统 | 变化较频繁     |

例如：

```text
ui  →  service  →  domain  →  infrastructure-interface
```

一般不应该让 `domain` 依赖 `ui`，因为界面层变化频繁，而领域层应该稳定。

> 易变的包可以依赖稳定的包，稳定的包不应该依赖易变的包。

> 易错点：稳定不是指“永远不变”，而是指“改动代价高、不应频繁变化”。

---

## 稳定抽象原则 SAP

稳定抽象原则要求：**越稳定的包，越应该包含抽象内容；越不稳定的包，越可以包含具体实现**。

也就是说，被很多模块依赖的稳定包，不应该全是具体类，否则一旦需求变化很难扩展。它更适合放：

```text
接口
抽象类
核心模型
稳定契约
```

例如：

```text
payment-api
├── PaymentService
├── PaymentResult
└── PaymentException
```

具体实现放在：

```text
payment-alipay
└── AlipayPaymentServiceImpl

payment-wechat
└── WechatPaymentServiceImpl
```

这样上层依赖稳定抽象：

$$
上层模块 \rightarrow payment\text{-}api \leftarrow 具体支付实现
$$

> 越多人依赖的包，越应该提供稳定接口，而不是写死具体实现。

> 易错点：稳定包不是不能变化，而是应该通过抽象降低变化带来的影响。

---

## 包设计原则的考试区分

| 原则  | 关键词      | 常见判断方式         |
| --- | -------- | -------------- |
| REP | 重用、发布、版本 | 是否应该一起发布       |
| CRP | 共同使用     | 使用一个类是否被迫依赖无关类 |
| CCP | 共同变化     | 一次变化是否集中在一个包内  |
| ADP | 不能有环     | 包依赖图是否成环       |
| SDP | 依赖稳定方向   | 是否由易变包依赖稳定包    |
| SAP | 稳定包要抽象   | 被广泛依赖的包是否以接口为主 |

> 记忆方式：前三个原则主要解决“包里应该放什么类”，后三个原则主要解决“包之间应该怎么依赖”。
