## 架构风格

**架构风格**通常指软件系统的宏观组织方式，它关注系统由哪些主要构件组成、构件之间如何连接、如何通信、如何部署，以及如何满足性能、安全、可维护性等非功能需求。

-------
## 常见体系结构风格

| 架构     | 核心思想       | 适用场景        | 主要优点     | 主要缺点        |
| ------ | ---------- | ----------- | -------- | ----------- |
| 分层架构   | 按职责分层      | 企业应用、Web 系统 | 职责清晰，易维护 | 可能有性能开销，忌跨层 |
| C/S 架构 | 客户端请求服务器   | 桌面客户端、网络应用  | 数据集中管理   | 依赖网络和服务器    |
| B/S 架构 | 浏览器访问服务器   | Web 系统      | 部署更新方便   | 复杂交互依赖前端    |
| MVC    | 模型、视图、控制分离 | Web、GUI 系统  | 界面和业务分离  | 小系统可能过度设计   |
| 管道-过滤器 | 数据流水线处理    | 编译器、数据处理    | 复用性好，易组合 | 不适合强交互      |
| 仓库架构   | 中央数据仓库共享数据 | 数据密集系统      | 数据统一管理   | 仓库易成瓶颈      |
| 事件驱动   | 发布事件，订阅处理  | 异步系统、消息系统   | 解耦、易扩展   | 调试困难，流程隐蔽   |
| 微服务    | 按业务能力拆服务   | 大型互联网系统     | 独立部署和扩展  | 分布式复杂度高     |
| SOA    | 以服务为中心     | 企业系统集成      | 服务复用     | 治理复杂        |
| 黑板架构   | 多知识源协作     | 专家系统、AI 推理  | 适合复杂问题   | 控制和调试复杂     |
| P2P    | 节点平等互联     | 文件共享、区块链    | 去中心化、容错好 | 安全和一致性难     |

## 分层架构

**分层架构**把系统按照职责分成若干层，每一层只依赖相邻或下层服务，常见于 Web 系统和企业应用。

常见三层结构：

| 层次    | 作用          | 常见对象                     |
| ----- | ----------- | ------------------------ |
| 表示层   | 负责界面展示和用户交互 | `Controller`、`View`、`VO` |
| 业务逻辑层 | 负责业务规则和流程控制 | `Service`、`BO`           |
| 数据访问层 | 负责数据库访问和持久化 | `DAO`、`Repository`、`PO`  |

简单结构：

```text
[ 表示层 Controller / View ]
              |
              v
[ 业务逻辑层 Service / Domain ]
              |
              v
[ 数据访问层 DAO / Repository ]
              |
              v
[ 数据库 Database ]
```

**上层负责接待用户，中层负责业务规则，下层负责存取数据。**

> 优点：职责清晰，便于维护、测试和替换某一层。
> 缺点：层间调用可能带来性能开销；如果随意跨层访问，会破坏架构。

## 客户端-服务器架构

**客户端-服务器架构**将系统分为客户端和服务器。客户端负责用户交互，服务器负责业务处理、数据管理和资源提供。

```text
[ Client 客户端 ]  <----网络请求/响应---->  [ Server 服务器 ]
```

> 优点：数据集中管理，客户端较轻，便于统一维护和安全控制。
> 缺点：依赖网络；服务器可能成为性能瓶颈或单点故障。

---

## 浏览器-服务器架构

**B/S 架构**是客户端-服务器架构的一种特殊形式。客户端主要是浏览器，服务器提供 Web 页面、接口和数据服务。

```text
[ Browser 浏览器 ]
        |
     HTTP/HTTPS
        |
[ Web Server / Application Server ]
        |
[ Database ]
```

**用户用浏览器访问系统，主要逻辑在服务器端完成。**

> 优点：客户端无需安装复杂软件，更新方便，跨平台。
> 缺点：依赖网络和服务器性能；浏览器端复杂交互需要前端技术支持。

## MVC 架构

**MVC** 将系统分为 Model、View、Controller 三部分，用于分离数据、界面和控制逻辑。

| 部分         | 含义  | 作用               |
| ---------- | --- | ---------------- |
| Model      | 模型  | 保存数据和业务逻辑        |
| View       | 视图  | 展示界面             |
| Controller | 控制器 | 接收用户请求，调用模型，选择视图 |

```text
用户操作
   |
   v
[ Controller ] ---> [ Model ]
      |
      v
[ View ]
```

**Controller 管请求，Model 管数据和业务，View 管显示。**

> 优点：界面和业务逻辑分离，便于维护和复用。
> 缺点：组件交互增多，小系统可能显得复杂。

## 管道-过滤器架构

**管道-过滤器架构**把系统处理过程拆成多个过滤器，每个过滤器完成一个独立处理步骤，数据通过管道依次流动。

```text
输入数据 -> [ 过滤器1 ] -> [ 过滤器2 ] -> [ 过滤器3 ] -> 输出数据
```

### 代码示例

```java
interface Filter {
    String handle(String input);
}

class TrimFilter implements Filter {
    public String handle(String input) {
        return input.trim();
    }
}

class UpperCaseFilter implements Filter {
    public String handle(String input) {
        return input.toUpperCase();
    }
}

class Pipeline {
    public String process(String input) {
        Filter trim = new TrimFilter();
        Filter upper = new UpperCaseFilter();

        String result = trim.handle(input);
        result = upper.handle(result);

        return result;
    }
}
```

**像流水线一样，数据经过一道道工序逐步加工。**

> 优点：每个过滤器职责单一，便于复用、替换和并行处理。
> 缺点：不适合强交互系统；需要统一数据格式；错误处理可能复杂。

## 仓库架构

**仓库架构**以一个中心数据仓库为核心，多个子系统围绕仓库读取、写入或共享数据。

```text
[ 子系统 A ] \
[ 子系统 B ] ---> [ 中央数据仓库 ]
[ 子系统 C ] /
```

### 代码示例

```java
class DataRepository {
    public void save(String data) {
        System.out.println("保存数据：" + data);
    }

    public String read() {
        return "共享数据";
    }
}

class AnalysisModule {
    public void analyze(DataRepository repository) {
        String data = repository.read();
        System.out.println("分析：" + data);
    }
}

class ReportModule {
    public void generate(DataRepository repository) {
        String data = repository.read();
        System.out.println("生成报表：" + data);
    }
}
```

> 优点：数据集中管理，便于共享和一致性维护。
> 缺点：中央仓库容易成为性能瓶颈；仓库结构变化会影响多个子系统。

## 事件驱动架构

**事件驱动架构**中，系统组件通过事件进行通信。事件发布者不直接调用具体接收者，而是发布事件；订阅者监听并处理事件。

```text
[ 事件发布者 ] ---> [ 事件总线 / 消息队列 ] ---> [ 事件订阅者 ]
```

### 代码示例

```java
interface EventListener {
    void onEvent(String event);
}

class EmailListener implements EventListener {
    public void onEvent(String event) {
        System.out.println("发送邮件，事件：" + event);
    }
}

class LogListener implements EventListener {
    public void onEvent(String event) {
        System.out.println("记录日志，事件：" + event);
    }
}

class EventBus {
    private final List<EventListener> listeners = new ArrayList<>();

    public void register(EventListener listener) {
        listeners.add(listener);
    }

    public void publish(String event) {
        for (EventListener listener : listeners) {
            listener.onEvent(event);
        }
    }
}
```

> **一个模块只负责喊一声“发生了某事”，关心这件事的模块自己去处理。**

> 优点：发布者和订阅者解耦，便于扩展异步处理。
> 缺点：事件流难追踪，调试复杂，可能出现消息丢失或顺序问题。

## 微服务架构

**微服务架构**把系统拆分为多个小型服务，每个服务围绕一个业务能力独立开发、部署和运行，服务之间通过网络接口通信。

```text
[ 用户服务 ]   [ 订单服务 ]   [ 支付服务 ]   [ 库存服务 ]
     \             |             |             /
              [ API Gateway ]
```

> 优点：服务可独立部署、独立扩展，适合大型复杂系统。
> 缺点：分布式通信、数据一致性、服务治理、监控运维复杂。

---

## 面向服务架构

**SOA**，即 Service-Oriented Architecture，面向服务架构。它把系统功能封装成可复用服务，服务之间通过标准接口进行通信。

```text
[ 系统 A ] ---> [ 服务总线 / 服务接口 ] <--- [ 系统 B ]
                         |
                    [ 公共服务 ]
```

> 优点：服务复用性强，适合企业级系统集成。
> 缺点：服务治理复杂，接口设计和版本管理要求高。

---

## 黑板架构

**黑板架构**是一种特殊的仓库风格，由多个知识源围绕一个公共黑板协作解决问题。各模块根据黑板上的当前状态决定是否参与处理。

```text
[ 知识源 A ] \
[ 知识源 B ] ---> [ 黑板 Blackboard ] <--- [ 控制器 ]
[ 知识源 C ] /
```


> 优点：适合问题求解过程复杂、解决步骤不固定的系统。
> 缺点：控制逻辑复杂，调试和性能优化困难。

---

## 点对点架构

**点对点架构**中，每个节点既可以作为客户端请求资源，也可以作为服务器提供资源，没有绝对中心服务器。

```text
[ 节点 A ] <----> [ 节点 B ]
    ^               |
    |               v
[ 节点 D ] <----> [ 节点 C ]
```


> 优点：去中心化，扩展性和容错性较好。
> 缺点：节点管理、安全控制、数据一致性较难。

常见场景：P2P 文件共享、区块链网络、分布式存储。
