## 代码质量常见问题总览

代码质量问题通常不是“代码不能运行”，而是“代码难读、难改、难测、容易出错”。

常见问题可以概括为：

| 问题类型     | 典型表现                    | 修改方向         |
| -------- | ----------------------- | ------------ |
| 命名不清晰    | `a`、`b`、`data` 含义不明     | 使用能表达业务含义的名称 |
| 函数过长     | 一个函数做很多事                | 拆分函数，单一职责    |
| 重复代码     | 多处复制相同逻辑                | 提取函数、提取类     |
| 魔法值      | 代码中直接写 `60`、`100`、`"A"` | 提取常量         |
| 过度使用全局变量 | 多处共享可变状态                | 封装状态，显式传参    |
| 高耦合      | 一个类直接依赖很多具体类            | 面向接口编程       |
| 低内聚      | 一个类承担多种职责               | 拆分类，职责分离     |
| 复杂条件判断   | 多层 `if-else`            | 提前返回、策略模式、多态 |
| 异常处理不当   | 吞掉异常或乱抛异常               | 明确错误语义       |
| 注释质量差    | 注释重复代码或与代码不一致           | 用清晰命名代替无效注释  |

---

## 命名不清晰

> 命名应当表达变量、函数、类的真实含义。

### 问题代码

```java
public double calc(double a, double b) {
    return a * b * 0.8;
}
```

问题：

| 问题           | 说明               |
| ------------ | ---------------- |
| `calc` 太笼统   | 不知道计算什么          |
| `a`、`b` 含义不明 | 不知道是价格、数量还是折扣    |
| `0.8` 是魔法值   | 不知道为什么乘以 `$0.8$` |

### 修改后代码

```java
public double calculateDiscountedPrice(double unitPrice, int quantity) {
    final double DISCOUNT_RATE = 0.8;
    return unitPrice * quantity * DISCOUNT_RATE;
}
```

> 通俗理解：好的命名能让读者不看注释也知道代码在做什么。

> 易错点：不要为了短而牺牲可读性，例如 `u`、`p`、`num` 在业务代码中通常不够清楚。

## 函数过长

> 一个函数应该只完成一个明确的任务。

### 问题代码

```java
public void handleOrder(Order order) {
    if (order == null) {
        throw new IllegalArgumentException("order is null");
    }

    double total = 0;
    for (OrderItem item : order.getItems()) {
        total += item.getPrice() * item.getQuantity();
    }

    if (total > 1000) {
        total = total * 0.9;
    }

    order.setTotalPrice(total);
    order.setStatus("PAID");

    System.out.println("Order paid, total = " + total);
}
```

问题：

| 问题     | 说明       |
| ------ | -------- |
| 校验订单   | 一个职责     |
| 计算金额   | 一个职责     |
| 修改订单状态 | 一个职责     |
| 输出日志   | 一个职责     |
| 函数职责过多 | 不利于测试和维护 |

### 修改后代码

```java
public void handleOrder(Order order) {
    validateOrder(order);

    double totalPrice = calculateTotalPrice(order);
    order.markAsPaid(totalPrice);

    logOrderPaid(order, totalPrice);
}

private void validateOrder(Order order) {
    if (order == null) {
        throw new IllegalArgumentException("order is null");
    }
}

private double calculateTotalPrice(Order order) {
    double total = 0;

    for (OrderItem item : order.getItems()) {
        total += item.getPrice() * item.getQuantity();
    }

    return applyDiscount(total);
}

private double applyDiscount(double total) {
    if (total > 1000) {
        return total * 0.9;
    }
    return total;
}

private void logOrderPaid(Order order, double totalPrice) {
    System.out.println("Order paid, total = " + totalPrice);
}
```

> 通俗理解：一个大函数拆成多个小函数，每个小函数只回答一个问题。

> 易错点：拆分函数不是越碎越好，关键是每个函数要有清晰语义。

## 重复代码

> 重复代码会导致修改时容易漏改，使系统行为不一致。

### 问题代码

```java
public double calculateVipPrice(double price) {
    return price * 0.8;
}

public double calculatePromotionPrice(double price) {
    return price * 0.8;
}

public double calculateFestivalPrice(double price) {
    return price * 0.8;
}
```

问题：

| 问题     | 说明                              |
| ------ | ------------------------------- |
| 折扣逻辑重复 | 多处都写了 `price * 0.8`             |
| 修改成本高  | 如果折扣从 `$0.8$` 变成 `$0.75$`，需要改多处 |
| 容易不一致  | 有些地方可能忘记修改                      |

### 修改后代码

```java
private static final double DISCOUNT_RATE = 0.8;

public double calculateVipPrice(double price) {
    return applyDiscount(price);
}

public double calculatePromotionPrice(double price) {
    return applyDiscount(price);
}

public double calculateFestivalPrice(double price) {
    return applyDiscount(price);
}

private double applyDiscount(double price) {
    return price * DISCOUNT_RATE;
}
```

> 通俗理解：同一知识只保留一份，其他地方复用它。

> 易错点：重复代码不仅指“完全相同的代码”，也包括“结构相似、逻辑相同”的代码。

## 魔法值

> 魔法值是直接写在代码中的数字或字符串，其含义不明显。

### 问题代码

```java
public boolean isPassed(int score) {
    return score >= 60;
}
```

问题：

| 问题         | 说明                  |
| ---------- | ------------------- |
| `60` 含义不明显 | 虽然能猜到是及格线，但代码没有明确表达 |
| 不便修改       | 如果及格线变化，需要查找所有 `60` |

### 修改后代码

```java
private static final int PASS_SCORE = 60;

public boolean isPassed(int score) {
    return score >= PASS_SCORE;
}
```

> 通俗理解：把“看不懂的数字”改成“有名字的常量”。

> 易错点：不是所有数字都必须提取，例如循环中的 `0`、`1` 有时含义很明显。

## 全局变量滥用

> 全局可变状态会让代码依赖关系隐蔽，导致调试困难。

### 问题代码

```java
public class Config {
    public static String currentUser;
}

public class OrderService {
    public void createOrder() {
        System.out.println("Create order for " + Config.currentUser);
    }
}
```

问题：

| 问题   | 说明                                               |
| ---- | ------------------------------------------------ |
| 隐式依赖 | `createOrder()` 表面没有参数，实际依赖 `Config.currentUser` |
| 难测试  | 测试前必须先修改全局状态                                     |
| 易出错  | 多线程环境下可能相互覆盖                                     |

### 修改后代码

```java
public class OrderService {
    public void createOrder(String currentUser) {
        System.out.println("Create order for " + currentUser);
    }
}
```

或者封装为对象：

```java
public class UserContext {
    private final String currentUser;

    public UserContext(String currentUser) {
        this.currentUser = currentUser;
    }

    public String getCurrentUser() {
        return currentUser;
    }
}

public class OrderService {
    public void createOrder(UserContext userContext) {
        System.out.println("Create order for " + userContext.getCurrentUser());
    }
}
```

> 通俗理解：不要让函数偷偷依赖外部状态，依赖什么就显式传进来。

> 易错点：全局常量通常可以接受，全局可变变量才是主要风险。

## 高耦合

> 高耦合是指一个模块过度依赖另一个具体模块，导致修改困难。

### 问题代码

```java
public class EmailSender {
    public void send(String message) {
        System.out.println("Send email: " + message);
    }
}

public class NotificationService {
    private EmailSender emailSender = new EmailSender();

    public void notifyUser(String message) {
        emailSender.send(message);
    }
}
```

问题：

| 问题      | 说明                                       |
| ------- | ---------------------------------------- |
| 直接依赖具体类 | `NotificationService` 固定依赖 `EmailSender` |
| 不易扩展    | 如果改成短信通知，需要修改原代码                         |
| 不易测试    | 很难替换为模拟对象                                |

### 修改后代码

```java
public interface MessageSender {
    void send(String message);
}

public class EmailSender implements MessageSender {
    @Override
    public void send(String message) {
        System.out.println("Send email: " + message);
    }
}

public class SmsSender implements MessageSender {
    @Override
    public void send(String message) {
        System.out.println("Send SMS: " + message);
    }
}

public class NotificationService {
    private final MessageSender messageSender;

    public NotificationService(MessageSender messageSender) {
        this.messageSender = messageSender;
    }

    public void notifyUser(String message) {
        messageSender.send(message);
    }
}
```

> 通俗理解：上层模块不要依赖具体实现，而应依赖抽象接口。

> 易错点：面向接口不是一定要给所有类都加接口，只有存在变化点或替换需求时才有明显价值。

## 低内聚

> 低内聚是指一个类承担了太多不相关职责。

### 问题代码

```java
public class UserManager {
    public void registerUser(String username, String password) {
        System.out.println("Register user");
    }

    public void saveToDatabase(String username, String password) {
        System.out.println("Save user to database");
    }

    public void sendWelcomeEmail(String username) {
        System.out.println("Send welcome email");
    }

    public void writeLog(String message) {
        System.out.println("Log: " + message);
    }
}
```

问题：

| 问题      | 说明       |
| ------- | -------- |
| 用户注册    | 业务职责     |
| 数据库存储   | 持久化职责    |
| 邮件发送    | 通知职责     |
| 日志记录    | 日志职责     |
| 一个类职责太多 | 违反单一职责原则 |

### 修改后代码

```java
public class UserService {
    private final UserRepository userRepository;
    private final EmailService emailService;
    private final Logger logger;

    public UserService(UserRepository userRepository, EmailService emailService, Logger logger) {
        this.userRepository = userRepository;
        this.emailService = emailService;
        this.logger = logger;
    }

    public void registerUser(String username, String password) {
        userRepository.save(username, password);
        emailService.sendWelcomeEmail(username);
        logger.info("User registered: " + username);
    }
}

public class UserRepository {
    public void save(String username, String password) {
        System.out.println("Save user to database");
    }
}

public class EmailService {
    public void sendWelcomeEmail(String username) {
        System.out.println("Send welcome email to " + username);
    }
}

public class Logger {
    public void info(String message) {
        System.out.println("Log: " + message);
    }
}
```

> 通俗理解：一个类只负责一类事情，不要把业务、数据库、邮件、日志全塞进一个类。

> 易错点：拆分类后需要注意协作关系，不要拆完之后调用链过度复杂。

## 复杂条件判断

> 复杂条件判断会降低可读性，也容易遗漏分支。

### 问题代码

```java
public double calculateSalary(Employee employee) {
    if (employee.getType().equals("FULL_TIME")) {
        return employee.getBaseSalary();
    } else if (employee.getType().equals("PART_TIME")) {
        return employee.getHours() * employee.getHourlyRate();
    } else if (employee.getType().equals("INTERN")) {
        return employee.getBaseSalary() * 0.5;
    } else {
        throw new IllegalArgumentException("Unknown employee type");
    }
}
```

问题：

| 问题           | 说明            |
| ------------ | ------------- |
| 多重 `if-else` | 条件越来越多时代码膨胀   |
| 字符串判断类型      | 容易写错          |
| 修改困难         | 新增员工类型需要修改原函数 |

### 修改后代码：使用多态

```java
public interface Employee {
    double calculateSalary();
}

public class FullTimeEmployee implements Employee {
    private final double baseSalary;

    public FullTimeEmployee(double baseSalary) {
        this.baseSalary = baseSalary;
    }

    @Override
    public double calculateSalary() {
        return baseSalary;
    }
}

public class PartTimeEmployee implements Employee {
    private final int hours;
    private final double hourlyRate;

    public PartTimeEmployee(int hours, double hourlyRate) {
        this.hours = hours;
        this.hourlyRate = hourlyRate;
    }

    @Override
    public double calculateSalary() {
        return hours * hourlyRate;
    }
}

public class InternEmployee implements Employee {
    private final double baseSalary;

    public InternEmployee(double baseSalary) {
        this.baseSalary = baseSalary;
    }

    @Override
    public double calculateSalary() {
        return baseSalary * 0.5;
    }
}
```

调用时：

```java
public double calculateSalary(Employee employee) {
    return employee.calculateSalary();
}
```

> 通俗理解：不要总是问“你是什么类型”，而是让对象自己完成对应行为。

> 易错点：不是所有 `if-else` 都需要改成多态；当分支稳定且很少时，普通条件判断也可以接受。

## 嵌套过深

> 嵌套过深会让代码难以阅读，容易看不清主流程。

### 问题代码

```java
public void process(User user) {
    if (user != null) {
        if (user.isActive()) {
            if (user.hasPermission()) {
                System.out.println("Process user");
            } else {
                System.out.println("No permission");
            }
        } else {
            System.out.println("User is inactive");
        }
    } else {
        System.out.println("User is null");
    }
}
```

问题：

| 问题   | 说明          |
| ---- | ----------- |
| 多层嵌套 | 主流程被埋在最里面   |
| 可读性差 | 需要不断匹配括号    |
| 维护困难 | 新增条件会继续加深嵌套 |

### 修改后代码：提前返回

```java
public void process(User user) {
    if (user == null) {
        System.out.println("User is null");
        return;
    }

    if (!user.isActive()) {
        System.out.println("User is inactive");
        return;
    }

    if (!user.hasPermission()) {
        System.out.println("No permission");
        return;
    }

    System.out.println("Process user");
}
```

> 通俗理解：先处理异常情况，让正常流程留在最后清晰呈现。

> 易错点：提前返回适合减少嵌套，但不要让函数中出现过多分散的返回点导致流程混乱。

## 异常处理不当

> 异常处理应当让错误可发现、可定位、可恢复，而不是隐藏问题。

### 问题代码

```java
public User findUser(String id) {
    try {
        return database.queryUser(id);
    } catch (Exception e) {
        return null;
    }
}
```

问题：

| 问题        | 说明                   |
| --------- | -------------------- |
| 吞掉异常      | 不知道真正错误原因            |
| 返回 `null` | 调用方容易空指针             |
| 捕获过宽      | `Exception` 包含太多错误类型 |

### 修改后代码

```java
public User findUser(String id) {
    try {
        return database.queryUser(id);
    } catch (DatabaseException e) {
        throw new UserQueryException("Failed to query user, id = " + id, e);
    }
}
```

或者使用 `Optional` 表达“可能不存在”：

```java
public Optional<User> findUser(String id) {
    try {
        User user = database.queryUser(id);
        return Optional.ofNullable(user);
    } catch (DatabaseException e) {
        throw new UserQueryException("Failed to query user, id = " + id, e);
    }
}
```

> 通俗理解：异常不能悄悄吃掉，要么处理，要么带着上下文继续抛出。

> 易错点：“查不到用户”和“数据库出错”不是同一种情况，不能都简单返回 `null`。

## 注释质量差

> 好注释解释“为什么这样做”，坏注释重复“代码做了什么”。

### 问题代码

```java
// i 加 1
i++;

// 如果年龄大于等于 18
if (age >= 18) {
    // 设置成年
    user.setAdult(true);
}
```

问题：

| 问题     | 说明           |
| ------ | ------------ |
| 注释重复代码 | 没有提供额外信息     |
| 维护成本高  | 代码变了注释可能忘记更新 |
| 掩盖命名问题 | 本应用更好的命名表达含义 |

### 修改后代码

```java
if (age >= ADULT_AGE) {
    user.markAsAdult();
}
```

必要注释示例：

```java
// 第三方支付接口要求金额单位为分，因此这里需要从元转换为分。
int amountInCents = yuan.multiply(new BigDecimal("100")).intValue();
```

> 通俗理解：能用命名说清楚的，不要用注释补救；注释应解释背景、原因和约束。

> 易错点：不是“注释越多越好”，而是“关键地方注释要有价值”。

## 数据暴露

> 对象内部数据不应被外部随意修改，否则会破坏封装性。

### 问题代码

```java
public class Order {
    public List<OrderItem> items = new ArrayList<>();
}
```

问题：

| 问题   | 说明        |
| ---- | --------- |
| 字段公开 | 外部可以随意修改  |
| 缺少约束 | 可以添加非法订单项 |
| 破坏封装 | 类无法保护自身状态 |

### 修改后代码

```java
public class Order {
    private final List<OrderItem> items = new ArrayList<>();

    public void addItem(OrderItem item) {
        if (item == null) {
            throw new IllegalArgumentException("item is null");
        }
        items.add(item);
    }

    public List<OrderItem> getItems() {
        return Collections.unmodifiableList(items);
    }
}
```

> 通俗理解：对象的数据要由对象自己管理，不能直接暴露给外部乱改。

> 易错点：`private` 字段加 `getter` 不一定就安全，如果直接返回可变集合，外部仍然能修改。

## 参数过多

> 参数过多会让函数难以调用，也容易传错顺序。

### 问题代码

```java
public void createUser(String username, String password, String email,
                       String phone, String address, int age) {
    System.out.println("Create user");
}
```

问题：

| 问题     | 说明                                         |
| ------ | ------------------------------------------ |
| 参数列表过长 | 调用时不清晰                                     |
| 同类型参数多 | `email`、`phone`、`address` 都是 `String`，容易传错 |
| 扩展困难   | 新增字段需要修改方法签名                               |

### 修改后代码

```java
public class CreateUserRequest {
    private String username;
    private String password;
    private String email;
    private String phone;
    private String address;
    private int age;

    // getter 和 setter 省略
}

public void createUser(CreateUserRequest request) {
    System.out.println("Create user: " + request.getUsername());
}
```

> 通俗理解：参数太多时，把它们封装成一个对象。

> 易错点：不要为了减少参数而随便传入大对象，参数对象本身也要有明确含义。

## 面向过程式对象设计

> 面向对象代码中，如果对象只有数据，没有行为，就容易变成过程式代码。

### 问题代码

```java
public class Account {
    public double balance;
}

public class AccountService {
    public void withdraw(Account account, double amount) {
        if (account.balance >= amount) {
            account.balance -= amount;
        }
    }
}
```

问题：

| 问题             | 说明             |
| -------------- | -------------- |
| `Account` 只有数据 | 行为放在外部服务中      |
| 数据被直接修改        | 缺少封装           |
| 业务规则分散         | 余额检查逻辑可能被复制到多处 |

### 修改后代码

```java
public class Account {
    private double balance;

    public Account(double initialBalance) {
        if (initialBalance < 0) {
            throw new IllegalArgumentException("balance cannot be negative");
        }
        this.balance = initialBalance;
    }

    public void withdraw(double amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("amount must be positive");
        }

        if (balance < amount) {
            throw new IllegalStateException("insufficient balance");
        }

        balance -= amount;
    }

    public double getBalance() {
        return balance;
    }
}
```

> 通俗理解：对象不应该只是“数据袋子”，还应该把相关业务行为放进自己内部。

> 易错点：不是所有数据类都错误，例如 DTO、VO 可以主要用于传输或展示数据。

## 代码质量修改方法总结

> 修改代码质量问题的核心思路是：提高可读性、降低耦合、增强内聚、减少重复、明确依赖。

| 问题     | 修改方法                   |
| ------ | ---------------------- |
| 命名不清晰  | 使用有业务含义的变量名、函数名、类名     |
| 函数过长   | 提取函数，每个函数只做一件事         |
| 重复代码   | 提取公共函数、公共类或策略          |
| 魔法值    | 提取常量或配置项               |
| 全局变量滥用 | 使用参数传递、依赖注入、上下文对象      |
| 高耦合    | 面向接口编程，依赖抽象            |
| 低内聚    | 按职责拆分类和模块              |
| 条件复杂   | 提前返回、提取判断函数、使用多态或策略模式  |
| 嵌套过深   | 处理异常分支后提前返回            |
| 异常处理不当 | 不吞异常，补充上下文，区分业务失败和系统异常 |
| 数据暴露   | 使用 `private` 字段，控制修改入口 |
| 参数过多   | 引入参数对象                 |
| 注释质量差  | 优化命名，注释解释原因而不是重复代码     |

> 考试记忆：
> **代码质量差的常见原因：命名差、函数长、重复多、耦合高、内聚低、异常乱、数据乱暴露。**
> **修改方向：清晰命名、提取函数、消除重复、封装数据、面向接口、职责分离。**
