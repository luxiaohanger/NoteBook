## 表驱动

表驱动是一种用“数据表”代替大量 `if-else` 或 `switch-case` 判断的设计方法。它把程序中的变化规则、映射关系、处理策略从控制逻辑中抽出来，放到表结构中统一管理。

通俗地说：**不要把所有规则都写死在代码分支里，而是把规则放进表里，让程序查表执行。**

表驱动常用于：状态转换、菜单分发、权限判断、价格计算、命令处理、错误码映射、策略选择等场景。

---

## 不使用表驱动的问题

> 如果业务规则很多，直接使用大量 `if-else`，代码会变得冗长、重复、难维护。

例如，根据用户等级计算折扣：

```java
public double getDiscount(String level) {
    if (level.equals("NORMAL")) {
        return 1.0;
    } else if (level.equals("VIP")) {
        return 0.9;
    } else if (level.equals("SVIP")) {
        return 0.8;
    } else if (level.equals("STUDENT")) {
        return 0.85;
    } else {
        return 1.0;
    }
}
```

这个写法的问题是：

| 问题      | 说明                |
| ------- | ----------------- |
| 分支过多    | 等级越多，`if-else` 越长 |
| 修改困难    | 新增等级必须修改方法内部逻辑    |
| 可读性差    | 规则和流程控制混在一起       |
| 容易出错    | 多个分支之间可能遗漏或重复     |
| 不符合开闭原则 | 扩展规则时需要修改已有代码     |

> 考试中常见表述：大量条件分支会降低代码的可维护性和可扩展性。

## 使用映射表进行表驱动

> 最简单的表驱动方式是使用 `Map` 保存“输入值”和“结果值”的对应关系。

修改为表驱动：

```java
import java.util.HashMap;
import java.util.Map;

public class DiscountService {

    private static final Map<String, Double> DISCOUNT_TABLE = new HashMap<>();

    static {
        DISCOUNT_TABLE.put("NORMAL", 1.0);
        DISCOUNT_TABLE.put("VIP", 0.9);
        DISCOUNT_TABLE.put("SVIP", 0.8);
        DISCOUNT_TABLE.put("STUDENT", 0.85);
    }

    public double getDiscount(String level) {
        return DISCOUNT_TABLE.getOrDefault(level, 1.0);
    }
}
```

此时程序逻辑变成：

```markdown
输入用户等级
    ↓
查找折扣表
    ↓
返回对应折扣
```

> 用一句话概括：**表驱动把“判断分支”改成了“查表”。**

> 易错点：表驱动不是简单地把 `if-else` 换成 `Map`，核心是把变化规则从流程逻辑中分离出来。

---

## 表驱动与代码质量

> 表驱动可以改善代码质量，主要体现在降低复杂度、减少重复、提高可维护性和可扩展性。

| 代码质量问题       | 表驱动的改进         |
| ------------ | -------------- |
| 大量 `if-else` | 用表查询代替复杂分支     |
| 规则分散         | 把规则集中到表中       |
| 修改影响范围大      | 修改表即可，核心逻辑基本不变 |
| 可读性差         | 数据规则和执行逻辑分离    |
| 难以测试         | 可以单独测试表项和默认规则  |

例如原来的核心逻辑是：

```java
if (type.equals("A")) {
    handleA();
} else if (type.equals("B")) {
    handleB();
} else if (type.equals("C")) {
    handleC();
}
```

使用表驱动后，核心逻辑变成：

```java
handlerTable.get(type).handle();
```

> 这体现了软件工程中的一个重要思想：**用数据结构表达变化，用统一流程处理变化。**

## 表驱动处理命令分发

> 当不同输入对应不同处理函数时，可以把“命令”和“处理器”放入表中。

不使用表驱动：

```java
public void execute(String command) {
    if (command.equals("start")) {
        start();
    } else if (command.equals("stop")) {
        stop();
    } else if (command.equals("restart")) {
        restart();
    } else {
        throw new IllegalArgumentException("Unknown command");
    }
}

private void start() {
    System.out.println("System started");
}

private void stop() {
    System.out.println("System stopped");
}

private void restart() {
    System.out.println("System restarted");
}
```

使用表驱动：

```java
import java.util.HashMap;
import java.util.Map;

public class CommandExecutor {

    private final Map<String, Runnable> commandTable = new HashMap<>();

    public CommandExecutor() {
        commandTable.put("start", this::start);
        commandTable.put("stop", this::stop);
        commandTable.put("restart", this::restart);
    }

    public void execute(String command) {
        Runnable handler = commandTable.get(command);

        if (handler == null) {
            throw new IllegalArgumentException("Unknown command: " + command);
        }

        handler.run();
    }

    private void start() {
        System.out.println("System started");
    }

    private void stop() {
        System.out.println("System stopped");
    }

    private void restart() {
        System.out.println("System restarted");
    }
}
```

此时表的含义是：

| 命令        | 处理函数        |
| --------- | ----------- |
| `start`   | `start()`   |
| `stop`    | `stop()`    |
| `restart` | `restart()` |

> 通俗解释：用户输入什么命令，程序就去表里找对应的函数执行。

> 考试注意：这种写法可以减少分支判断，也更符合“对扩展开放，对修改关闭”的思想。

## 表驱动处理状态转换

> 表驱动非常适合描述状态机，因为状态机本质上就是“当前状态 + 输入事件 → 下一个状态”。

例如订单状态转换：

| 当前状态      | 事件        | 下一个状态       |
| --------- | --------- | ----------- |
| `CREATED` | `PAY`     | `PAID`      |
| `PAID`    | `SHIP`    | `SHIPPED`   |
| `SHIPPED` | `RECEIVE` | `FINISHED`  |
| `CREATED` | `CANCEL`  | `CANCELLED` |

可以用代码表示为：

```java
import java.util.HashMap;
import java.util.Map;

public class OrderStateMachine {

    enum State {
        CREATED, PAID, SHIPPED, FINISHED, CANCELLED
    }

    enum Event {
        PAY, SHIP, RECEIVE, CANCEL
    }

    static class Key {
        private final State state;
        private final Event event;

        public Key(State state, Event event) {
            this.state = state;
            this.event = event;
        }

        @Override
        public boolean equals(Object obj) {
            if (!(obj instanceof Key)) {
                return false;
            }

            Key other = (Key) obj;
            return this.state == other.state && this.event == other.event;
        }

        @Override
        public int hashCode() {
            return state.hashCode() * 31 + event.hashCode();
        }
    }

    private final Map<Key, State> transitionTable = new HashMap<>();

    public OrderStateMachine() {
        transitionTable.put(new Key(State.CREATED, Event.PAY), State.PAID);
        transitionTable.put(new Key(State.PAID, Event.SHIP), State.SHIPPED);
        transitionTable.put(new Key(State.SHIPPED, Event.RECEIVE), State.FINISHED);
        transitionTable.put(new Key(State.CREATED, Event.CANCEL), State.CANCELLED);
    }

    public State next(State currentState, Event event) {
        State nextState = transitionTable.get(new Key(currentState, event));

        if (nextState == null) {
            throw new IllegalStateException(
                "Invalid transition: " + currentState + " -> " + event
            );
        }

        return nextState;
    }
}
```

使用示例：

```java
public class Main {
    public static void main(String[] args) {
        OrderStateMachine machine = new OrderStateMachine();

        OrderStateMachine.State state = OrderStateMachine.State.CREATED;

        state = machine.next(state, OrderStateMachine.Event.PAY);
        System.out.println(state); // PAID

        state = machine.next(state, OrderStateMachine.Event.SHIP);
        System.out.println(state); // SHIPPED
    }
}
```

> 用一句话概括：**状态机中的状态转换表就是典型的表驱动。**

> 易错点：状态转换不应该散落在多个 `if-else` 中，否则状态越多，逻辑越难维护。

## 表驱动处理区间规则

> 有些规则不是简单的一一对应，而是根据范围判断，例如成绩等级、价格区间、年龄段分类等。

不使用表驱动：

```java
public String getGrade(int score) {
    if (score >= 90) {
        return "A";
    } else if (score >= 80) {
        return "B";
    } else if (score >= 70) {
        return "C";
    } else if (score >= 60) {
        return "D";
    } else {
        return "F";
    }
}
```

使用表驱动：

```java
import java.util.Arrays;
import java.util.List;

public class GradeService {

    static class GradeRule {
        int minScore;
        String grade;

        GradeRule(int minScore, String grade) {
            this.minScore = minScore;
            this.grade = grade;
        }
    }

    private final List<GradeRule> gradeTable = Arrays.asList(
        new GradeRule(90, "A"),
        new GradeRule(80, "B"),
        new GradeRule(70, "C"),
        new GradeRule(60, "D"),
        new GradeRule(0, "F")
    );

    public String getGrade(int score) {
        for (GradeRule rule : gradeTable) {
            if (score >= rule.minScore) {
                return rule.grade;
            }
        }

        return "F";
    }
}
```

表的含义是：

| 最低分数 | 等级 |
| ---- | -- |
| 90   | A  |
| 80   | B  |
| 70   | C  |
| 60   | D  |
| 0    | F  |

> 通俗解释：程序不是写死每个判断，而是按照表中的规则逐项匹配。

> 易错点：区间表通常对顺序敏感，例如这里必须从高分到低分匹配。

## 表驱动与策略模式

> 当表中保存的不只是普通数据，而是不同算法或处理对象时，表驱动常常会和策略模式结合。

例如不同支付方式对应不同支付策略：

```java
interface PayStrategy {
    void pay(double amount);
}

class AliPayStrategy implements PayStrategy {
    public void pay(double amount) {
        System.out.println("Use AliPay: " + amount);
    }
}

class WeChatPayStrategy implements PayStrategy {
    public void pay(double amount) {
        System.out.println("Use WeChat Pay: " + amount);
    }
}

class BankCardPayStrategy implements PayStrategy {
    public void pay(double amount) {
        System.out.println("Use Bank Card: " + amount);
    }
}
```

使用表驱动保存策略：

```java
import java.util.HashMap;
import java.util.Map;

public class PayService {

    private final Map<String, PayStrategy> strategyTable = new HashMap<>();

    public PayService() {
        strategyTable.put("ALI_PAY", new AliPayStrategy());
        strategyTable.put("WECHAT_PAY", new WeChatPayStrategy());
        strategyTable.put("BANK_CARD", new BankCardPayStrategy());
    }

    public void pay(String payType, double amount) {
        PayStrategy strategy = strategyTable.get(payType);

        if (strategy == null) {
            throw new IllegalArgumentException("Unsupported pay type: " + payType);
        }

        strategy.pay(amount);
    }
}
```

此时对应关系是：

| 支付类型         | 支付策略                  |
| ------------ | --------------------- |
| `ALI_PAY`    | `AliPayStrategy`      |
| `WECHAT_PAY` | `WeChatPayStrategy`   |
| `BANK_CARD`  | `BankCardPayStrategy` |

> 表驱动负责“选择策略”，策略模式负责“封装算法”。

> 考试注意：表驱动不是一种典型 GoF 设计模式，但经常和策略模式、命令模式、状态模式一起使用。

## 表驱动的优点

> 表驱动的主要优点是让程序结构更清晰，使规则集中、扩展方便。

| 优点     | 说明                           |
| ------ | ---------------------------- |
| 降低圈复杂度 | 减少 `if-else` 和 `switch-case` |
| 提高可读性  | 规则以表的形式集中表达                  |
| 提高可维护性 | 修改规则时主要修改表                   |
| 提高可扩展性 | 新增规则可以增加表项                   |
| 便于配置化  | 表可以进一步放到配置文件或数据库中            |
| 便于测试   | 每条规则可以独立测试                   |

> 通俗解释：表驱动让代码从“写流程判断”变成“维护规则数据”。

---

## 表驱动的缺点

> 表驱动虽然能提升可维护性，但不是所有场景都适合使用。

| 缺点         | 说明                  |
| ---------- | ------------------- |
| 简单场景可能过度设计 | 只有两三个分支时，表驱动反而增加复杂度 |
| 调试不够直观     | 执行逻辑通过表间接找到，不如分支直接  |
| 表项错误不易发现   | 配置表写错可能导致运行时错误      |
| 类型安全可能下降   | 使用字符串作为键时容易拼写错误     |
| 初始化成本增加    | 需要额外维护表结构           |
| 顺序规则易出错    | 区间匹配、优先级匹配依赖表项顺序    |

> 易错点：表驱动不是为了完全消灭 `if-else`，而是为了管理复杂、稳定结构下的多规则变化。

---

## 表驱动适用场景

> 当程序中存在大量稳定形式的映射关系、规则关系或分发关系时，适合使用表驱动。

常见适用场景：

| 场景    | 示例                |
| ----- | ----------------- |
| 值映射   | 用户等级 → 折扣         |
| 命令分发  | 命令名称 → 处理函数       |
| 状态转换  | 当前状态 + 事件 → 下一个状态 |
| 错误码处理 | 错误码 → 错误信息        |
| 类型选择  | 文件类型 → 解析器        |
| 规则匹配  | 分数区间 → 等级         |
| 策略选择  | 支付类型 → 支付策略       |
| 菜单系统  | 菜单编号 → 操作函数       |

> 判断口诀：**如果代码里出现大量形式相似的分支判断，就可以考虑表驱动。**

---

## 表驱动不适用场景

> 如果分支很少、逻辑差异很大，或者规则高度动态且难以抽象成表，表驱动未必合适。

不适合的情况：

| 场景             | 原因            |
| -------------- | ------------- |
| 分支数量很少         | 直接 `if` 更简单   |
| 每个分支逻辑差异巨大     | 很难抽象出统一表结构    |
| 规则临时且不稳定       | 表结构频繁变化，维护成本高 |
| 强依赖复杂上下文       | 表项难以表达完整逻辑    |
| 对性能极度敏感且表查询成本高 | 需要评估额外间接访问开销  |

> 考试注意：表驱动是一种改善设计的方法，不是任何时候都必须使用。

---

## 表驱动与开闭原则

> 开闭原则要求软件实体应该对扩展开放、对修改关闭。表驱动可以在一定程度上支持开闭原则。

例如新增一种折扣等级：

```java
DISCOUNT_TABLE.put("TEACHER", 0.88);
```

只需要新增表项，而不需要修改原来的判断流程。

对比原来的 `if-else`：

```java
else if (level.equals("TEACHER")) {
    return 0.88;
}
```

原写法需要修改方法内部逻辑，而表驱动只需要增加数据项。

> 通俗解释：新增规则时，最好只是“加一行表项”，而不是“改一堆流程代码”。

> 易错点：如果新增规则还需要修改核心处理流程，说明表驱动抽象不够充分。

---

## 表驱动与配置化

> 表驱动进一步发展，可以把表从代码中移动到配置文件、数据库或规则平台中。

例如使用 JSON 配置折扣表：

```json
{
  "NORMAL": 1.0,
  "VIP": 0.9,
  "SVIP": 0.8,
  "STUDENT": 0.85
}
```

程序读取配置后建立映射：

```java
Map<String, Double> discountTable = loadDiscountConfig();

public double getDiscount(String level) {
    return discountTable.getOrDefault(level, 1.0);
}
```

这样修改折扣规则时，可能不需要重新编译代码。

> 通俗解释：配置化表驱动就是把规则从代码里搬到配置文件里。

> 易错点：配置化提高灵活性，但也需要校验配置的正确性，否则错误会推迟到运行时暴露。

---

## 表驱动总结

> 表驱动是一种用表结构表达规则、映射和分发关系的设计方法，目的是减少复杂条件分支，提高代码的可维护性、可扩展性和清晰度。

核心思想：

```markdown
变化的规则 → 放入表
稳定的流程 → 写在代码中
程序执行 → 根据输入查表并执行
```

对比总结：

| 方面   | 条件分支写法                    | 表驱动写法            |
| ---- | ------------------------- | ---------------- |
| 核心形式 | `if-else` / `switch-case` | `Map` / 数组 / 配置表 |
| 规则位置 | 分散在代码分支中                  | 集中在表中            |
| 扩展方式 | 修改代码逻辑                    | 增加或修改表项          |
| 复杂度  | 分支越多越复杂                   | 流程相对稳定           |
| 适用情况 | 少量简单判断                    | 大量规则、映射、状态转换     |

> 一句话记忆：**表驱动就是用“查表”代替“写死判断”，把变化规则数据化。**

> 考试注意：表驱动的本质不是某一种具体语法，而是一种设计思想：**将规则与控制逻辑分离**。
