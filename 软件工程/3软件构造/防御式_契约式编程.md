## 防御式编程

防御式编程是一种假设“外部输入、调用者、运行环境都可能出错”的编程思想。它要求程序在执行前主动检查异常情况，避免错误数据继续传播。

> 防御式编程的核心是：**不完全相信外部输入，先检查，再执行。**

典型做法包括：

| 做法    | 说明                         |
| ----- | -------------------------- |
| 参数校验  | 检查参数是否为 `null`、是否越界、格式是否合法 |
| 边界检查  | 检查数组下标、集合大小、分页参数等          |
| 异常处理  | 对可能失败的操作进行 `try-catch`     |
| 默认值处理 | 输入缺失时给出安全默认值               |
| 失败保护  | 出错时返回明确错误，而不是让系统崩溃         |
| 日志记录  | 记录异常信息，便于定位问题              |

例如，计算商品折扣价：

```java
public double calculatePrice(double price, double discount) {
    return price * discount;
}
```

这个写法的问题是：如果 `price` 为负数，或者 `discount` 大于 $1$，就会得到不合理结果。

防御式写法：

```java
public double calculatePrice(double price, double discount) {
    if (price < 0) {
        throw new IllegalArgumentException("price cannot be negative");
    }

    if (discount < 0 || discount > 1) {
        throw new IllegalArgumentException("discount must be between 0 and 1");
    }

    return price * discount;
}
```

> 防御式编程适合处理来自用户、网络、文件、数据库、第三方接口等不可信来源的数据。

> 易错点：防御式编程不是到处写 `try-catch`，而是有针对性地检查不可信输入和可能失败的操作。

## 契约式编程

契约式编程是一种把软件模块之间的调用关系看成“契约”的编程思想。它明确规定调用者和被调用者各自的责任。

契约通常包括三类条件：

| 契约内容 | 含义                | 责任方    |
| ---- | ----------------- | ------ |
| 前置条件 | 方法调用前必须满足的条件      | 调用者负责  |
| 后置条件 | 方法执行后必须保证的结果      | 被调用者负责 |
| 不变式  | 对象生命周期中始终应保持成立的条件 | 类或对象负责 |

> 契约式编程的核心是：**调用前说清要求，执行后保证结果，过程中维持状态正确。**

例如，银行账户取款：

```java
public class Account {
    private double balance;

    public void withdraw(double amount) {
        balance -= amount;
    }
}
```

这个方法没有明确说明 `amount` 应该满足什么条件，也没有保证执行后账户状态是否合法。

契约式写法：

```java
public class Account {
    private double balance;

    public Account(double balance) {
        if (balance < 0) {
            throw new IllegalArgumentException("initial balance cannot be negative");
        }

        this.balance = balance;
    }

    public void withdraw(double amount) {
        // 前置条件：取款金额必须大于 0
        if (amount <= 0) {
            throw new IllegalArgumentException("amount must be positive");
        }

        // 前置条件：余额必须足够
        if (amount > balance) {
            throw new IllegalArgumentException("insufficient balance");
        }

        double oldBalance = balance;

        balance -= amount;

        // 后置条件：新余额应等于旧余额减取款金额
        assert balance == oldBalance - amount;

        // 不变式：余额不能为负
        assert balance >= 0;
    }

    public double getBalance() {
        return balance;
    }
}
```

其中：

| 条件   | 示例                               |
| ---- | -------------------------------- |
| 前置条件 | `amount > 0`，`amount <= balance` |
| 后置条件 | `balance == oldBalance - amount` |
| 不变式  | `balance >= 0`                   |

> 契约式编程强调“责任划分”：如果前置条件不满足，是调用者的问题；如果前置条件满足但后置条件不满足，是被调用者的问题。

> 易错点：契约式编程不是单纯写注释，而是要明确模块之间的输入条件、输出保证和状态约束。

## 防御式编程与契约式编程的区别

防御式编程和契约式编程都关注程序正确性，但侧重点不同。

| 对比项  | 防御式编程                | 契约式编程                |
| ---- | -------------------- | -------------------- |
| 基本假设 | 外部输入可能不可信            | 调用者和被调用者有明确责任        |
| 核心思想 | 尽量防止错误输入造成系统故障       | 明确前置条件、后置条件和不变式      |
| 关注重点 | 安全性、健壮性、异常处理         | 接口语义、责任划分、正确性        |
| 常见手段 | 参数校验、异常处理、默认值、日志     | 前置条件、后置条件、不变式、断言     |
| 适合场景 | 用户输入、网络请求、文件读取、第三方接口 | 模块接口、类设计、算法实现、核心业务逻辑 |
| 错误处理 | 尽量识别并处理异常情况          | 违反契约时通常快速失败          |
| 代码风格 | 可能包含较多检查代码           | 强调规范化的接口条件           |

> 防御式编程更像“我不信任外部，所以我要检查”；契约式编程更像“我们先约定规则，谁违反谁负责”。
