## 需求的三个层次

| 层次 | 回答的问题 | 例子 |
|---|---|---|
| 业务需求 | 组织为什么要做系统 | 学校希望减少校园卡人工窗口排队时间 |
| 用户需求 | 用户希望系统帮助自己完成什么目标 | 学生希望在线充值、查询余额、挂失 |
| 系统需求 | 系统必须提供的具体功能、数据或约束 | 充值金额必须为正数；充值成功后 3 秒内更新余额 |

判断方法：

- 如果说的是组织收益和业务目标，是业务需求。
- 如果说的是某类用户想完成的任务，是用户需求。
- 如果说的是系统具体行为、数据、性能、接口或约束，是系统需求。

--------

## 需求类型

| 类型 | 说明 | 例子 |
|---|---|---|
| 功能需求 | 系统必须做什么 | 输入错误密码时提示验证失败 |
| 数据需求 | 数据格式、范围、结构和约束 | 金额必须是非负数字 |
| 质量需求 | 可用性、安全性、可靠性、可维护性等 | 新用户 10 分钟内能完成充值 |
| 性能需求 | 响应时间、吞吐量、并发量等 | 95% 查询请求在 2 秒内返回 |
| 约束 | 技术、平台、法规、过程限制 | 必须使用学校统一认证 |
| 对外接口 | 与外部系统、设备或用户界面的交互 | 通过 REST API 调用支付平台 |

高频易错点：

> “输入补货数含非数字字符时系统提示输入错误”是功能需求，因为它描述系统行为。  
> “补货数不能含非数字字符”是数据需求，因为它约束数据格式。
