## 概念类图

> 概念类图用于描述问题域中的重要概念、概念之间的关系，以及概念本身的主要属性。它属于**分析阶段**常用的模型，重点是理解业务领域，而不是描述程序代码实现。

-----
### 概念类图和设计类图的区别

> 概念类图面向业务分析，设计类图面向软件实现。

| 对比项  | 概念类图        | 设计类图                                         |
| ---- | ----------- | -------------------------------------------- |
| 阶段   | 需求分析 / 领域建模 | 设计阶段                                         |
| 关注点  | 业务概念        | 程序结构                                         |
| 类名来源 | 现实业务名词      | 软件类、接口、模块                                    |
| 属性   | 业务属性        | 代码属性、类型、可见性                                  |
| 方法   | 一般不写        | 通常要写                                         |
| 关系   | 业务关系        | 代码依赖、继承、实现、组合等                               |
| 例子   | 学生、课程、订单、商品 | `StudentService`、`OrderController`、`UserDAO` |

> 一句话记忆：**概念类图画“业务中有什么”，设计类图画“程序中怎么实现”。**


> 不需要画接口类，只关心实体类，关系只有关联、依赖、继承、聚合、组合，而没有实现

### 示例：学生选课系统概念类图

```text
┌──────────┐        1        0..*       ┌──────────┐
│  学生     │──────────────────────────│  选课记录  │
├──────────┤                          ├──────────┤
│ 学号      │                          │ 选课时间   │
│ 姓名      │                          │ 成绩      │
└──────────┘                          └──────────┘
                                                │
                                                │ 0..*
                                                │
                                                │ 1
                                          ┌──────────┐
                                          │  课程     │
                                          ├──────────┤
                                          │ 课程号    │
                                          │ 课程名    │
                                          │ 学分      │
                                          └──────────┘
                                                │ 0..*
                                                │
                                                │ 1
                                          ┌──────────┐
                                          │  教师     │
                                          ├──────────┤
                                          │ 教师号    │
                                          │ 姓名      │
                                          │ 职称      │
                                          └──────────┘
```

解释：

```text
一个学生可以有多条选课记录；
一条选课记录属于一个学生；
一门课程可以被多次选课；
一条选课记录对应一门课程；
一个教师可以教授多门课程；
一门课程由一个教师讲授。
```

> 易错点：学生和课程是多对多关系，通常不要直接只画 `学生 ─ 课程`，更推荐引入“选课记录”作为关联类，因为选课关系本身有“成绩、选课时间”等属性。

> 类的属性一般写简单值，例如编号、姓名、余额、状态；另一个业务对象通常用关联线表示，不写成属性。

## 技巧

如何把需求动作拆分成类？
- 取决于动作是一个过程，还是业务系统中需要长期保存和管理的一条事实？
- 如果动作是瞬时的，没有保存和记录，则无需生成实体类；需求功能必须被模型支持，但不一定要在概念类图中逐字出现。
- 反之可以用xx记录作为类。
