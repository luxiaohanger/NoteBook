## 进程调度

| 算法             |     是否抢占 | 核心思想           | 优点                    | 主要问题                | 适用场景            |
| -------------- | -------: | -------------- | --------------------- | ------------------- | --------------- |
| FCFS / FIFO    |        否 | 谁先到，谁先运行       | 简单、公平按到达顺序            | 护航效应，短任务可能等很久       | 批处理、简单队列        |
| SJF            |        否 | 运行时间最短的任务先运行   | 平均等待时间最优              | 需要预知运行时间，可能饿死长任务    | 批处理任务           |
| SRTF           |        是 | 剩余时间最短的任务先运行   | 比 SJF 更灵活，适合新短任务到达    | 需要预测剩余时间，长任务可能饿死    | 抢占式批处理          |
| RR             |        是 | 每个进程轮流获得一个时间片  | 响应较好，公平性较好            | 时间片太小开销大，太大退化为 FCFS | 交互式系统           |
| Priority       | 可抢占/不可抢占 | 优先级高的先运行       | 能区分重要任务               | 低优先级可能饿死            | 有优先级需求的系统       |
| MFQ / MLFQ     |        是 | 多个优先级队列，进程可升降级 | 兼顾响应、吞吐、公平            | 规则复杂，仍可能被“钻空子”      | 通用操作系统          |
| Lottery        |        是 | 按彩票数随机抽取运行任务   | 按比例分配 CPU，避免绝对饥饿      | 有随机波动               | 需要比例公平的场景       |
| Rate Monotonic |        是 | 周期越短，静态优先级越高   | 实时系统中静态优先级最优          | 利用率高时可能失败           | 固定周期实时任务        |
| EDF            |        是 | 截止日期越早，优先级越高   | 单核抢占实时调度中理论最优         | 实现和分析更复杂            | 实时系统            |
| CFS            |        是 | 选择虚拟运行时间最少的任务  | 追求公平，Linux 普通任务调度核心思想 | 实现复杂                | 现代 Linux 普通任务调度 |

## 机制与策略

调度中要区分机制和策略。机制回答“怎么切换”，策略回答“选谁运行”。已有的上下文切换机制和就绪队列机制，可以支持不同的调度策略，例如随机选择、轮转选择、按优先级选择等。

| 部分 | 含义                       | 例子                           |
| -- | ------------------------ | ---------------------------- |
| 机制 | 操作系统如何保存当前进程上下文、恢复新进程上下文 | context switch、PCB/TCB、就绪队列  |
| 策略 | 每次调度时选择哪个进程运行            | FCFS、SJF、RR、Priority、MFQ、CFS |

调度器的抽象逻辑可以理解为：

```c
on_interrupt() {
    save(current);
    current = select_one(ready_queue);
    restore(current);
}
```

其中 `save()` 和 `restore()` 是机制，`select_one()` 是策略。

> 调度机制像“换人上场的动作”，调度策略像“教练决定谁上场”。

> 某些策略需要额外机制支持，例如 SJF 需要知道或预测任务运行时间，RR 需要时钟中断和时间片机制。

## 抢占式与非抢占式的重排序时机

| 类型     | 是否允许打断当前运行进程 | 什么时候重新排序 / 重新选择                                                        | 典型算法                              |
| ------ | -----------: | ---------------------------------------------------------------------- | --------------------------------- |
| 非抢占式调度 |      不允许主动打断 | 当前进程**运行结束**、**主动阻塞**、**主动让出 CPU** 时                                   | FCFS、非抢占式 SJF、非抢占式 Priority       |
| 抢占式调度  |   允许打断当前运行进程 | 除了非抢占式的时机外，还会在**时间片用完**、**新进程到达**、**优先级变化**、**I/O 完成唤醒**、**时钟中断**时重新选择 | RR、SRTF、抢占式 Priority、MLFQ、EDF、CFS |

## FCFS / FIFO 先来先服务

FCFS （First-Come, First-Served）按进程到达就绪队列的先后顺序调度，也叫 FIFO。先到的任务先运行，运行期间一般不抢占。

1. **输入任务队列**：按到达顺序排列；
2. **选择队首进程**：CPU 空闲时选择最早到达的进程；
3. **运行到结束或阻塞**：非抢占式执行；
4. **继续下一个进程**：队首出队，后续进程依次执行。

> 最大问题是护航效应：长任务排在前面时，后面很多短任务都会被拖住。

---

## SJF 最短任务优先

SJF （Shortest Job First）每次选择预计运行时间最短的任务先执行，是一种非抢占式调度算法。在所有进程同时到达、运行时间已知、无抢占开销的理想条件下，SJF 对平均等待时间最优。

1. **记录运行时间**：每个任务关联一个预计 CPU burst；
2. **选择最短任务**：就绪队列中执行时间最短者先运行；
3. **非抢占执行**：一旦开始运行，通常运行到完成或阻塞；
4. **重复选择**：继续选择剩余任务中最短者。

> 考试注意：SJF 的“最优”通常指平均等待时间最优，但前提是任务运行时间已知且比较理想；现实中运行时间很难准确知道，长任务也可能饿死。

---

## SRTF 最短剩余时间优先

SRTF (Shortest Remaining Time First) 是 SJF 的抢占式版本。每次调度时，选择剩余运行时间最短的进程。如果新到达进程的执行时间比当前进程剩余时间更短，当前进程会被抢占。

1. **维护剩余时间**：每个就绪进程都有剩余执行时间；
2. **比较当前最短者**：选择剩余时间最短的进程运行；
3. **新进程到达时重新比较**：若新进程更短，则抢占当前进程；
4. **直到所有进程完成**：动态选择最短剩余时间任务。

典型例子：

| 进程 | 到达时间 | 执行时间 |
| -- | ---: | ---: |
| P1 |  0ms |  8ms |
| P2 |  1ms |  4ms |
| P3 |  2ms |  9ms |
| P4 |  3ms |  5ms |

调度结果：

```text
P1  P2  P4  P1  P3
0   1   5   10  17  26
```

- 0ms p1 in,run p1
- 1ms p2 in,r(p1) = 7,r(p2) = 4,run p2
- 2ms p3 in,r(p1) = 7,r(p2) = 3,r(p3) = 9,run p2
- 3ms p4 in,r(p1) = 7,r(p2) = 2,r(p3) = 9,r(p4) = 5,run p2
- 5ms p2 out,run p4
- 10ms p4 out,run p1
- 17ms p1 out,run p3
- 26ms finish all

## CPU 密集型与 I/O 密集型

调度中常把进程分为 CPU 密集型和 I/O 密集型。CPU 密集型进程主要消耗 CPU 计算资源；I/O 密集型进程经常等待磁盘、键盘、网络等 I/O 操作，CPU burst 通常较短。

| 类型      | 特点                 | 调度关注点      |
| ------- | ------------------ | ---------- |
| CPU 密集型 | 长时间使用 CPU，很少主动阻塞   | 避免长期霸占 CPU |
| I/O 密集型 | 经常等待 I/O，CPU 使用时间短 | 提高响应速度     |

> CPU 密集型像一直算题的人，I/O 密集型像不断问问题、等输入的人。

> 交互式系统通常更偏向照顾 I/O 密集型任务，因为用户更关心响应时间。

## RR 时间片轮转调度

RR (Round Robin) 给每个进程分配一个固定时间片。进程在时间片内运行，若时间片用完仍未完成，就被放回就绪队列尾部。

1. **就绪队列排队**：进程按队列顺序等待；
2. **分配时间片**：队首进程获得最多一个时间片；
3. **时间片用完则抢占**：未完成进程回到队尾；
4. **阻塞或完成则让出 CPU**：调度器选择下一个进程。

例如时间片为 4ms：

| 进程 | 执行时间 |
| -- | ---: |
| P1 |  8ms |
| P2 |  5ms |
| P3 |  3ms |

调度顺序：

```text
P1  P2  P3  P1  P2
0   4   8   11  15  16
```

时间片选择很关键：

| 时间片大小 | 后果              |
| ----- | --------------- |
| 太小    | 上下文切换频繁，系统开销大   |
| 太大    | 退化为 FCFS，响应时间变差 |
| 适中    | 兼顾响应时间和切换开销     |

## Priority 优先级调度

优先级调度为每个进程分配一个优先级，每次调度时选择优先级最高的进程。SJF 也可以看作一种特殊的优先级调度：运行时间越短，优先级越高。

1. **设置优先级**：每个进程有一个优先级数；
2. **选择最高优先级进程**：调度器优先运行最高优先级任务；
3. **可抢占或不可抢占**：高优先级任务到达时可立即抢占，也可等待当前任务结束；
4. **处理饥饿问题**：常用老化机制提高长期等待进程的优先级。

> 主要问题是饿死：如果高优先级任务持续出现，低优先级任务可能长期得不到 CPU；解决方法是老化，即等待越久优先级越高。

---

## MFQ / MLFQ 多级反馈队列

多级反馈队列(Multilevel Feedback Queue)使用多个不同优先级的队列。进程可以根据行为在队列之间移动：交互式、I/O 密集型任务倾向保持高优先级；CPU 密集型任务如果用完整个时间片，会逐步下降到低优先级队列。

| 规则     | 含义          |
| ------ | ----------- |
| 多个队列   | 每个队列代表不同优先级 |
| 高优先级队列 | 时间片短，响应快    |
| 低优先级队列 | 时间片长，减少切换开销 |
| 新进程    | 通常进入最高优先级队列 |
| 用完时间片  | 降低优先级       |
| 主动阻塞   | 可能保持原优先级    |
| 定期提升   | 防止低优先级进程饿死  |

典型结构：

```text
Priority 3: RR, time slice = 8ms
Priority 2: RR, time slice = 16ms
Priority 1: FCFS
```

调度策略：

1. **总是先看高优先级队列**；
2. **同一队列内部按该队列算法调度**；
3. **CPU 密集型任务逐渐下沉**；
4. **I/O 密集型任务保持较高优先级**；
5. **定期 priority boost 防止饥饿**。

> MFQ 不是单纯优先级调度，它的关键在于“反馈”，即根据进程过去的运行行为动态调整优先级。

---

## Lottery 彩票调度

Lottery 调度给每个任务分配一定数量的彩票，每次调度时随机抽取一张中奖票，拥有该票的进程获得 CPU。票数越多，被选中的概率越高。

若进程 $P_i$ 拥有 $tickets_i$ 张票，总票数为 $T$，则其被调度概率为：

$$
Pr(P_i)=\frac{tickets_i}{T}
$$

例如：

| 进程 | 彩票数 |     获得 CPU 的期望比例 |
| -- | --: | ---------------: |
| P1 |  10 | $\frac{10}{100}$ |
| P2 |  30 | $\frac{30}{100}$ |
| P3 |  60 | $\frac{60}{100}$ |

---

## Rate Monotonic 单调速率调度

单调速率调度用于周期性实时任务。它按照任务速率分配静态优先级：周期越短，速率越高，优先级越高。

对于周期任务：

| 符号    | 含义                 |
| ----- | ------------------ |
| $C_i$ | 每个周期内需要的计算时间       |
| $P_i$ | 周期                 |
| $D_i$ | 截止时间，常默认 $D_i=P_i$ |

准入控制的基本条件：

$$
\sum_{i=1}^{m}\frac{C_i}{P_i}\le 1
$$

单调速率调度规则：

1. **计算周期**：确定每个任务的周期 $P_i$；
2. **分配静态优先级**：周期越短，优先级越高；
3. **抢占执行**：高优先级任务到达时可抢占低优先级任务；
4. **检查 deadline**：任务必须在截止时间前完成。

> 单调速率就是“越频繁来的任务越优先”。

> 注意：$\sum \frac{C_i}{P_i}\le 1$ 只是 CPU 总利用率不超过 100%，不代表所有静态优先级调度一定成功；RM 有自己的利用率上界。

---

## EDF 最早截止日期优先

EDF (Earliest Deadline First) 根据任务的绝对截止日期动态分配优先级。截止日期越早，优先级越高。它与 Rate Monotonic 不同：RM 是静态优先级，EDF 是动态优先级。

1. **记录每个任务的 deadline**；
2. **选择最早 deadline 的任务运行**；
3. **新任务到达时重新比较 deadline**；
4. **必要时抢占当前任务**。

在单处理器、抢占式、周期任务调度中，EDF 的利用率上界可达到：

$$
\sum_{i=1}^{m}\frac{C_i}{P_i}\le 1
$$

> EDF 就是谁最快到期，谁先处理。

> 考试注意：RM 看周期，EDF 看截止日期；RM 优先级固定，EDF 优先级动态变化。

## CFS 完全公平调度

CFS(Completely Fair Scheduler) 是 Linux 普通任务调度的重要思想。它的目标是让每个进程获得尽可能公平的 CPU 时间。理想情况下，如果有 $N$ 个线程，每个线程在时间 $t$ 内应获得：

$$
\frac{t}{N}
$$

但硬件不能真正做到同时均分，所以 CFS 维护每个任务的虚拟运行时间 `vruntime`，每次选择 `vruntime` 最小的任务运行。

基本思想：

1. **维护虚拟运行时间**：记录任务已经“公平意义上”运行了多久；
2. **选择最小者**：每次调度选择 `vruntime` 最小的任务；
3. **使用红黑树管理任务**：按 `vruntime` 排序；
4. **考虑权重**：高权重任务虚拟时间增长更慢，获得更多 CPU。

加权 CPU 分配公式：

$$
Q_i=\frac{w_i}{\sum_{j=1}^{N}w_j}\times CPU\ time
$$

其中 $w_i$ 是进程 $i$ 的权重。

> CFS 不是按固定时间片轮转，而是不断补偿“运行得少”的任务，让整体趋向公平。

> 易错点：CFS 跟踪的是虚拟运行时间，不是简单物理运行时间；权重越高，`vruntime` 增长越慢。

## 多核 CPU 调度

多核系统中，调度不再只是“一个 CPU 从一个队列选任务”。常见做法有两种：所有 CPU 共享一个全局就绪队列，或每个 CPU 维护自己的运行队列。现代系统常采用每 CPU 私有队列，以减少锁竞争并提升缓存利用率。

| 方式       | 优点          | 缺点                |
| -------- | ----------- | ----------------- |
| 全局队列     | 负载天然比较均衡    | 多 CPU 竞争同一队列，需要加锁 |
| 每 CPU 队列 | 锁竞争少，缓存局部性好 | 需要额外负载均衡机制        |

多核调度还涉及：

| 概念             | 含义                      |
| -------------- | ----------------------- |
| 负载均衡           | 尽量让各 CPU 工作量接近          |
| Push migration | 过载 CPU 主动把任务推给其他 CPU    |
| Pull migration | 空闲 CPU 从繁忙 CPU 拉取任务     |
| 处理器亲和性         | 尽量让线程继续在原 CPU 上运行，以利用缓存 |
| NUMA 感知        | 尽量让线程访问离自己 CPU 更近的内存    |

> 多核调度既要让 CPU 都忙起来，又要尽量别让线程频繁搬家导致缓存失效。

## 优先级反转

优先级反转指高优先级任务反而被低优先级任务间接阻塞。典型情况是低优先级任务持有锁，高优先级任务等待该锁，而中优先级任务又抢占低优先级任务，导致低优先级任务无法释放锁，高优先级任务也无法继续执行。

典型场景：

```text
Low:    获取锁，进入临界区
High:   需要锁，但被阻塞
Medium: 抢占 Low 运行
结果:   Low 无法释放锁，High 被间接阻塞
```

解决方法：

| 方法                        | 核心思想                              |
| ------------------------- | --------------------------------- |
| Priority Ceiling Protocol | 任务获取锁时，优先级提升到该锁的优先级上限             |
| Priority Inheritance      | 低优先级任务持有锁时，如果高优先级任务等待该锁，则临时继承高优先级 |

> 优先级反转就是“重要任务等不重要任务释放锁，但不重要任务又被中等任务挤走”。

> 考试常考三线程例子：低优先级持锁，高优先级等锁，中优先级抢占低优先级。
