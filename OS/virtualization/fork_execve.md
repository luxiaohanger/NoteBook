## fork execve

| 对比项      | `fork`                 | `execve`                   |
| -------- | ---------------------- | -------------------------- |
| 作用       | 创建一个新进程                | 用新程序替换当前进程映像               |
| 是否产生新进程  | 是                      | 否                          |
| PID 是否变化 | 子进程获得新 PID             | 不变                         |
| 地址空间     | 子进程复制父进程地址空间，通常用写时复制   | 当前地址空间被新程序覆盖               |
| 文件描述符    | 子进程继承父进程打开的文件描述符       | 默认保留，除非设置了 `close-on-exec` |
| 返回值      | 父进程返回子进程 PID，子进程返回 `0` | 成功不返回，失败返回 `-1`            |
| 常见配合     | `fork` 后父子并发执行         | 子进程中调用 `execve` 加载目标程序     |


-----
## fork

`fork` 是操作系统用于**创建子进程**的系统调用。调用 `fork` 后，内核会复制当前进程的进程控制信息，为子进程创建独立的进程实体。

```c
pid_t pid = fork();
```

`fork` 调用后，会出现两个几乎相同的进程：

| 进程  | `fork` 返回值 | 含义              |
| --- | ---------: | --------------- |
| 父进程 |    子进程 PID | 父进程知道自己创建了哪个子进程 |
| 子进程 |        `0` | 子进程知道自己是新创建出来的  |
| 出错  |       `-1` | 创建失败，没有子进程产生    |

例如：

```c
pid_t pid = fork();

if (pid < 0) {
    perror("fork failed");
} else if (pid == 0) {
    printf("child process\n");
} else {
    printf("parent process, child pid = %d\n", pid);
}
```

执行后，父进程和子进程都会从 `fork` 返回后的下一条语句继续执行，只是返回值不同。

---

## fork 的复制内容

`fork` 后，子进程在逻辑上继承父进程的大部分运行环境，包括地址空间、寄存器上下文、打开的文件描述符、当前工作目录、环境变量等。

| 内容      | `fork` 后表现              |
| ------- | ----------------------- |
| 代码段     | 子进程与父进程执行同一份程序代码        |
| 数据段、堆、栈 | 逻辑上复制一份                 |
| 寄存器上下文  | 复制，使子进程从 `fork` 返回处继续执行 |
| 文件描述符表  | 复制，指向相同的打开文件对象          |
| 当前工作目录  | 继承                      |
| 环境变量    | 继承                      |
| PID     | 子进程获得新的 PID             |
| PPID    | 子进程的父进程 ID 是父进程 PID     |


> 文件描述符虽然被复制，但底层可能指向同一个打开文件对象，所以文件偏移量可能共享，这是容易混淆的点。

## 写时复制 Copy-on-Write

现代操作系统通常不会在 `fork` 时立刻复制整个地址空间，而是使用**写时复制**，即父子进程先共享同一批物理页，并将这些页标记为只读；只有当某个进程尝试写入共享页时，才复制该页。

流程可以理解为：

1. **调用 `fork`**：内核创建子进程；
2. **复制页表**：父子进程的虚拟地址映射到相同物理页；
3. **标记只读**：共享页被标记为只读；
4. **发生写入**：某个进程写共享页时触发缺页异常；
5. **复制物理页**：内核复制该页，使写入进程获得自己的副本；
6. **恢复执行**：写入只影响自己的地址空间。

> 写时复制的核心思想是：先不急着复制，等真的要修改时再复制。

## execve

`execve` 是操作系统用于**执行新程序**的系统调用。它不会创建新进程，而是把当前进程的用户态地址空间替换成另一个程序的内容。

```c
int execve(const char *pathname, char *const argv[], char *const envp[]);
```

三个参数含义：

| 参数         | 含义       |
| ---------- | -------- |
| `pathname` | 要执行的程序路径 |
| `argv`     | 命令行参数数组  |
| `envp`     | 环境变量数组   |

例如：

```c
char *argv[] = {"/bin/ls", "-l", NULL};
char *envp[] = {NULL};

execve("/bin/ls", argv, envp);
perror("execve failed");
```

如果 `execve` 成功，当前进程会被 `/bin/ls` 替换，后面的 `perror` 不会执行。只有 `execve` 失败时，才会返回 `-1` 并继续执行后续代码。

---

## execve 替换了什么

`execve` 成功后，当前进程的用户态程序映像被替换。

| 内容       | `execve` 后表现            |
| -------- | ----------------------- |
| 代码段      | 替换为新程序代码                |
| 数据段      | 替换为新程序数据                |
| 堆        | 重新初始化                   |
| 栈        | 重新构造，放入 `argv`、`envp` 等 |
| 程序入口     | 跳转到新程序入口点               |
| PID      | 不变                      |
| 打开的文件描述符 | 默认保留                    |
| 当前工作目录   | 保留                      |
| 环境变量     | 使用传入的 `envp`            |
| 信号处理     | 部分状态会重置                 |
| 父子关系     | 不变                      |

所以 `execve` 的本质不是“启动另一个进程”，而是“把当前进程装载成另一个程序”。

> `execve` 改的是进程的程序映像，不改进程身份。

> 文件描述符默认跨 `execve` 保留，因此 shell 才能先重定向标准输入输出，再 `exec` 目标程序。

---

## fork + execve 的典型用法

在 Unix/Linux 中，创建新程序通常不是直接“启动程序”，而是：

1. 父进程调用 `fork` 创建子进程；
2. 子进程调用 `execve` 加载新程序；
3. 父进程可以调用 `wait` 等待子进程结束。

```c
pid_t pid = fork();

if (pid < 0) {
    perror("fork failed");
} else if (pid == 0) {
    char *argv[] = {"/bin/ls", "-l", NULL};
    char *envp[] = {NULL};

    execve("/bin/ls", argv, envp);

    perror("execve failed");
    _exit(1);
} else {
    wait(NULL);
    printf("child finished\n");
}
```

## fork 后的执行次数问题

如果程序中有多个 `fork`，进程数会成倍增加。每个成功的 `fork` 都会让当前进程复制出一个新进程。

例如：

```c
fork();
fork();
printf("hello\n");
```

假设两个 `fork` 都成功：

| 执行阶段          | 进程数 |
| ------------- | --: |
| 初始            |   1 |
| 第一次 `fork` 后  |   2 |
| 第二次 `fork` 后  |   4 |
| `printf` 执行次数 |   4 |

所以会输出 4 次：

```text
hello
hello
hello
hello
```

一般规律：

$$
n\text{ 个连续成功的 fork 之后，进程数}=2^n
$$
