## 总览

| 考点        | 核心问题             | 关键点                                 |
| --------- | ---------------- | ----------------------------------- |
| 文件读写 API  | 用户程序如何操作文件       | `open/read/write/close/lseek/fsync` |
| 文件描述符与偏移量 | 内核如何记住“读写到哪了”    | FD、打开文件表、file offset                |
| 读文件流程     | `read()` 到底发生了什么 | 路径解析、inode、权限检查、数据块读取、offset 更新     |
| 写文件流程     | `write()` 为什么更复杂 | 可能分配新块、更新 bitmap、inode、数据块          |
| 缓存与缓冲     | 为什么读写不一定直接访问磁盘   | page cache、预取、延迟写、fsync             |
| 一致性问题     | 崩溃时为什么可能损坏文件系统   | 多个元数据更新不是整体原子的                      |

## 文件读写 API

文件系统通过系统调用向用户程序提供统一的文件访问接口，典型 API 包括 `open`、`read`、`write`、`close`、`lseek`、`fsync`。用户程序不直接操作磁盘块，而是通过这些接口请求内核完成路径解析、权限检查、块映射和磁盘 I/O。

| API                         | 作用                | 关键影响         |
| --------------------------- | ----------------- | ------------ |
| `open(path, flags)`         | 打开文件，返回文件描述符      | 建立进程到文件的访问入口 |
| `read(fd, buf, n)`          | 从文件当前位置最多读 $n$ 字节 | 读完后文件偏移量自动前进 |
| `write(fd, buf, n)`         | 从文件当前位置写入 $n$ 字节  | 写完后文件偏移量自动前进 |
| `close(fd)`                 | 关闭文件描述符           | 释放打开文件表相关引用  |
| `lseek(fd, offset, whence)` | 显式移动文件偏移量         | 支持随机访问       |
| `fsync(fd)`                 | 强制把脏数据写回磁盘        | 提高持久性保证      |

> 文件读写本质上是“拿着文件描述符，对文件当前偏移量处的数据进行读或写”。

> `write()` 返回成功不一定代表数据已经立刻落盘，可能只是写入了内核缓冲区；真正需要强持久性时要考虑 `fsync()`。

## 文件描述符与文件偏移量

文件描述符是操作系统分配给某个进程的整数，用于引用已经打开的文件。它是进程私有的，不同进程里的同一个数字不一定指向同一个文件。内核通过文件描述符表、系统打开文件表和 inode 表来管理文件访问。

核心结构可以理解为：

```text
进程
  └── 文件描述符表 FD
        └── 指向系统打开文件表项
              ├── 文件偏移量 offset
              ├── 打开模式 flags
              └── 指向 inode
                    ├── 文件类型
                    ├── 文件大小
                    ├── 权限
                    └── 数据块索引
```

文件偏移量决定下一次 `read()` 或 `write()` 从哪里开始。若读写了 $N$ 字节，则偏移量通常自动增加：

$$
offset_{\text{new}} = offset_{\text{old}} + N
$$

也可以用 `lseek()` 显式修改：

$$
offset_{\text{new}} = \text{lseek指定的位置}
$$

> 文件描述符像“书签入口”，文件偏移量像“书签到哪一页”。

> `fork()` 后父子进程可能共享同一个打开文件表项，因此可能共享同一个文件偏移量；而两个进程分别 `open()` 同一个文件时，通常各有自己的打开文件表项和偏移量。

## 读文件流程

读文件一般分为两步：先 `open()`，再 `read()`。`open()` 负责把路径名转换成 inode，并完成权限检查；`read()` 负责根据当前偏移量找到对应数据块，把数据复制给用户程序。

1. **路径解析**：从根目录或当前目录开始，逐级查找目录项；
2. **找到 inode**：目录项给出文件名到 inode 号的映射；
3. **权限检查**：检查当前用户是否有读取权限；
4. **建立打开文件项**：内核创建或引用打开文件表项，并返回文件描述符；
5. **执行 read**：根据文件偏移量和 inode 中的块索引找到数据块；
6. **读取数据块**：从缓存或磁盘读取数据；
7. **更新元数据**：可能更新最近访问时间；
8. **更新 offset**：读了多少字节，偏移量就增加多少。

```text
read(fd, buf, n)
        │
        ▼
根据 fd 找到打开文件表项
        │
        ▼
取得当前 offset 和 inode
        │
        ▼
由 inode 把 文件偏移量 映射为 磁盘块号
        │
        ▼
读缓存 / 读磁盘块
        │
        ▼
复制数据到用户缓冲区 buf
        │
        ▼
offset += 实际读到的字节数
```

> 读文件不是每次都重新找文件名，而是通过文件描述符直接找到已经打开的文件状态。

> 到达文件末尾时，`read()` 可能返回 $0$；这表示 EOF，不是错误。

## 写文件流程

写文件比读文件复杂，因为写操作不仅可能修改已有数据，还可能让文件变大，从而需要分配新的数据块，并更新多个文件系统元数据。

写已有块时，大致过程是：

1. **根据 fd 找到打开文件项**；
2. **根据 offset 定位文件逻辑块**；
3. **找到或读入对应磁盘块**；
4. **把用户数据写入缓存或磁盘块**；
5. **更新 offset**；
6. **更新 inode 元数据**，如文件大小、修改时间。

如果写入位置超出当前文件已有块，则还要：

1. **读取空闲空间结构**：例如 data bitmap；
2. **寻找空闲数据块**；
3. **修改 bitmap**：把该块标记为已占用；
4. **修改 inode**：把新数据块加入文件的块索引；
5. **写入数据块**；
6. **更新文件大小和修改时间**。

```text
write(fd, buf, n)
        │
        ▼
根据 fd 找到打开文件表项
        │
        ▼
根据 offset 和 inode 定位写入位置
        │
        ├── 覆写已有块：直接修改数据块
        │
        └── 扩展文件：分配新块 + 更新 bitmap + 更新 inode
        │
        ▼
写入数据
        │
        ▼
offset += 实际写入字节数
```

> 写文件不仅写“数据”，还经常要写“管理数据的数据”，也就是元数据。

> 创建新文件更复杂：除了分配 inode 和数据块，还要修改父目录的数据，把文件名和 inode 号加入目录项。

## 文件偏移与顺序访问、随机访问

文件读写默认是顺序访问：每次 `read()` 或 `write()` 后，文件偏移量自动向后移动。若想跳到文件中间读写，就使用 `lseek()` 修改偏移量。

顺序访问：

```c
read(fd, buf, 4096);
read(fd, buf, 4096);
read(fd, buf, 4096);
```

每次读取后：

$$
offset = offset + 4096
$$

随机访问：

```c
lseek(fd, 8192, SEEK_SET);
read(fd, buf, 4096);
```

表示先跳到文件第 $8192$ 字节处，再读取。

> 顺序读写像从头到尾看书，随机读写像直接翻到某一页。

> `lseek()` 只改变文件偏移量，本身不读写数据。

## 缓存与缓冲

文件系统不会总是直接访问磁盘。读文件时，常用数据块可能已经在内存缓存中；写文件时，数据可能先进入内核缓冲区，稍后再批量写回磁盘。这样可以减少磁盘 I/O，提高性能。

| 机制             | 作用             | 典型场景        |
| -------------- | -------------- | ----------- |
| 缓存 Caching     | 保存常用磁盘块，避免重复读盘 | 多次读取同一文件    |
| 预取 Prefetching | 顺序读时提前加载后续块    | 复制文件、顺序扫描   |
| 写缓冲 Buffering  | 延迟、合并、调度写操作    | 多次小写入合并成大写入 |
| `fsync()`      | 强制写回磁盘         | 数据库、重要文件保存  |

写缓冲的好处是性能高，但代价是崩溃时可能丢失尚未落盘的数据。

> 缓存让读更快，缓冲让写更快，但缓冲写会带来崩溃一致性风险。

> `close()` 不等价于严格意义上的 `fsync()`；考试中要区分“写入内核缓冲”和“真正持久化到磁盘”。

## 文件读写中的一致性问题

一次写文件可能涉及多个磁盘写操作，例如：

1. 写新数据块；
2. 更新 inode，让 inode 指向新数据块；
3. 更新 bitmap，把该块标记为已占用。

如果系统在这些操作中途崩溃，就可能出现不一致状态。

| 成功写入的部分                 | 可能结果                             |
| ----------------------- | -------------------------------- |
| 只写数据块                   | 数据不可达，通常元数据仍一致                   |
| 只写 inode                | inode 指向一个 bitmap 认为空闲的块，严重不一致   |
| 只写 bitmap               | bitmap 认为块已占用，但没有 inode 指向它，空间泄露 |
| inode + bitmap 成功，数据块失败 | 元数据一致，但文件可能读到垃圾数据                |
| inode + 数据块成功，bitmap 失败 | inode 与 bitmap 不一致               |

为了解决这类问题，常见方法有：

| 方法            | 思路                  | 缺点                     |
| ------------- | ------------------- | ---------------------- |
| `fsck`        | 崩溃后扫描文件系统，修复元数据不一致  | 慢，且只能修到“一致”，不一定恢复到“正确” |
| 日志 Journaling | 修改前先写日志，崩溃后按日志重做或忽略 | 正常写入路径上有额外开销           |

> 一致性问题的根源是：文件系统的一次逻辑更新，底层通常需要多个磁盘写操作。

> 磁盘通常只能保证单个块写入的原子性，不能自动保证一组元数据更新整体原子。
