## 总览

| 类型                            | 本质         | 指向对象      | 是否新建 inode | 删除原文件后的影响       | 是否可跨文件系统 | 是否可链接目录 |
| ----------------------------- | ---------- | --------- | ---------- | --------------- | -------- | ------- |
| 硬链接 Hard Link                 | inode 号的别名 | 同一个 inode | 否          | 只要还有硬链接，文件数据仍存在 | 否        | 通常不允许   |
| 软链接 Symbolic Link / Soft Link | 路径名的别名     | 目标文件路径    | 是          | 目标删除后变成悬空链接     | 可以       | 可以      |

---

## 硬链接

硬链接是在目录中创建另一个文件名，并让这个文件名指向原文件相同的 inode 号。也就是说，两个文件名本质上引用的是同一个文件控制块和同一批数据块。

```bash
ln file.txt hardlink.txt
```

创建后：

```text
file.txt       ─┐
                ├── inode 100 ─── 数据块
hardlink.txt   ─┘
```

硬链接的核心特点：

| 特点          | 说明                        |
| ----------- | ------------------------- |
| 不复制文件数据     | 只是多了一个目录项                 |
| 共享 inode    | 文件大小、权限、数据块等元数据相同         |
| 链接计数增加      | inode 中的 `links_count` 加一 |
| 删除任一名字不影响数据 | 只要链接计数不为 $0$，文件仍存在        |
| 不能跨文件系统     | inode 号只在同一个文件系统内唯一       |
| 通常不能链接目录    | 防止目录结构出现循环，避免 `..` 指向不明确  |

文件真正删除的条件是：

$$
links_count = 0
$$

也就是所有硬链接都被 `unlink()` 删除后，inode 和数据块才会被释放。

> 硬链接可以理解为“同一个文件有多个正式名字”。

> 考试常考：硬链接不是快捷方式，也不是副本；它和原文件地位平等，没有“谁指向谁”的主从关系。

---

## 软链接

软链接又叫符号链接，是一种特殊类型的文件。它有自己的 inode，文件内容保存的是目标文件的路径名。访问软链接时，文件系统会继续解析它保存的路径，找到真正的目标文件。

```bash
ln -s file.txt softlink.txt
```

创建后：

```text
softlink.txt ── inode 200 ── 内容："file.txt"
                              │
                              ▼
file.txt     ── inode 100 ── 数据块
```

软链接的核心特点：

| 特点         | 说明                            |
| ---------- | ----------------------------- |
| 有自己的 inode | 软链接本身也是一个文件                   |
| 保存路径名      | 链接文件内容是目标路径                   |
| 可以跨文件系统    | 因为依赖路径名，不依赖同一文件系统内的 inode 号   |
| 可以链接目录     | 常用于给目录建立快捷入口                  |
| 目标删除后链接仍存在 | 但会变成 dangling reference，即悬空引用 |
| 解析效率略低     | 需要额外解析路径                      |

> 软链接可以理解为“保存目标路径的快捷方式”。

> 考试常考：删除软链接不会删除目标文件；但删除目标文件后，软链接会失效。

---

## 硬链接与软链接对比

| 对比项     | 硬链接           | 软链接          |
| ------- | ------------- | ------------ |
| 本质      | 目录项指向同一 inode | 一个保存路径名的特殊文件 |
| inode   | 与原文件相同        | 拥有独立 inode   |
| 数据块     | 共享同一批数据块      | 自己的数据块中保存路径名 |
| 删除原文件名  | 文件仍可通过其他硬链接访问 | 软链接可能失效      |
| 删除链接本身  | 只减少链接计数       | 只删除软链接文件     |
| 跨文件系统   | 不可以           | 可以           |
| 链接目录    | 通常不允许         | 可以           |
| 是否有主从关系 | 没有，多个名字地位平等   | 有，软链接依赖目标路径  |

---

## 删除时的区别

假设：

```bash
ln file.txt hard.txt
ln -s file.txt soft.txt
rm file.txt
```

结果：

```text
hard.txt 仍然能访问原数据
soft.txt 失效，因为它保存的路径 file.txt 不存在了
```

原因是：

```text
hard.txt ── inode 100 ── 数据块仍存在
soft.txt ── 内容："file.txt" ── 该路径已不存在
```

> 硬链接关心 inode 是否还被引用；软链接关心目标路径是否还能被解析。

> 判断链接类型时，看它是“指向 inode”还是“保存路径名”。
