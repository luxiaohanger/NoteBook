## 硬链接 软链接

---

# 1. 硬链接是什么？

**硬链接 hard link**：多个文件名指向**同一个 inode**。

例如：

```bash
ln a.txt b.txt
```

得到：

```text
目录项
├── a.txt → inode 100
└── b.txt → inode 100
```

此时 `a.txt` 和 `b.txt` 不是“一个原文件 + 一个快捷方式”，而是**两个名字指向同一个文件实体**。

---

## 硬链接的特点

### 1. 共享同一个 inode

```text
a.txt ─┐
       ├── inode 100 ─── 数据块
b.txt ─┘
```

所以它们的：

```text
文件内容
权限
所有者
大小
时间戳
数据块位置
```

本质上都来自同一个 inode。

---

### 2. 修改任意一个，另一个也变化

因为它们指向同一份数据。

```bash
echo "hello" >> a.txt
cat b.txt
```

可以看到 `b.txt` 中也有 `hello`。

原因不是同步复制，而是它们本来就是同一个 inode。

---

### 3. 删除一个名字，文件不一定消失

例如：

```bash
rm a.txt
```

之后：

```text
b.txt → inode 100
```

只要还有目录项指向这个 inode，文件内容就还在。

inode 中有一个字段叫**硬链接计数 link count**：

```text
inode 100
├── link count = 2
```

删除 `a.txt` 后：

```text
inode 100
├── link count = 1
```

只有当 link count 变成 0，且没有进程打开该文件时，文件数据才会被释放。

---

### 4. 一般不能跨文件系统

硬链接保存的是 inode 号，而 inode 号只在同一个文件系统内部有意义。

所以：

```text
文件系统 A 的 inode 100
文件系统 B 的 inode 100
```

不是同一个东西。

因此硬链接通常不能跨文件系统或跨分区。

---

### 5. 一般不能给目录创建硬链接

普通用户通常不能对目录创建硬链接。

原因是如果目录可以随便硬链接，可能造成目录树出现环，路径遍历和引用计数管理会变复杂。

例如可能出现：

```text
dir1/sub → dir1
```

这样会产生循环目录结构。

---

# 2. 软链接是什么？

**软链接 symbolic link，也叫符号链接**：一个特殊文件，里面保存的是另一个文件的路径名。

例如：

```bash
ln -s a.txt c.txt
```

得到：

```text
c.txt → "a.txt"
```

注意：`c.txt` 自己有自己的 inode。

```text
c.txt → inode 200
inode 200 的数据内容 = "a.txt"

a.txt → inode 100
inode 100 → 真正文件内容
```

---

## 软链接的特点

### 1. 软链接有自己的 inode

```text
c.txt ── inode 200 ── 保存路径 "a.txt"
                  ↓
               找到 a.txt
                  ↓
a.txt ── inode 100 ── 数据块
```

所以软链接和目标文件不是同一个 inode。

---

### 2. 软链接类似快捷方式

访问软链接时，系统会读取软链接保存的路径，再继续解析目标路径。

例如：

```bash
cat c.txt
```

系统过程大致是：

```text
找到 c.txt 的 inode
↓
发现它是符号链接
↓
读取其中保存的路径 "a.txt"
↓
再去查找 a.txt
↓
读取 a.txt 的内容
```

---

### 3. 可以跨文件系统

因为软链接保存的是路径名，而不是 inode 号。

例如：

```bash
ln -s /mnt/disk2/a.txt link.txt
```

只要路径能解析到目标，就可以访问。

---

### 4. 可以指向目录

例如：

```bash
ln -s /home/user/docs docs_link
```

此时 `docs_link` 可以像目录一样进入：

```bash
cd docs_link
```

---

### 5. 目标删除后，软链接会失效

例如：

```bash
ln -s a.txt c.txt
rm a.txt
cat c.txt
```

此时 `c.txt` 还存在，但指向的目标 `a.txt` 不存在了。

这种链接叫**悬空链接 dangling symlink**。

```text
c.txt → "a.txt"
        但是 a.txt 已不存在
```

---

# 3. 硬链接 vs 软链接对比

| 对比项            | 硬链接              | 软链接             |
| -------------- | ---------------- | --------------- |
| 本质             | 目录项指向同一个 inode   | 一个特殊文件，保存目标路径   |
| 是否有独立 inode    | 没有新的文件实体；只是新增目录项 | 有自己的 inode      |
| 与目标 inode 是否相同 | 相同               | 不同              |
| 删除原文件后         | 只要还有硬链接，数据仍在     | 软链接失效           |
| 能否跨文件系统        | 通常不能             | 可以              |
| 能否链接目录         | 普通用户通常不能         | 可以              |
| 是否像快捷方式        | 不太像              | 很像              |
| 创建命令           | `ln old new`     | `ln -s old new` |

# 4. 用图理解

## 硬链接

```text
目录项
├── a.txt ─┐
└── b.txt ─┘
          ↓
       inode 100
          ↓
       数据块
```

`a.txt` 和 `b.txt` 是同一个文件的两个名字。

---

## 软链接

```text
目录项
├── a.txt → inode 100 → 数据块
└── c.txt → inode 200 → 内容是路径 "a.txt"
                         ↓
                      再解析到 a.txt
```

`c.txt` 是一个保存路径的特殊文件。

---

# 5. 删除时的区别

假设：

```bash
echo "hello" > a.txt
ln a.txt b.txt
ln -s a.txt c.txt
```

此时：

```text
a.txt → inode 100
b.txt → inode 100
c.txt → inode 200，内容是 "a.txt"
```

删除 `a.txt`：

```bash
rm a.txt
```

结果：

```text
b.txt → inode 100，仍然能访问 hello
c.txt → "a.txt"，失效
```

原因：

```text
硬链接 b.txt 直接指向 inode 100
软链接 c.txt 只是保存路径 a.txt
```

---

# 6. 考试记忆版

> 硬链接是在目录中增加一个目录项，使多个文件名指向同一个 inode，因此它们共享同一份文件数据和元数据。删除其中一个文件名不会删除文件本身，只有硬链接计数为 0 且没有进程打开时，文件数据才会被释放。硬链接通常不能跨文件系统，也通常不能链接目录。
>
> 软链接又称符号链接，是一种特殊文件，拥有自己的 inode，其内容是目标文件的路径名。访问软链接时系统会根据保存的路径重新解析目标文件。软链接可以跨文件系统，也可以指向目录，但如果目标文件被删除，软链接会变成悬空链接。
