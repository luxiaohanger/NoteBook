## 内存模型

| 模型/概念    | 核心保证                        | 允许的乱序            | 典型结论                |
| -------- | --------------------------- | ---------------- | ------------------- |
| 内存一致性模型  | 规定多核对共享内存读写顺序的可见规则          | 取决于具体模型          | 解决“不同核心看到的内存顺序是否一致” |
| 顺序一致性 SC | 所有核心看到同一个全局访存顺序，且每个线程内部顺序不变 | 基本不允许跨线程可见乱序     | 最符合直觉，但硬件代价高        |
| TSO      | 保证多数操作全局顺序，但不保证“写后读”全局一致    | 允许 Store-Load 乱序 | x86 近似采用该模型         |
| 宽松内存模型   | 不保证不同地址、无依赖访存操作的顺序          | 读读、读写、写读、写写都可能乱序 | ARM、RISC-V 更接近此类模型  |
| 内存屏障     | 强制屏障前后的内存操作按要求可见            | 用屏障禁止某些乱序        | 并发通信、锁实现中非常关键       |


---

简单来说，内存模型主要关心这些问题：
- 1.同线程指令乱序重排规则 
- 2.不同线程（核心）间写指令（store）的可见性

SC要求线程内指令不允许重排 + 写指令执行后对其他核心立即可见

TSO要求线程内指令不允许重排 + 写指令对自己立即可见，对其他核心没有保证

RMM没有保证

barrier要求线程内屏障前指令一定先于屏障后指令执行 + 不保证全局可见性

---

## 内存一致性模型

内存一致性模型，即 Memory Consistency Model，是多处理器系统中规定共享内存读写操作“以什么顺序被其他处理器看见”的规则。

在单线程程序中，我们通常默认代码按程序顺序执行；但在多处理器并发程序中，编译器和 CPU 都可能为了性能重排指令。课件中指出，现代处理器允许指令乱序执行，尤其是高延迟访存指令，例如 cache miss 时，处理器可能先执行后面的指令来隐藏访存延迟；这种乱序会导致多个处理器看到不一致的访存顺序，因此需要内存一致性模型定义共享内存操作的顺序约束。

> 内存一致性模型就是规定：多核同时读写共享变量时，大家到底应该“看到什么先发生、什么后发生”。

> 不要把“程序写的顺序”直接等同于“其他核心看到的顺序”；并发程序正确性经常就错在这里。

---

## 顺序一致性模型

顺序一致性模型，Sequential Consistency，简称 SC，是最符合直觉的内存模型。

它提供两个保证：

1. **全局顺序一致**：所有核心看到的共享内存访问顺序完全一致；
2. **线程内部顺序一致**：在这个全局顺序中，每个线程自己的读写操作顺序必须和程序顺序一致。

可以理解为：所有线程的内存操作被放进一个全局队列中，每次只执行一个共享内存读写操作，并且每个线程自己的操作不能前后颠倒。

例如：

```c
int x = 0, y = 0;

void T1() {
    x = 1;
    int t = y;
    printf("%d", t);
}

void T2() {
    y = 1;
    int t = x;
    printf("%d", t);
}
```

在顺序一致性模型下，可能输出：

| T1 读到的 `y` | T2 读到的 `x` | 是否可能 |
| ---------- | ---------: | ---- |
| 0          |          1 | 可能   |
| 1          |          0 | 可能   |
| 1          |          1 | 可能   |
| 0          |          0 | 不可能  |

因为若 T1 读到 `y = 0`，说明 `T1` 的读发生在 `T2` 的写 `y = 1` 之前；若 T2 又读到 `x = 0`，说明 `T2` 的读发生在 `T1` 的写 `x = 1` 之前。这样会形成矛盾：

$$
x=1 < read(y) < y=1 < read(x) < x=1
$$

> 顺序一致性就是“所有人都看同一部时间线，而且每个线程自己的剧情不能乱序”。

> SC 很好理解，但现代 CPU 很少直接提供完整 SC，因为它会限制乱序执行、缓存和流水线优化，性能代价较大。

---

## TSO 内存模型

TSO，Total Store Ordering，是比顺序一致性更宽松的一种内存模型。课件中说明，TSO 保证不同地址且无依赖的“读读、读写、写写”操作之间的全局可见顺序，但不保证“写读”的全局可见顺序；x86 支持的就是这个模型。

也就是说，TSO 主要放松的是：

$$
Store \rightarrow Load
$$

例如：

```c
int x = 0, y = 0;

void T1() {
    x = 1;
    int t = y;
}

void T2() {
    y = 1;
    int t = x;
}
```

在 TSO 下，可能出现：

| T1 读到的 `y` | T2 读到的 `x` | 原因                                               |
| ---------- | ---------: | ------------------------------------------------ |
| 0          |          0 | 每个处理器自己的写先进入本地 store buffer，还没被其他核心看见，就先执行了后续读操作 |

核心原因是：一个处理器可能比其他处理器更早看到自己的写。于是线程 1 已经认为自己执行了 `x = 1`，但线程 2 还看不到；线程 2 已经认为自己执行了 `y = 1`，但线程 1 还看不到。

> TSO 不是完全乱序，它主要允许“先写后读”在全局可见效果上像是被反过来了。

> 考试常考：SC 下 `0,0` 不可能；TSO 下 `0,0` 可能。

---

## 宽松内存模型

宽松内存模型，Relaxed Memory Model，比 TSO 更弱。课件中说明，宽松内存模型不保证任何不同地址且无依赖访存操作之间的顺序，也就是说读读、读写、写读、写写都可以在全局可见顺序上乱序；ARM 和 RISC-V 的内存模型属于此类。

可乱序类型包括：

| 原顺序           | 是否可能乱序 | 例子                          |
| ------------- | ------ | --------------------------- |
| Load → Load   | 可能     | 先读 `a`，后读 `b`，其他核心可能观察到相反效果 |
| Load → Store  | 可能     | 读后写可能被重排                    |
| Store → Load  | 可能     | TSO 也允许这种情况                 |
| Store → Store | 可能     | 两次写入的可见顺序可能被其他核心看成反的        |

经典问题是消息传递：

```c
#define NOT_READY 0
#define READY 1

int data = 0;
int flag = NOT_READY;

void thread_A(void) {
    data = 123;
    flag = READY;
}

void thread_B(void) {
    while (flag != READY);
    handle(data);
}
```

程序员的直觉是：

$$
data = 123 \rightarrow flag = READY
$$

所以只要线程 B 看到 `flag == READY`，就应该能看到 `data == 123`。

但在宽松内存模型下，`data = 123` 和 `flag = READY` 是两个不同地址的写操作，可能发生写写乱序。于是线程 B 可能先看到 `flag == READY`，却还看不到 `data = 123`，从而读到旧值 `data = 0`。

> 宽松内存模型下，不能只靠“代码顺序”实现线程通信，必须使用同步原语或内存屏障。

> 看到“先写数据，再写 flag 通知”这类题，要立刻想到写写乱序和内存屏障。

## 内存屏障

内存屏障，Memory Barrier / Fence，用来限制编译器或 CPU 对内存操作的重排，使屏障前后的内存访问按照要求对其他核心可见。

课件中给出的硬件内存屏障例子是：

```c
__sync_synchronize();
```

在消息传递例子中，可以写成：

```c
void thread_A(void) {
    data = 123;
    __sync_synchronize();
    flag = READY;
}

void thread_B(void) {
    while (flag != READY);
    __sync_synchronize();
    handle(data);
}
```

这里屏障的作用是保证：

$$
data = 123 \rightarrow flag = READY
$$

并且接收者看到 `flag == READY` 后，再读取 `data` 时不会越过同步点。

> 内存屏障就是给乱序执行加一道“不能越过这里”的栅栏。

> `volatile` 不能等价替代锁或内存屏障；它主要限制编译器优化，不一定提供跨 CPU 的完整同步语义。

## 与原子性、顺序性、全局一致性的关系

课件把并发程序难点概括为三类：原子性丧失、顺序性丧失、全局一致性丧失。内存一致性模型主要对应第三类：全局一致性丧失。

| 问题      | 含义                  | 例子                                    | 常见解决                  |
| ------- | ------------------- | ------------------------------------- | --------------------- |
| 原子性丧失   | 一个操作被拆成多步，中间被其他线程干扰 | `sum++` 不是原子的                         | 锁、原子指令                |
| 顺序性丧失   | 编译器或 CPU 改变执行顺序     | `while(!done);` 被优化成死循环               | 编译器屏障、`volatile`、同步原语 |
| 全局一致性丧失 | 不同核心看到的内存操作顺序不同     | TSO 下 `0,0`，Relaxed 下 flag 先于 data 可见 | 内存屏障、锁、条件变量、原子变量      |

> 原子性关心“一个操作会不会被打断”，顺序性关心“本线程语句会不会被重排”，一致性关心“其他核心看到的顺序是不是一样”。

> 并发题不要只分析代码表面顺序，还要问：是否有数据竞争？是否有同步？硬件模型是否允许乱序？

## 考试速记

| 模型      | 一句话记忆             | 典型判断                                            |
| ------- | ----------------- | ----------------------------------------------- |
| SC      | 所有线程共享同一条全局时间线    | `Store x; Load y` / `Store y; Load x` 不可能同时读到 0 |
| TSO     | 写后读可能乱序，自己的写自己先看到 | x86 下可能出现两个线程都读到 0                              |
| Relaxed | 只要不同地址无依赖，几乎都可能乱序 | `data` 和 `flag` 的写入顺序可能被看反                      |
| Barrier | 禁止屏障两侧内存操作越界乱序    | 用于实现可靠同步                                        |

> 最核心的判断方法：先按 SC 分析是否可能；如果 SC 不可能，再看 TSO 或 Relaxed 是否允许对应乱序。
