## 运输层的多路复用与多路分解

| 概念   | 方向                         | 作用                     | 依据             |
| ---- | -------------------------- | ---------------------- | -------------- |
| 多路复用 | 发送端：应用进程 $\rightarrow$ 运输层 | 多个应用进程的数据共用一个运输层协议发送出去 | 端口号、IP 地址、协议类型 |
| 多路分解 | 接收端：运输层 $\rightarrow$ 应用进程 | 收到报文后交给正确的应用进程         | 端口号、IP 地址、协议类型 |

## TCP 可靠传输实现机制

TCP 通过**序号、确认、重传、校验、流量控制、拥塞控制**等机制，在不可靠的 IP 服务之上实现可靠的字节流传输。

| 机制      | 作用           | 核心含义               |
| ------- | ------------ | ------------------ |
| 序号机制    | 保证数据有序       | 每个字节都有序号           |
| 确认机制    | 告诉发送方哪些数据已收到 | ACK 表示期望收到的下一个字节序号 |
| 超时重传    | 处理数据或 ACK 丢失 | 超过 RTO 未确认就重传      |
| 快速重传    | 更快发现丢包       | 收到多个重复 ACK 后立即重传   |
| 校验和     | 检测传输差错       | 出错报文段直接丢弃          |
| 接收缓存与重排 | 处理乱序到达       | 先缓存乱序段，最后按序交付      |
| 滑动窗口    | 提高效率并控制发送范围  | 允许多个报文段连续发送        |
| 流量控制    | 防止接收方被淹没     | 接收方通告接收窗口          |
| 拥塞控制    | 防止网络被压垮      | 根据网络拥塞程度调整发送速率     |

## 自动重选请求协议（ARQ）分类

| 重传方式               |    发送窗口 |    接收窗口 | 是否连续发送     | 是否缓存乱序帧 | ACK 特点                    | 重传规则                  | 优点           | 缺点            |
| ------------------ | ------: | ------: | ---------- | ------- | ------------------------- | --------------------- | ------------ | ------------- |
| 停等协议 Stop-and-Wait |     $1$ |     $1$ | 否          | 否       | 每发一个帧等一个 ACK              | 某帧超时后只重传该帧            | 实现最简单        | 信道利用率低        |
| 回退 $N$ 帧 GBN       | $W_T>1$ |     $1$ | 是          | 否       | 累积确认，$ACK\ k$ 表示期望第 $k$ 帧 | 某帧出错或超时后，从该帧开始后续帧全部重传 | 效率高于停等，实现较简单 | 出错时重传浪费大      |
| 选择重传 SR            | $W_T>1$ | $W_R>1$ | 是          | 是       | 对每个正确帧分别确认                | 只重传出错或丢失的帧            | 出错时代价小，效率高   | 实现复杂，需要缓存乱序帧  |

## GBN 的最大平均速率

GBN 的**最大平均速率**通常指：在理想情况下没有差错、没有重传时，发送方能达到的最大有效发送速率。

| 情况      | 最大平均速率                                                  |
| ------- | ------------------------------------------------------- |
| 发送窗口足够大 | 可达到信道带宽 $B$                                             |
| 发送窗口不够大 | 受窗口限制，最大平均速率为 $\dfrac{W \cdot L}{RTT+T_f}$              |
| 综合写法    | $R_{\max}=\min\left(B,\dfrac{W\cdot L}{RTT+T_f}\right)$ |

其中：


帧发送时延为：

$$
T_f=\frac{L}{B}
$$

所以最大平均速率可以写成：

$$
R_{\max}=\min\left(B,\frac{W\cdot L}{RTT+T_f}\right)
$$

> 一句话理解：窗口里最多压入 $W$ 个帧，如果这 $W$ 个帧发完后 ACK 还没回来，发送方就得停下来等，所以速率会被窗口大小限制。

## 吞吐量和利用率
设链路带宽为 $B$，信道利用率为 $U$，吞吐量为 $R$，则：

$$
R=B\cdot U
$$

> 带宽是理论上限，吞吐量是实际达到的速率，利用率是达到上限的比例。

> 最大吞吐量 = 最大平均速率

| 符号    | 含义                 |
| ----- | ------------------ |
| $B$   | 信道带宽，单位 bit/s      |
| $W$   | GBN 发送窗口大小         |
| $L$   | 每个数据帧长度，单位 bit     |
| $RTT$ | 往返传播时延             |
| $T_f$ | 发送一个数据帧所需时间，即帧发送时延 |


GBN 或 SR（由于不考虑出错，此情况二者等价）：

$$
U=\min\left(1,\frac{W T_f}{T_f+RTT}\right)
$$

吞吐量（速率）：

$$
R=\min\left(B,\frac{W\cdot L}{RTT+T_f}\right)
$$

$$
T_f=\frac{L}{B}
$$

> 题目给 $RTT$ 就用 $RTT$；题目给单程传播时延 $T_p$ 就用 $2T_p$。

## 超时重传与 NAK 

| 机制       | 触发条件             | 作用                   | 是否必须            | 典型问题                  |
| -------- | ---------------- | -------------------- | --------------- | --------------------- |
| **超时重传** | 发送方等不到 ACK，计时器超时 | 兜底发现丢包、ACK 丢失、NAK 丢失 | **可靠传输中基本必须有**  | 超时时间不好设，太短会误重传，太长会恢复慢 |
| **NAK**  | 接收方发现缺帧、错帧、乱序等异常 | 主动通知发送方“某帧没收到/错了”    | **不是必须，但可提高效率** | NAK 本身也可能丢失，不能单独保证可靠性 |

#### 超时重传
超时重传是可靠传输的**兜底机制**：发送方发出帧后启动计时器，如果在规定时间内没有收到确认，就认为该帧可能丢失，于是重新发送。

它能处理三类情况：

1. **数据帧丢失**：接收方根本没收到；
2. **ACK 丢失**：接收方收到了，但确认帧丢了；
3. **NAK 丢失**：接收方发了否定确认，但发送方没收到。

> 超时重传的本质是：发送方不能永远等，所以到时间没等到确认就自己重发。

> 只靠超时重传也能实现可靠传输，但丢包恢复速度可能比较慢。

#### NAK
NAK 是 Negative Acknowledgement，否定确认。接收方发现某个帧出错或缺失时，主动告诉发送方：“这个帧没有正确收到，请重传。”

例如接收方按顺序应该收到帧 $3$，却收到了帧 $4$，说明帧 $3$ 可能丢了，于是可以发送：

$$
NAK(3)
$$

发送方收到后就可以立即重传帧 $3$，不用等计时器超时。

> NAK 的本质是：接收方提前告诉发送方哪里出错，加快恢复。

> NAK 不是可靠传输的必要条件，因为即使没有 NAK，发送方最终也会因为超时而重传。

| 可靠传输方案            | 能否可靠 | 原因                                 |
| ----------------- | ---: | ---------------------------------- |
| **只有超时重传，没有 NAK** |   可以 | ACK + 超时足以发现未确认的数据                 |
| **有 NAK，但没有超时重传** |  不可靠 | 如果数据帧丢了且接收方没法发现，或者 NAK 丢了，发送方可能永远等 |
| **超时重传 + NAK**    |   更好 | 超时兜底，NAK 加速恢复                      |

## UDP 检验和计算

UDP 检验和用于**检错**，计算对象是：

1. **拼接内容**：把伪首部、UDP 首部、UDP 数据连续拼起来；
2. **按 $16$ 位分组**：每 $16$ 位切成一组，不足 $16$ 位末尾补 $0$；
3. **反码求和**：所有 $16$ 位数相加，若最高位产生进位，要回卷加到最低位；
4. **按位取反**：发送端把最终求和结果取反，得到 UDP 检验和；
5. **接收端验证**：接收端重新求和，若结果为全 $1$，说明未检测出错误。
