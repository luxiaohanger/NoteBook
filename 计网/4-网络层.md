## 路由器组件

| 组件      | 所属平面 | 主要作用             |
| ------- | ---- | ---------------- |
| 输入端口    | 数据平面 | 接收分组、查转发表、决定输出端口 |
| 交换结构    | 数据平面 | 把分组从输入端口转移到输出端口  |
| 输出端口    | 数据平面 | 排队、调度、发送分组       |
| 路由选择处理器 | 控制平面 | 运行路由协议、计算路由表/转发表 |

---

## 数据平面

数据平面负责**真正转发每一个到达的分组**，动作发生在路由器本地，速度要求高。

数据平面主要包括：

$$
输入端口 \rightarrow 交换结构 \rightarrow 输出端口
$$

基本过程：

1. **输入端口接收分组**：完成物理层、链路层处理；
2. **查找转发表**：根据目的 IP 地址查找下一跳或输出端口；
3. **通过交换结构转发**：把分组送到对应输出端口；
4. **输出端口发送**：排队、调度，并发送到下一条链路。

> 数据平面管“每个分组怎么走”，核心是查表转发。

---

## 控制平面

控制平面负责**决定路由器应该如何转发分组**，也就是生成数据平面使用的转发表。

控制平面主要由：

$$
路由选择处理器
$$

完成。

它运行路由协议，例如：

| 协议   | 作用        |
| ---- | --------- |
| RIP  | 距离向量路由协议  |
| OSPF | 链路状态路由协议  |
| BGP  | 自治系统间路由协议 |

控制平面的工作结果是生成：

$$
路由表 \rightarrow 转发表
$$

然后交给输入端口用于快速查找。

> 控制平面管“路线怎么算出来”，核心是路由选择。

## 路由器中的三种交换技术

| 交换技术    | 基本思想                      | 转发过程                                       | 优点            | 缺点                     |
| ------- | ------------------------- | ------------------------------------------ | ------------- | ---------------------- |
| 经内存交换   | 输入端口把分组交给路由器内存，再由内存送到输出端口 | 输入端口 $\rightarrow$ 内存 $\rightarrow$ 输出端口   | 实现简单          | 受内存读写速度限制，一次通常只能处理一个分组 |
| 经总线交换   | 输入端口通过共享总线把分组传到输出端口       | 输入端口 $\rightarrow$ 总线 $\rightarrow$ 输出端口   | 比内存交换快，实现较简单  | 所有端口共享总线，容易形成瓶颈        |
| 经互连网络交换 | 使用交叉开关等互连结构，同时连接多个输入/输出端口 | 输入端口 $\rightarrow$ 互连网络 $\rightarrow$ 输出端口 | 可并行转发多个分组，性能高 | 硬件复杂，成本较高              |

## 分组调度算法

| 调度算法   | 英文/缩写            | 基本思想                   | 优点           | 缺点                   | 适用理解          |
| ------ | ---------------- | ---------------------- | ------------ | -------------------- | ------------- |
| 先进先出   | FIFO / FCFS      | 按分组到达顺序排队，先到先发         | 简单、公平性直观     | 不区分业务类型，重要分组也要排队     | “排队买饭，谁先来谁先走” |
| 优先权排队  | Priority Queuing | 按优先级分类，高优先级队列先发送       | 可保证重要/实时业务优先 | 低优先级可能长期等待，甚至饥饿      | “急诊优先”        |
| 循环排队   | Round Robin      | 多个队列轮流发送，每个队列依次获得服务机会  | 防止某类业务长期占用链路 | 不考虑分组长度或业务权重，精细公平性一般 | “每个窗口轮流服务一个人” |
| 加权公平排队 | WFQ              | 按权重给不同队列分配带宽，权重大获得更多服务 | 兼顾公平和差异化服务   | 实现复杂                 | “按权重分配服务时间”   |

> FIFO 最简单；优先权排队可能导致低优先级饥饿；循环排队强调轮流服务；WFQ 强调按权重公平分配带宽。

## NAT 网络地址转换

| 类型      | 也叫           | 映射方式                      | 端口是否转换  | 特点                            | 典型用途           |
| ------- | ------------ | ------------------------- | ------- | ----------------------------- | -------------- |
| 静态 NAT  | Static NAT   | 一个私有 IP 固定映射到一个公网 IP      | 通常不转换端口 | 映射关系固定，外部可通过公网 IP 访问内网主机      | 内网服务器对外提供服务    |
| 动态 NAT  | Dynamic NAT  | 私有 IP 临时映射到公网地址池中的某个公网 IP | 通常不转换端口 | 公网 IP 不固定，用完释放；公网地址池数量限制并发主机数 | 企业内网访问互联网      |
| NAPT    | PAT / 端口地址转换 | 多个私有 IP 共享一个公网 IP，用端口区分   | 转换端口    | 最常见，最节省公网 IP                  | 家庭路由器、校园网、企业网关 |


静态 NAT 示例：

$$
192.168.1.10 \leftrightarrow 203.0.113.10
$$

动态 NAT 示例：

$$
192.168.1.10 \rightarrow 公网地址池中的某个IP
$$

NAPT 示例：

$$
192.168.1.10:5000 \rightarrow 203.0.113.5:40001
$$

$$
192.168.1.11:5000 \rightarrow 203.0.113.5:40002
$$

> 静态 NAT 是一对一固定映射；动态 NAT 是一对一临时映射；NAPT 是多对一，通过端口号区分不同内网连接。

## IPv4 向 IPv6 迁移中的建隧道方法

建隧道用于解决：**IPv6 网络尚未完全连通时，让 IPv6 数据报穿过 IPv4 网络传输**。

| 项目         | 说明                         |
| ---------- | -------------------------- |
| 核心思想       | 把 **IPv6 数据报封装进 IPv4 数据报** |
| 适用场景       | 两个 IPv6 网络之间隔着 IPv4 网络     |
| 隧道入口       | IPv6/IPv4 双协议栈路由器          |
| 隧道出口       | IPv6/IPv4 双协议栈路由器          |
| 中间 IPv4 网络 | 只把它当普通 IPv4 数据报转发          |
| 到达出口后      | 去掉 IPv4 首部，恢复原 IPv6 数据报    |

建隧道过程：

1. **IPv6 主机发送 IPv6 数据报**：数据报先到达隧道入口路由器；
2. **入口路由器封装**：在原 IPv6 数据报外面加一个 IPv4 首部；
3. **IPv4 网络转发**：中间 IPv4 路由器只看 IPv4 首部，不关心里面是 IPv6；
4. **出口路由器解封装**：去掉 IPv4 首部，取出原 IPv6 数据报；
5. **继续按 IPv6 转发**：出口路由器把 IPv6 数据报送往目标 IPv6 网络。

> 建隧道本质上就是“把 IPv6 包装成 IPv4 包”，让 IPv6 数据报临时借用 IPv4 网络通行。

> 注意：隧道中间的 IPv4 路由器不需要支持 IPv6，只有隧道两端需要支持 IPv4 和 IPv6。

## ICMP

ICMP，即 **Internet Control Message Protocol，网际控制报文协议**，属于网络层的辅助协议，用来在 IP 主机、路由器之间传递**差错报告**和**网络探测控制信息**。


#### ICMP 的作用

ICMP 主要不是用来传输普通数据，而是用来告诉发送方：**你的 IP 数据报在传输过程中出问题了，或者我收到了你的探测请求**。

常见作用有两类：

| 类型    | 作用                  |
| ----- | ------------------- |
| 差错报告  | 路由不可达、端口不可达、TTL 超时等 |
| 查询/探测 | 回显请求与回显应答，用于测试连通性   |

> 通俗地说，IP 只负责“尽力送包”，ICMP 负责“送不成时回个信”。

> ICMP 只能报告问题，不能纠正问题；它也不能保证自己的报文本身一定能送达。


#### ICMP 报文封装

ICMP 报文封装在 IP 数据报的数据部分中。

$$
IP数据报 = IP首部 + ICMP报文
$$

ICMP 报文格式大致为：

| 字段       | 作用                    |
| -------- | --------------------- |
| Type     | 报文类型，表示是哪一类 ICMP 消息   |
| Code     | 具体原因，对 Type 进一步细分     |
| Checksum | 校验和                   |
| Data     | 相关数据，通常包含原 IP 数据报的一部分 |

> 虽然 ICMP 是网络层协议，但它不是直接跑在链路层上，而是作为 IP 数据报的载荷被封装。


#### 常见 ICMP 报文类型

| ICMP 类型                 | 含义    | 常见场景                      |
| ----------------------- | ----- | ------------------------- |
| Echo Request            | 回显请求  | `ping` 发出的请求              |
| Echo Reply              | 回显应答  | `ping` 收到的回应              |
| Destination Unreachable | 目的不可达 | 网络不可达、主机不可达、端口不可达         |
| Time Exceeded           | 时间超过  | TTL 减到 0，常用于 `traceroute` |
| Redirect                | 重定向   | 路由器提示主机走更好的下一跳            |
| Source Quench           | 源站抑制  | 早期用于拥塞控制，现基本废弃            |

> 考试中最常见的是：`ping` 用 Echo，`traceroute` 用 TTL 超时。


#### ping

`ping` 使用 ICMP 的 **Echo Request** 和 **Echo Reply**。

过程是：

1. 主机 A 向主机 B 发送 ICMP Echo Request；
2. 主机 B 收到后返回 ICMP Echo Reply；
3. 主机 A 根据是否收到 Reply 判断网络是否连通，并计算往返时间 RTT。

$$
RTT = 接收应答时间 - 发送请求时间
$$

> `ping` 测的是 IP 层连通性，不等于应用服务一定可用。

> 有些主机或防火墙会禁用 ICMP，所以 `ping` 不通不一定代表目标主机真的不可达。


#### traceroute

`traceroute` 利用 ICMP 的 **Time Exceeded** 报文发现路径上的路由器。

基本过程：

1. 发送 TTL 为 1 的探测包；
2. 第一个路由器将 TTL 减为 0，丢弃数据报，并返回 ICMP Time Exceeded；
3. 再发送 TTL 为 2 的探测包，得到第二跳路由器；
4. 依次增加 TTL，直到到达目的主机。

> `traceroute` 的核心思想是：故意让数据报在不同跳数处超时，从而让沿途路由器暴露出来。

## 每路由器控制、逻辑集中式控制

| 对比项   | 每路由器控制           | 逻辑集中式控制           |
| ----- | ---------------- | ----------------- |
| 控制位置  | 每台路由器自己运行控制逻辑    | 控制逻辑集中在远程控制器中     |
| 控制平面  | 分布在各个路由器上        | 逻辑上集中             |
| 数据平面  | 路由器本地负责转发分组      | 路由器本地负责转发分组       |
| 路由计算  | 每台路由器通过路由协议共同计算  | 控制器统一计算转发表        |
| 路由器功能 | 既负责转发，又参与路由计算    | 主要负责转发，按控制器下发规则工作 |
| 典型例子  | OSPF、BGP 等传统路由协议 | SDN 网络            |
| 优点    | 分布式，单点依赖较少       | 全局视图强，便于统一管理和优化   |
| 缺点    | 协调复杂，全局优化困难      | 依赖控制器，控制器故障影响较大   |

> 每路由器控制是“每台路由器自己商量路怎么走”，逻辑集中式控制是“控制器统一算好路，再告诉路由器怎么转发”。

## IGP：自治系统内部路由协议

| 协议    | 类型/算法              | 主要交换内容            | 核心特点                       |
| ----- | ------------------ | ----------------- | -------------------------- |
| RIP   | 距离向量算法             | 到各目的网络的距离，通常用跳数表示 | 简单，最大跳数为 15，适合小型网络         |
| OSPF  | 链路状态算法，Dijkstra 算法 | 链路状态信息，包括邻居、链路代价等 | 每台路由器形成拓扑图后计算最短路径，收敛较快     |
| IS-IS | 链路状态算法，Dijkstra 算法 | 链路状态信息            | 与 OSPF 类似，常用于大型运营商网络       |
| EIGRP | 高级距离向量算法 / DUAL 算法 | 到目的网络的距离、可行后继等信息  | Cisco 提出，收敛快，结合距离向量和链路状态特点 |
| IGRP  | 距离向量算法             | 到目的网络的距离          | Cisco 早期协议，基本被 EIGRP 取代    |

> IGP 主要解决“一个 AS 内部怎么选路”的问题，重点是快速、稳定地找到内部路径。

## BGP：自治系统之间路由协议

| 协议/类型  | 作用范围                | 主要机制   | 主要交换内容                                   | 核心特点                   |
| ------ | ------------------- | ------ | ---------------------------------------- | ---------------------- |
| BGP-4  | AS 之间               | 路径向量机制 | 网络前缀、AS-PATH、NEXT-HOP、LOCAL-PREF、MED 等属性 | 当前互联网主要使用的外部路由协议       |
| eBGP   | 不同 AS 的边界路由器之间      | 路径向量机制 | 跨 AS 的可达前缀和路径属性                          | 用于 AS 与 AS 之间交换路由      |
| iBGP   | 同一 AS 内部的 BGP 路由器之间 | 路径向量机制 | 从外部学到的 BGP 路由及属性                         | 用于在 AS 内部分发外部路由        |
| MP-BGP | 多协议扩展 BGP           | 路径向量机制 | IPv4、IPv6、VPN 等多种地址族路由信息                 | 支持 IPv6、MPLS VPN 等扩展场景 |

> BGP 主要解决“不同 AS 之间能不能到达、经过哪些 AS、是否允许走这条路”的问题。

> BGP 并不是只作用于边界路由器，内部非边界路由器要学习外部前缀要使用 iBGP

> 源目的在同一 AS --> 只用 IGP ； 不在同一 AS ---> BGP + IGP

## SDN 体系结构的 4 个关键特征

| 关键特征          | 含义                            | 重点理解                              |
| ------------- | ----------------------------- | --------------------------------- |
| 基于流的转发        | 交换机根据“流表”转发分组，而不是只按目的 IP 查路由表 | 转发规则可以匹配多种首部字段，如源/目的 IP、端口号、协议类型等 |
| 数据平面与控制平面分离   | 交换机主要负责分组转发，控制逻辑从设备中分离出来      | 路由/转发决策不再由每台交换机独立完成               |
| 网络控制功能位于外部控制器 | SDN 控制器负责计算、管理和下发转发规则         | 控制器相当于网络的“大脑”                     |
| 可编程网络         | 网络行为可以通过软件应用程序控制和调整           | 便于实现流量工程、安全策略、负载均衡等功能             |

## SNMP

SNMP（Simple Network Management Protocol，简单网络管理协议）是用于**网络管理服务器/控制器与被管设备上的 Agent 之间交换管理信息**的协议。

| 项目   | 说明                       |
| ---- | ------------------------ |
| 协议名称 | SNMP，简单网络管理协议            |
| 所属场景 | 网络管理                     |
| 通信双方 | 管理服务器/控制器 与 被管设备上的 Agent |
| 管理对象 | 设备状态、接口流量、错误计数、配置信息等管理数据 |
| 数据组织 | 通常通过 MIB 描述管理对象          |
| 主要作用 | 查询设备状态、接收告警、修改部分配置       |
| 传输基础 | 通常使用 UDP                 |

SNMP 的基本工作方式：

1. **管理服务器发起查询**：向 Agent 请求某个管理对象的值；
2. **Agent 返回结果**：从本设备中读取对应管理数据并返回；
3. **设备主动告警**：当设备出现异常时，Agent 可以发送 Trap/Inform 通知管理服务器；
4. **管理服务器修改配置**：在允许的情况下，可以通过 SNMP 修改某些管理对象。

常见 SNMP 操作：

| 操作                | 含义                 |
| ----------------- | ------------------ |
| Get               | 管理服务器读取某个管理对象      |
| GetNext / GetBulk | 管理服务器连续读取多个管理对象    |
| Set               | 管理服务器修改某个管理对象      |
| Trap              | Agent 主动向管理服务器发送告警 |
| Inform            | 类似 Trap，但需要确认      |
