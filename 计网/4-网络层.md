## BGP IGP 

## 1. 先把三者放到同一张图里

```text
                  Internet
                     │
        ┌────────────┴────────────┐
        │                         │
       AS1                       AS2
   运营商/学校/公司网络        另一个自治系统
        │                         │
   ┌────┴────┐               ┌────┴────┐
   │ AS 内部 │               │ AS 内部 │
   │ 用 IGP  │               │ 用 IGP  │
   └────┬────┘               └────┬────┘
        │                         │
        └──────── BGP ────────────┘
              AS 之间交换路由
```

一句话：

> **IGP 管 AS 内部怎么走；BGP 管 AS 之间怎么到达；hot potato 是 BGP 选择出口时的一种策略：尽快把流量丢出自己的 AS。**

---

# 2. IGP 怎么理解？

**IGP** 全称是 **Interior Gateway Protocol**，即**内部网关协议**。

它用于：

> **一个 AS 内部的路由选择。**

常见 IGP：

| 协议    | 类型     | 算法           |
| ----- | ------ | ------------ |
| RIP   | 距离向量协议 | Bellman-Ford |
| OSPF  | 链路状态协议 | Dijkstra     |
| IS-IS | 链路状态协议 | Dijkstra     |

例如一个 AS 内部有这些路由器：

```text
AS100 内部

R1 —— R2 —— R3
│      │      │
R4 —— R5 —— R6
```

AS100 内部可以运行 OSPF。OSPF 负责计算：

```text
R1 到 R6 怎么走？
R3 到 R4 怎么走？
R5 到出口路由器怎么走？
```

所以 IGP 的目标一般是：

> 在 AS 内部按照链路代价找到较优路径，通常偏向“最短路径”。

---

# 3. BGP 怎么理解？

**BGP** 全称是 **Border Gateway Protocol**，即**边界网关协议**。

它用于：

> **AS 与 AS 之间交换可达的 IP 前缀。**

BGP 不是单纯找“最短路”，而是更强调**策略路由**。

例如：

```text
AS100 可以通过 AS200 到达 8.8.8.0/24
AS100 也可以通过 AS300 到达 8.8.8.0/24
```

BGP 要决定：

```text
去 8.8.8.0/24，到底从 AS200 出去，还是从 AS300 出去？
```

BGP 路由表项关心的不是“某台具体主机”，而是网络前缀：

```text
目的前缀 → 下一跳 AS / 下一跳路由器
```

例如：

| 目的前缀         | AS-PATH         | 下一跳      |
| ------------ | --------------- | -------- |
| `8.8.8.0/24` | `AS200 AS15169` | 边界路由器 B1 |
| `8.8.8.0/24` | `AS300 AS15169` | 边界路由器 B2 |

其中 **AS-PATH** 表示到达该前缀要经过哪些 AS。

---

# 4. BGP 和 IGP 的核心区别

| 对比项      | IGP            | BGP              |
| -------- | -------------- | ---------------- |
| 作用范围     | AS 内部          | AS 之间            |
| 典型协议     | OSPF、RIP、IS-IS | BGP              |
| 主要目标     | 内部最短路径/低代价     | 策略、商业关系、可达性      |
| 交换信息     | 内部链路/距离信息      | IP 前缀、AS-PATH、属性 |
| 是否知道全网拓扑 | AS 内部较清楚       | 不知道全互联网拓扑        |
| 算法思想     | 最短路算法          | 路径向量 + 策略选择      |
| 收敛速度     | 通常较快           | 相对较慢             |
| 使用位置     | 内部路由器之间        | 边界路由器之间          |

最重要的一点：

> **IGP 更像“地图导航”，BGP 更像“跨公司/跨运营商的政策协商”。**

---

# 5. BGP 分成 eBGP 和 iBGP

BGP 又可以分成两类：

| 类型   | 含义           | 使用位置                 |
| ---- | ------------ | -------------------- |
| eBGP | external BGP | 不同 AS 的边界路由器之间       |
| iBGP | internal BGP | 同一个 AS 内部的 BGP 路由器之间 |

例如：

```text
        AS100                           AS200
   ┌─────────────┐                 ┌─────────────┐
   │ R1 —— R2    │                 │ R5 —— R6    │
   │ │      │    │                 │             │
   │ R3 —— R4────┼──── eBGP ───────┼──── R7      │
   └─────────────┘                 └─────────────┘
```

* AS100 和 AS200 的边界路由器之间运行 **eBGP**；
* AS100 内部的 BGP 路由器之间可以运行 **iBGP**；
* AS100 内部的普通路由计算仍然依赖 **IGP**。

---

# 6. IGP 和 BGP 如何配合？

假设 AS100 有两个出口：

```text
                 AS200
                   │
                  R1
                  │
AS100 内部：  R3 —— R4 —— R5
                  │
                  R2
                   │
                 AS300
```

其中：

* R1 是连接 AS200 的出口路由器；
* R2 是连接 AS300 的出口路由器；
* AS100 内部运行 OSPF；
* R1、R2 通过 BGP 学到外部前缀。

如果 AS100 内部的 R5 要访问外部网络 `8.8.8.0/24`，它要解决两个问题：

### 问题 1：外部前缀从哪个出口走？

这是 **BGP** 决定的。

例如 BGP 可能告诉它：

```text
8.8.8.0/24 可以从 R1 出去
8.8.8.0/24 也可以从 R2 出去
```

### 问题 2：在 AS 内部怎么到达这个出口？

这是 **IGP** 决定的。

例如 OSPF 告诉它：

```text
R5 到 R1 的内部代价是 20
R5 到 R2 的内部代价是 5
```

所以：

> **BGP 决定选哪个出口，IGP 决定怎么走到这个出口。**

---

# 7. hot potato routing 是什么？

**Hot potato routing** 中文常译为：

> **热土豆路由 / 甩锅路由 / 最近出口路由。**

它的思想是：

> 一个 AS 收到要发往外部的流量后，尽量选择离自己最近的出口，把流量尽快交给别的 AS。

“热土豆”的比喻是：

```text
手里拿着烫手山芋，越快扔出去越好。
```

对应到网络中：

```text
我的 AS 内部链路资源有限，
所以我希望外部流量尽快离开我的网络，
不要在我自己的 AS 内部绕太远。
```

---

# 8. hot potato 的例子

假设 AS100 有两个出口可以到达同一个目的前缀 `P`：

```text
                      AS200
                        │
出口 E1 ────────────────┘

AS100 内部：

      R1
      │
      │ 内部代价 2
      │
      X
      │
      │ 内部代价 10
      │
出口 E2 ────────────────┐
                        │
                      AS300
```

对于 AS100 内部路由器 X：

| 出口 | 能否到达目的前缀 P | X 到出口的 IGP 代价 |
| -- | ---------- | ------------: |
| E1 | 能          |             2 |
| E2 | 能          |            10 |

如果 BGP 的其他属性差不多，那么 X 会选择 E1，因为：

```text
X 到 E1 更近
```

这就是 hot potato：

> **选最近的出口，把流量尽快送出本 AS。**

---

# 9. hot potato 和 IGP 的关系

hot potato 实际上依赖 IGP 代价。

因为“最近出口”怎么判断？

靠 AS 内部的 IGP metric。

例如：

```text
到出口 E1 的 OSPF cost = 2
到出口 E2 的 OSPF cost = 10
```

那么 hot potato 会倾向选择：

```text
出口 E1
```

所以可以理解为：

> **BGP 在多个外部路由差不多时，会参考 IGP 到 BGP 下一跳的距离，选择最近出口。**

---

# 10. hot potato 在 BGP 决策中的位置

BGP 选择路径时，不是只看一个因素，而是按一系列规则比较。

简化版顺序可以理解为：

1. 本地优先级 Local Preference；
2. AS-PATH 长度；
3. Origin 类型；
4. MED；
5. eBGP 优于 iBGP；
6. 到 BGP 下一跳的 IGP 代价；
7. 其他 tie-breaker。

**hot potato routing** 主要对应第 6 点：

> 当前面更重要的 BGP 属性无法区分路径时，选择 IGP 代价更小的出口。

也就是说，hot potato 不一定总是最高优先级；如果管理员配置了更高的策略，例如 Local Preference，那么会优先服从策略。

---

# 11. Hot potato 和 cold potato 对比

除了 hot potato，还有一个相反概念：**cold potato routing**。

| 类型          | 思想                      | 效果           |
| ----------- | ----------------------- | ------------ |
| Hot potato  | 尽快把流量送出本 AS             | 减少自己 AS 内部负担 |
| Cold potato | 尽量在自己 AS 内部多传一段，选择更合适出口 | 更好控制服务质量     |

### Hot potato

```text
离哪个出口近，就从哪个出口出去。
```

优点：

* 降低本 AS 内部传输成本；
* 减少内部链路负载；
* 实现简单。

缺点：

* 可能导致端到端路径不是最优；
* 把流量较早交给其他 AS，自己对后续路径控制较少。

### Cold potato

```text
不急着丢出去，而是在自己的网络里送到更优出口。
```

优点：

* 自己能控制更多路径；
* 可能改善用户体验；
* 大型内容网络、云网络、骨干网可能更关注这个。

缺点：

* 占用更多本 AS 内部链路资源；
* 运营成本更高。

---

# 12. 三者关系总结

可以这样记：

```text
AS 内部：
    IGP 负责内部最短路径
    例如 OSPF、IS-IS、RIP

AS 之间：
    BGP 负责跨 AS 可达性和策略
    例如选择经过哪个邻居 AS

BGP 选出口时：
    如果多个出口都可用
    可以参考 IGP 到出口的代价
    选择最近出口
    这就是 hot potato routing
```

---

## 考试版答案

> IGP 是自治系统内部使用的路由协议，如 RIP、OSPF、IS-IS，用于计算 AS 内部到各网络或路由器的路径。BGP 是自治系统之间使用的路径向量路由协议，用于在不同 AS 之间通告 IP 前缀可达性，并根据策略选择跨 AS 路径。
>
> 在一个 AS 内部，BGP 可能学到多个到同一外部前缀的出口。若这些 BGP 路由在较高优先级属性上相同，则路由器会根据 IGP 到 BGP 下一跳的代价选择最近出口，将流量尽快交给外部 AS，这种策略称为 hot potato routing。其核心思想是减少本 AS 内部传输成本。
